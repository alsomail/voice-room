### 数据库表设计
```sql
CREATE TABLE IF NOT EXISTS rooms (
    id            UUID         PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id      UUID         NOT NULL REFERENCES users(id),
    title         VARCHAR(30)  NOT NULL,
    room_type     VARCHAR(20)  NOT NULL DEFAULT 'normal',
    member_count  INT          NOT NULL DEFAULT 0,
    status        VARCHAR(20)  NOT NULL DEFAULT 'active',
    password_hash VARCHAR(255),
    max_members   INT          NOT NULL DEFAULT 50,
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    deleted_at    TIMESTAMPTZ,
    
    CONSTRAINT chk_rooms_title_length CHECK (char_length(title) BETWEEN 1 AND 30),
    CONSTRAINT chk_rooms_room_type CHECK (room_type IN ('normal', 'password', 'paid')),
    CONSTRAINT chk_rooms_status CHECK (status IN ('active', 'closed')),
    CONSTRAINT chk_rooms_member_count_non_negative CHECK (member_count >= 0),
    CONSTRAINT chk_rooms_max_members_positive CHECK (max_members > 0)
);
```

### 索引设计
- `idx_rooms_status_created_at`: 复合索引，支持按状态+时间排序查询
- `idx_rooms_owner_id`: 单列索引，支持按房主查询
- `idx_rooms_member_count`: 单列降序索引，支持按热度排序

## 五、Reviewer意见
> 状态枚举：🟢 通过 | 🔴 未通过

### Review 结论：🔴 未通过（1 HIGH）

**H-01：迁移文件编号跳跃 — `004_create_rooms.sql` 应为 `002_create_rooms.sql`**
- 位置：`app/server/migrations/004_create_rooms.sql`（文件名本身）
- 根因：`app/server/migrations/` 内仅有 `001_create_users.sql` 和 `004_create_rooms.sql`，中间 002/003 不存在。T-00002~T-00005 均为 Auth API / Redis 任务，均不产生 DB 迁移文件，rooms 表是 App Server 第二个迁移，编号应为 `002`。
- 影响：sqlx-migrate 一旦在数据库记录了 `004`，未来任何需要插入 002/003 序号的迁移（seats 表、messages 表等）均会被 sqlx 拒绝执行，造成迁移序列永久锁死，生产环境不可恢复。
- 修复：
  1. `mv app/server/migrations/004_create_rooms.sql app/server/migrations/002_create_rooms.sql`
  2. 同步修改 `room.rs` 测试中的路径引用（第 168 行 `004_create_rooms.sql` → `002_create_rooms.sql`）
  3. 同步修改 SQL 文件顶部注释行

---

**M-01：外键 `owner_id` 缺少显式 ON DELETE 策略**
- 位置：`004_create_rooms.sql:16`
- 问题：`REFERENCES users(id)` 未写 `ON DELETE` 子句，缺省为 `NO ACTION`（等同 RESTRICT），意图不明。
- 修复：`REFERENCES users(id) ON DELETE RESTRICT`（显式声明，与软删除策略配套）

**M-02：缺少 `member_count <= max_members` CHECK 约束**
- 位置：`004_create_rooms.sql:30-31`
- 问题：只约束了 `member_count >= 0` 和 `max_members > 0`，未约束上界，并发 bug 可产生 `member_count > max_members` 的脏数据。
- 修复：新增 `CONSTRAINT chk_rooms_member_count_le_max CHECK (member_count <= max_members)`

**M-03：热度/列表索引未过滤软删除行**
- 位置：`004_create_rooms.sql:34-36`
- 问题：三个索引均未加 `WHERE deleted_at IS NULL`，与项目 users 表 partial index 模式不一致；T-00008 列表查询每次需额外过滤软删除行。
- 修复：`idx_rooms_status_created_at` 和 `idx_rooms_member_count` 加 `WHERE deleted_at IS NULL`

**L-01：`updated_at` 无自动更新触发器（LOW）**
- T-00007/T-00010 实现时需在应用层手动维护，建议在后续 TDS 中明确说明。
