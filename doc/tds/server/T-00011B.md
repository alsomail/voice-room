## 一、功能描述

订阅 Redis `admin:events` 频道，将管理员操作指令实时推送至目标 WebSocket 连接。

核心能力：
1. **`ban_user`**：找到目标用户的所有在线连接 → 发送封禁通知 → 断开连接
2. **`close_room`**：广播房间关闭消息至所有房间成员 → 批量断开连接
3. **`broadcast_notice`**：向所有在线用户推送系统公告
4. **容错**：事件处理在独立 `tokio::spawn` 中执行，失败不影响主服务

## 二、事件格式规范

### ban_user

```json
{
  "type": "ban_user",
  "payload": { "user_id": "uuid" },
  "admin_id": "uuid",
  "ts": 1700000000
}
```

**服务端推送给用户**：
```json
{"type":"ban_user","message":"You have been banned from this platform."}
```

### close_room

```json
{
  "type": "close_room",
  "payload": { "room_id": "uuid" },
  "admin_id": "uuid",
  "ts": 1700000001
}
```

**服务端推送给房间成员**：
```json
{"type":"close_room","message":"This room has been closed by an administrator."}
```

### broadcast_notice

```json
{
  "type": "broadcast_notice",
  "payload": { "message": "系统维护通知" },
  "admin_id": "uuid",
  "ts": 1700000002
}
```

**服务端推送给所有在线用户**：
```json
{"type":"broadcast_notice","message":"系统维护通知"}
```

### AdminEvent（serde internally-tagged）

```rust
#[derive(Debug, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum AdminEvent {
    BanUser { payload: BanUserPayload, admin_id: Uuid, ts: i64 },
    CloseRoom { payload: CloseRoomPayload, admin_id: Uuid, ts: i64 },
    BroadcastNotice { payload: BroadcastNoticePayload, admin_id: Uuid, ts: i64 },
}
```

### handle_admin_event 逻辑

```rust
pub async fn handle_admin_event(event: AdminEvent, registry: Arc<ConnectionRegistry>) {
    match event {
        AdminEvent::BanUser { payload, .. } => {
            for (conn_id, sender) in registry.get_by_user_id(payload.user_id) {
                let _ = sender.send(ban_notification_json());
                registry.unregister(conn_id);
            }
        }
        AdminEvent::CloseRoom { payload, .. } => {
            let conns = registry.get_connections_in_room(payload.room_id);
            for (_, sender) in &conns { let _ = sender.send(room_closed_json()); }
            for (conn_id, _) in conns { registry.unregister(conn_id); }
        }
        AdminEvent::BroadcastNotice { payload, .. } => {
            registry.broadcast_to_all(&notice_json(payload.message));
        }
    }
}
```

### subscriber.rs（Redis 自动重连 + 优雅停机）

### 问题汇总

| ID | 级别 | 位置 | 问题描述 |
|----|------|------|----------|
| M-01 | MEDIUM | `subscriber.rs` `else` 分支 | `else` 分支仅 `break` 内层循环，回到外层循环无 sleep。Redis 处于"接受连接但立即断流"状态时形成紧密重连循环。修复：在 `break` 前补加 `tokio::time::sleep(Duration::from_secs(2)).await` |
| M-02 | MEDIUM | `subscriber.rs` 两处 sleep | 固定 2s 重连间隔，无指数退避。多实例部署下 Redis 恢复时存在重连风暴风险。建议实现指数退避（1s→2s→4s→…→60s cap） |
| L-01 | LOW | `subscriber.rs:53` | `msg.get_payload().unwrap_or_default()` 将 payload 解码错误静默转为空字符串，丢失原始错误上下文 |
| L-02 | LOW | `handler.rs` E07 | E07 直接调用 `handle_admin_event`，未覆盖 `tokio::spawn` panic 隔离场景。当前实现永不 panic，暂可接受 |
