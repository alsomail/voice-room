# TDS: ranking_test::r08 perf flake known-issue 收口 (Task ID: T-0000O)

## 一、背景与目标

- **关联需求**：模块 9 E2E 测试基建 follow-up；T-0000M Round 1 GlobalReview 缺陷 2 拆出。
- **本 Task 的目标**：将 `app/server/tests/ranking_test.rs` 中 `r08_response_time_under_100ms` 在共享开发机器上 p95 偶发超阈（实测 315ms，阈值 100ms）的现象正式登记为 **known-issue**，明确临时规避策略，让 `cargo test -p voice-room-server` 全套在 dev / CI 主管道默认绿色，并把真正的性能验证迁移到独立 perf 任务，避免 perf flake 阻塞功能 PR。
- **现状证据**：
  - 测试文件位置：`app/server/tests/ranking_test.rs:458`。
  - 当前规避手段：`cargo test -p voice-room-server -- --skip r08_response_time_under_100ms`（T-0000M 联调期间已实际采用）。
  - 触发条件：与其它 DB 集成测试并发跑（cargo 默认 `--test-threads`）+ 冷连接池 warm-up 抖动 + 共享 docker postgres 资源争用。
- **不在范围**：
  - 不重写 ranking 业务逻辑（该测试为性能基线测试，不涉及功能正确性）。
  - 不做完整的性能基准框架（criterion / iai 引入由后续独立 Task 决策）。
  - 不调整 ranking API 100ms p95 SLO 本身（产品契约不变）。

## 二、方案设计

### 三选项分析

| 方案 | 描述 | 优点 | 缺点 | 推荐度 |
|---|---|---|---|---|
| A | **迁出独立 perf 套件**（`tests/perf/`，仅在 perf CI job 跑） | 主管道彻底干净；perf 测试可独立调度；未来扩展容易 | 需新建测试组织结构；CI workflow 微调 | ⭐⭐⭐ |
| B | **保留位置，宽放阈值至 300ms 并加 warmup** | 改动最小（仅测试文件内部） | p95 SLO 在测试中失真；本质是降低质量门 | ⭐ |
| C | **保留位置，改测量协议**（warmup N=5 + 取 p95 over N≥20）| 协议更科学；阈值不变 | 测试运行时长 ×4；DB 资源占用增加；仍受并发抖动影响 | ⭐⭐ |

**最终方案：A + 文档化降级**。

### 核心策略

1. **Phase 1（本 Task 落地，工时 1h）**：
   - 在 `doc/tests/known-issues.md`（新建）登记 r08 的现象、触发条件、规避命令、长期方向。
   - 修改 `cargo test -p voice-room-server` 默认调用方（CI workflow + RUNBOOK 推荐命令）显式 `-- --skip r08_response_time_under_100ms`。
   - 在 `r08_response_time_under_100ms` 函数顶部添加 `#[ignore = "perf flake; see doc/tests/known-issues.md#r08"]` 属性 → cargo 默认 skip，需 `--ignored` 显式启用；这比命令行 `--skip` 更可靠（避免开发者忘记加 flag）。
2. **Phase 2（未来独立 Task，本 TDS 不交付）**：
   - 设立 `tests/perf/` 或 `app/server/perf/`，迁移 r08；CI 加 `perf-nightly` job 在低并发节点跑。
