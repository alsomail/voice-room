## 零、产品架构定位

### 四端职责划分

| 端 | 定位 | 核心用户 | 主要功能 | 部署方式 |
|----|------|----------|----------|----------|
| **Android** | C 端用户应用 | 普通用户 + 主播 | 语音聊天、送礼、充值、社交 | Google Play |
| **App Server** | C 端业务后端 | - | 用户认证、房间管理、实时通信 (WebSocket)、支付回调 | 公网部署 (AWS/阿里云) |
| **Admin Server** | B 端管理后端 | - | 后台管理 API、数据统计、报表生成 | **内网部署** (VPN 访问) |
| **Web** | B 端管理后台 | 运营人员 + 客服 | 用户管理、内容审核、数据统计、素材配置 | 内网部署 |

---

### 架构设计说明

#### 🔄 **为什么要分离 App Server 和 Admin Server？**

| 维度 | App Server | Admin Server | 原因 |
|------|-----------|--------------|------|
| **安全性** | 公网暴露 | **内网隔离 + VPN** | 管理后台涉及敏感操作（充值/封禁），必须隔离 |
| **并发要求** | **极高** (10万+ QPS) | 低 (< 100 QPS) | 用户端需要抗住大流量，后台只有运营人员使用 |
| **数据库** | 读多写少 (主从分离) | **读写权限** | 后台需要直接修改数据（充值/扣款） |
| **实时性** | WebSocket 长连接 | HTTP 短连接 | 用户端需要实时推送，后台不需要 |
| **部署策略** | 多区域部署 + CDN | 单区域内网 | 用户端需要低延迟，后台访问量小 |
| **监控** | APM + 全链路追踪 | 基础日志 | 用户端需要精细监控，后台简单即可 |

#### 🗄️ **数据库设计**

**共享数据库 vs 分离数据库？**  
→ **Phase 0 共享同一个 PostgreSQL**，通过不同账号权限隔离：
- `app_server_user`: 只读权限（SELECT）+ 部分写权限（INSERT users, rooms 等）
- `admin_server_user`: 全部读写权限 + DDL 权限

**Phase 1 后可考虑读写分离**：
- App Server → 读库（主从复制）
- Admin Server → 主库（可写）

---

**重要说明**：
- 🚫 Web 端**不是**用户端，普通用户无法访问
- ✅ Web 端是**运营管理系统**，需要通过 VPN 访问
- ✅ 两个 Server 共享数据库，但 API 完全独立
- ✅ 未来可能增加 iOS 端（与 Android 功能一致，复用 App Server）

## 二、业务流程与规则说明

### 2.1 核心业务流：手机号一键登录（Phase 0 首要功能）

#### ✅ 正向流程 (Happy Path)

```
1. 用户打开 App
   ↓
2. 输入手机号（自动识别国际区号 +966/+971 等）
   ↓
3. 点击"获取验证码" → App 请求 App Server: POST /auth/send-code
   ↓
4. App Server → Twilio → 用户手机收到 6 位验证码
   ↓
5. 用户输入验证码 → 点击"登录"
   ↓
6. App 请求 App Server: POST /auth/login {phone, code}
   ↓
7. App Server 校验验证码：
   ├─ 手机号已注册 → 查询 user 记录 → 签发 JWT
   └─ 手机号未注册 → 自动创建 user（默认昵称"用户XXXX"） → 签发 JWT
   ↓
8. App 保存 JWT 到本地 → 跳转到房间大厅
```

**设计要点**：
- 注册和登录合二为一，减少用户流失（参考 Yalla 的设计）
- 新用户无需设置密码，手机号即身份
- JWT 有效期 30 天，减少频繁登录

#### ❌ 异常流程

**A. 短信发送失败**
- **原因**: Twilio 限流、运营商拦截、手机号格式错误
- **方案**: 
  - 返回具体错误信息（格式错误/频率过快/服务异常）
  - 60 秒冷却期防刷
  - 每日同一手机号限 10 次

**B. 验证码过期/错误**
- **过期**: 5 分钟有效期，过期返回 401 + "验证码已过期，请重新获取"
- **错误**: 最多尝试 5 次，超过后锁定 30 分钟
- **重发**: 60 秒后可重新发送，生成新验证码，旧码失效

**C. 网络断开**
- **发送验证码时断网**: 本地显示"网络异常，请检查连接"
- **登录请求时断网**: 本地保留输入，恢复后自动重试 1 次
- **登录成功后断网**: JWT 已保存，下次打开 App 直接进入大厅

**D. 设备被禁用/账号被封**
- **方案**: App Server 返回 403 + 封禁原因 + 封禁到期时间
- **展示**: 弹窗提示"您的账号因XXX已被封禁至YYYY-MM-DD"

### 2.2 核心业务流：后台管理员登录（Web 端）

#### ✅ 正向流程

```
1. 管理员通过 VPN 访问后台地址
   ↓
2. 输入账号 + 密码
   ↓
3. Web 请求 Admin Server: POST /admin/login
   ↓
4. Admin Server 校验 bcrypt 密码 → 签发 JWT (含 role)
   ↓
5. Web 保存 JWT → 跳转数据看板
```

#### ❌ 异常流程
- **密码错误 5 次**: 账号锁定 30 分钟，记录 IP
- **非 VPN 访问**: 网络层直接拒绝（Admin Server 仅内网暴露）
- **权限不足**: 访问无权限页面时显示 403 提示

### 2.3 核心业务流：虚拟礼物打赏（Phase 1）

#### ✅ 正向流程 (Happy Path)

```
1. 用户进入房间
   ↓
2. 点击"礼物"按钮 → 弹出礼物面板
   ↓
3. 选择礼物 (检查钻石余额) → 选择赠送对象 (房主/主麦/指定麦位)
   ↓
4. 点击"赠送" → 客户端发送 SendGift 请求
   ↓
5. Server 验证 & 扣款 (事务) → 广播 GiftSent 事件
   ↓
6. 所有房间内用户收到广播 → 播放礼物特效动画
   ↓
7. 接收者魅力值增加，赠送者财富值增加
   ↓
8. 刷新排行榜
```

#### ❌ 异常流程

**A. 网络断开/重连场景**
- **问题**: 用户点击"赠送"后网络断开，不知道是否成功
- **方案**: 
  - Server 基于 `msg_id` 去重，重复请求返回相同结果
  - 客户端重连后，拉取最近 10 条房间事件，补偿渲染
  - 本地显示"赠送中..."状态，超时 5 秒自动重试 1 次

**B. 余额不足**
- **问题**: 用户选礼物时余额够，点击时余额不足 (并发消费)
- **方案**: 
  - Server 返回 `InsufficientBalance` 错误码
  - 客户端弹窗提示"钻石不足，立即充值？" + 跳转充值页

**C. 赠送对象已下麦/离开房间**
- **问题**: 用户选中 3 号麦赠送，点击时该用户已下麦
- **方案**: 
  - Server 检查目标用户是否仍在房间，若不在则拒绝
  - 客户端提示"该用户已离开，礼物未赠送，钻石未扣除"

**D. 并发抢购限量礼物**
- **问题**: 节日限定礼物库存 100 份，多人同时抢
- **方案**: 
  - Redis 原子递减库存，售罄后返回 `SoldOut`
  - 客户端显示实时剩余数量 (WebSocket 推送库存变更)

### 2.4 核心业务流：麦位管理

#### ✅ 正向流程：用户主动上麦

```
1. 用户点击空闲麦位
   ↓
2. 客户端请求麦克风权限 (首次)
   ↓
3. 发送 TakeMic 请求 (携带麦位号)
   ↓
4. Server 检查麦位是否空闲 + 用户是否被禁麦
   ↓
5. 成功后广播 MicTaken 事件 (携带用户信息)
   ↓
6. 所有客户端渲染头像到对应麦位
   ↓
7. 用户开启本地麦克风推流
```

#### ❌ 异常流程

**A. 麦克风权限拒绝**
- **Android**: 弹窗提示"需要麦克风权限才能上麦，去设置？"
- **Web**: 浏览器原生提示 + 引导刷新页面重新授权

**B. 并发抢麦**
- **问题**: 10 人同时点击 3 号麦
- **方案**: 
  - Server 基于 Redis 分布式锁，只有第一个请求成功
  - 失败者收到 `MicOccupied` 错误 + 提示"手慢了，已被抢占"

**C. 房主抱用户上麦**
- **前置条件**: 目标用户在房间内 + 未在其他麦位
- **流程**: 房主发送 `ForceTakeMic` 请求，Server 校验权限后强制上麦
- **通知**: 被抱用户收到通知"房主邀请您上X号麦"

### 2.5 核心业务流：贵族购买（Phase 1）

#### ✅ 正向流程

```
1. 用户打开"贵族中心"
   ↓
2. 选择等级 (骑士 30$/月)
   ↓
3. 选择支付方式 (Apple Pay / Google Pay / 信用卡)
   ↓
4. 跳转第三方支付
   ↓
5. 支付成功回调 → Server 验证订单 → 开通贵族

   ↓
7. 立即生效贵族特权 (进场特效/踢人权限等)
```

#### ❌ 异常流程

**A. 支付成功但回调丢失**
- **方案**: 
  - Server 定时轮询支付平台订单状态 (每 5 分钟)
  - 最长 24 小时内补偿到账
  - 客户端显示"处理中，请耐心等待"

**B. 重复购买同等级贵族**
- **方案**: 自动续期，叠加时长 (30 天 + 30 天 = 60 天)

**C. 贵族过期处理**
- **方案**: 
  - 到期前 3 天推送续费提醒
  - 到期后进入 3 天宽限期 (特权保留)
  - 宽限期后失效，需重新购买

---

### 2.6 跨服务通信：Admin Server → App Server

**核心问题**: Admin Server 的某些操作需要实时影响 App Server 的在线用户，例如：
- 管理员封禁用户 → App Server 需要立即踢出该用户
- 管理员强制关闭房间 → App Server 需要解散房间并通知所有成员
- 管理员广播全服公告 → App Server 需要推送给所有在线用户

**Phase 0 方案：基于共享 Redis 的事件队列**

```
Admin Server                     Redis                    App Server
    │                              │                         │
    │── PUBLISH admin:events ──▶  │  ◀── SUBSCRIBE ──────── │
    │   {type: "ban_user",        │      admin:events       │
    │    user_id: 123}            │                         │
    │                              │                    ┌────┴────┐
    │                              │                    │ 踢出用户 │
    │                              │                    │ 关闭WS  │
    │                              │                    └─────────┘
```

- **技术**: Redis Pub/Sub（简单可靠，Phase 0 足够）
- **频道**: `admin:events`
- **消息格式**: `{type: string, payload: object, admin_id: number, timestamp: number}`
- **支持事件**: `ban_user`, `unban_user`, `close_room`, `broadcast_notice`
- **Phase 1+ 可选升级**: 引入 NATS/Kafka 做更强的消息总线

| **只读** | 仅查看权限，无操作权限 | 实习生 |

**审计日志**:
- 记录所有敏感操作（封禁/充值/删除等）
- 日志字段：操作人、操作时间、操作类型、目标对象、IP地址
- 日志保留 180 天

---

### 5.3 技术选型建议

| 技术栈 | 推荐方案 | 原因 |
|--------|---------|------|
| **Admin Server 框架** | **Rust (Axum)** | 与 App Server 共享领域模型 crate，减少重复代码 |
| **Admin Server ORM** | SQLx | 与 App Server 一致，复用 migration |
| **Web 前端框架** | React + TypeScript | 团队熟悉，生态丰富 |
| **Web UI 组件库** | Ant Design | 后台系统标准选择，组件丰富 |
| **Web 状态管理** | Zustand | 轻量高效 |
| **Web 图表库** | ECharts | 数据可视化 |
| **Web 表格** | Ant Design Table + Virtual Scroll | 大数据量性能优化 |
| **权限** | RBAC (Role-Based Access Control) | 细粒度权限控制 |

**重要**: App Server 和 Admin Server 都使用 Rust，共享一个 `shared` crate 包含：
- 数据库表结构定义 (models)
- 公共错误码
- 工具函数 (JWT 签发/校验, 密码哈希)
- Redis 封装

## 六、风险与应对

### 6.1 合规风险

| 风险 | 影响 | 应对方案 | 优先级 |
|------|------|----------|--------|
