# TDS: 上麦/下麦逻辑 (Task ID: T-30013)

## 一、背景与目标

- **关联需求**：语聊房核心体验——麦位上麦/下麦完整交互流程（Mic 模块）
- **前置依赖**：
  - T-00014（App Server TakeMic 接口 ✅）：服务端处理 TakeMic/LeaveMic 信令，广播 MicTaken/MicLeft 事件
  - T-30012（麦克风权限 ✅）：`onMicPermissionGranted(slotIndex)` 回调确保权限已授予
  - T-30010（RoomViewModel ✅）：已有 wsClient 发送/接收 WS 消息，RoomViewState 已定义 micSlots
- **本 Task 目标**：
  1. 扩展 `IMediaService` 防腐层接口（增加 joinChannel / leaveChannel / startPublishAudio / stopPublishAudio）
  2. 实现上麦：权限通过 → 发送 `TakeMic` → 等待 `MicTaken` 广播 → 调用 `IMediaService.joinChannel/startPublishAudio`
  3. 实现下麦：点击自己的麦位 → 发送 `LeaveMic` → 等待 `MicLeft` 广播 → 调用 `IMediaService.stopPublishAudio/leaveChannel`
  4. 区分"点击空麦位（上麦）"与"点击自己的麦位（下麦）"的事件路由

## 二、方案设计

### §2.1 扩展防腐层接口 IMediaService

```kotlin
interface IMediaService {
    fun providerName(): String
    suspend fun joinChannel(roomId: String, userId: String): Result<Unit>
    suspend fun leaveChannel(): Result<Unit>
    suspend fun startPublishAudio(): Result<Unit>
    suspend fun stopPublishAudio(): Result<Unit>
}
```

现有 `NoOpMediaService` 追加空实现；新增 `FakeMediaService`（位于 `core/media/`）供测试注入，记录所有调用。

### §2.2 WS 消息格式（基于现有代码约定）

| 方向 | type | 字段 |
|------|------|------|
| Client → Server | `TakeMic` | `roomId`, `slotIndex` |
| Server → Client | `MicTaken` | `slotIndex`, `userId`, `nickname` |
| Client → Server | `LeaveMic` | `roomId`, `slotIndex` |
| Server → Client | `MicLeft` | `slotIndex` |

### §2.3 上麦流程

```
onMicPermissionGranted(slotIndex)
  → currentRoomId 非空校验
  → wsClient.send({"type":"TakeMic","roomId":...,"slotIndex":...})
  
handleWsMessage("MicTaken")：
  → 更新 micSlots UI 状态（已有逻辑不变）
  → 若 userId == currentUserId：
      mediaService.joinChannel(roomId, userId)
      if 失败 → ShowToast 错误提示
      mediaService.startPublishAudio()
```

### §2.4 下麦流程

```
onMicSlotClick(slotIndex)
  → 取 micSlots[slotIndex].userId
  → 若 userId == currentUserId（且非空）：
      wsClient.send({"type":"LeaveMic","roomId":...,"slotIndex":...})

handleWsMessage("MicLeft")：
  → 读取该槽位原有 userId（在清空前）
  → 更新 micSlots UI 状态（已有逻辑不变）
  → 若原 userId == currentUserId：
      mediaService.stopPublishAudio()
      mediaService.leaveChannel()
```

### §2.5 currentUserId 持有

`joinRoom(roomId: String, userId: String = "")` 新增 `userId` 默认参数，内部存至 `currentUserId` 字段。
现有测试均使用 `joinRoom("room-1")` 单参数调用，默认值保持向后兼容。

#### [HIGH] CancellationException 未 re-throw — MicTaken / MicLeft 媒体协程

**文件**：`feature/room/RoomViewModel.kt` 第 263–274 行（MicTaken）、第 294–297 行（MicLeft）

**问题描述**：

项目已在 T-30010 Review 中确立"所有 `viewModelScope.launch` 块必须 try-catch 并 re-throw `CancellationException`"的强约定（`joinRoom`、`onMicPermissionGranted`、`leaveMic` 均已遵守）。但以下两个媒体服务协程完全缺少 try-catch：

```kotlin
// MicTaken — 无任何 try-catch
viewModelScope.launch {
    val joinResult = mediaService.joinChannel(roomId, userId)
    if (joinResult.isFailure) { ... return@launch }
    mediaService.startPublishAudio()   // 失败结果被静默忽略
}

// MicLeft — 无任何 try-catch
viewModelScope.launch {
    mediaService.stopPublishAudio()
    mediaService.leaveChannel()
}
```

**风险**：
1. 真实 RTC SDK（Agora/Zego）接入后，`joinChannel`/`leaveChannel` 内部可能通过 `withContext` 切换调度器，一旦 `viewModelScope` 被取消，`CancellationException` 将被静默吞掉，破坏协程取消语义，导致难以复现的泄漏 / 状态错乱。
2. `startPublishAudio()` 的 `Result` 返回值未检查，失败时用户无任何反馈，与 `joinChannel` 失败有 ShowToast 的处理不一致。

**修复方案**：

```kotlin
// MicTaken 修复
viewModelScope.launch {
    try {
        val joinResult = mediaService.joinChannel(roomId, userId)
        if (joinResult.isFailure) {
            _events.trySend(RoomEvent.ShowToast("加入频道失败：${joinResult.exceptionOrNull()?.message}"))
            return@launch
        }
        val publishResult = mediaService.startPublishAudio()
        if (publishResult.isFailure) {
            _events.trySend(RoomEvent.ShowToast("开启推流失败：${publishResult.exceptionOrNull()?.message}"))
        }
    } catch (e: CancellationException) {
        throw e
    } catch (e: Exception) {
        _events.trySend(RoomEvent.ShowToast("上麦媒体操作异常：${e.message}"))
    }
}

// MicLeft 修复
viewModelScope.launch {
    try {
        mediaService.stopPublishAudio()
        mediaService.leaveChannel()
    } catch (e: CancellationException) {
        throw e
    } catch (e: Exception) {
        // 下麦清理失败可静默（或可选 ShowToast）
    }
}
```

#### [MEDIUM] startPublishAudio() 失败被静默忽略

**文件**：`RoomViewModel.kt` 第 273 行

**问题**：`mediaService.startPublishAudio()` 返回 `Result<Unit>`，但返回值完全未检查。如果推流启动失败，用户不会收到任何提示，而业务上"加入频道但未推流"是一个静默错误状态，对用户体验影响严重。

**修复**：同上，在 `joinResult` 成功后检查 `startPublishAudio()` 的返回值并在失败时发出 ShowToast。

---

#### [MEDIUM] MicLeft 协程整体缺少错误处理

**文件**：`RoomViewModel.kt` 第 294–297 行

**问题**：`stopPublishAudio()` / `leaveChannel()` 失败时无任何处理，异常会直接传播到 `viewModelScope` 的异常处理器（`CoroutineExceptionHandler` 未配置时为 `Thread.uncaughtExceptionHandler`），在生产环境可能引发 crash。

**修复**：同上，添加 try-catch 块，至少 re-throw `CancellationException`。

---

#### [LOW] FakeMediaService 可注入失败场景受限

**文件**：`core/media/FakeMediaService.kt`

**问题**：仅 `joinChannelResult` 支持注入失败，`startPublishAudio`/`stopPublishAudio`/`leaveChannel` 均无对应的 `xxxResult` 字段。修复上述 MEDIUM 问题后，`startPublishAudio` 失败路径需要对应测试，但当前 Fake 无法支持。

**建议**：

```kotlin
var startPublishAudioResult: Result<Unit> = Result.success(Unit)
var stopPublishAudioResult: Result<Unit> = Result.success(Unit)
var leaveChannelResult: Result<Unit> = Result.success(Unit)
```
