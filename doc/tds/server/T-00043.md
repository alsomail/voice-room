# TDS: Chat 消息持久化 + REST 历史查询接口 (Task ID: T-00043)

## 一、需求背景

- **关联需求**：QA 战报 BUG-FUNC-003（模拟）暴露聊天消息未持久化，用户断线重连后无法查看历史消息。
- **影响测试用例**：TC-CHAT-00007（历史消息查询）缺失，需补充；TC-WS-00003（断线重连补偿）仅覆盖 `last_msg_id` 回放，不含聊天 REST 拉取兜底。
- **业务影响**：用户体验受损，无法回溯房间聊天记录；纠纷举报场景缺少证据链。
- **现状**：
  - T-00016（文本消息广播）已实现 `SendMessage` WS 信令，广播 `ChatMessage` 给房间内在线用户。
  - **缺陷**：消息仅在内存中（`RoomState.recent_broadcasts`），未持久化到数据库。
  - `doc/protocol/websocket_signals.md §6.7` 定义了 `last_msg_id` 回放机制，但缓冲窗口仅 200 条 / 3 分钟，超出窗口后客户端无 REST 接口拉取。
- **本 Task 目标**：
  1. 新建 `chat_messages` 表，存储所有房间聊天消息。
  2. 在 `SendMessage` 处理逻辑中，广播后同步插入 DB（异步写可选，P1 先同步保证一致性）。
  3. 新增 REST 接口 `GET /api/v1/rooms/:room_id/messages`，支持分页查询历史消息。

## 二、技术方案

### 数据迁移

**文件**：`app/server/migrations/010_create_chat_messages.sql`（假定序号 010，以实际为准）

```sql
-- Migration: 010 - 创建聊天消息表
-- Up
CREATE TABLE IF NOT EXISTS chat_messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    room_id UUID NOT NULL REFERENCES rooms(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE SET NULL,
    content TEXT NOT NULL CHECK (char_length(content) > 0 AND char_length(content) <= 500),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 索引优化：按房间+时间倒序查询
CREATE INDEX idx_chat_messages_room_time ON chat_messages (room_id, created_at DESC);

-- 索引优化：按用户查询（举报场景）
CREATE INDEX idx_chat_messages_user ON chat_messages (user_id, created_at DESC);

-- Down
DROP TABLE IF EXISTS chat_messages;
```

**字段说明**：
- `id`：消息唯一标识（UUID）。
- `room_id`：所属房间（外键，房间删除时级联删除消息）。
- `user_id`：发送者（外键，用户删除时置 NULL 保留消息但匿名化）。
- `content`：消息内容（CHECK 约束：1-500 字符，与 T-00016 一致）。
- `created_at`：发送时间（用于排序与分页）。

### 改造点位 1：SendMessage 持久化

**文件**：`app/server/src/modules/chat/handler.rs`（或 `message.rs`，需 grep `SendMessage` 定位）

```rust
// 伪代码片段
async fn handle_send_message(
    user_id: Uuid,
    room_id: Uuid,
    content: String,
    db: &PgPool,
    room_manager: &RoomManager,
) -> Result<(), ApiError> {
    // 1. 校验（长度、敏感词、禁言）— 已有逻辑
    validate_message(&content)?;
    check_mute_status(user_id, room_id).await?;

    // 2. 插入数据库（新增）
    let message_id = sqlx::query_scalar!(
        "INSERT INTO chat_messages (room_id, user_id, content) VALUES ($1, $2, $3) RETURNING id",
        room_id,
        user_id,
        content
    )
    .fetch_one(db)
    .await?;

    // 3. 构造广播消息
    let broadcast = json!({
        "type": "ChatMessage",
        "msg_id": message_id.to_string(), // 使用 DB id 作为 msg_id
        "payload": {
            "user_id": user_id,
            "content": content,
            "ts": Utc::now().timestamp(),
        }
    });

    // 4. 广播给房间（已有逻辑）
    room_manager.broadcast(room_id, broadcast).await?;

    Ok(())
}
```

**关键点**：
- 使用 DB 自动生成的 `id` 作为 WS 消息的 `msg_id`，确保幂等（重复消息可通过 `msg_id` 去重）。
- 插入 DB 与广播**串行**执行（P1 阶段）；若需高 QPS 优化，P2 可改为异步写（先广播 → 后台任务批量插入）。

### 改造点位 2：历史消息查询接口

**文件**：`app/server/src/modules/chat/routes.rs` + `handler.rs`

```rust
// routes.rs 新增路由
Router::new()
    .route("/rooms/:room_id/messages", get(get_room_messages))

// handler.rs 实现
async fn get_room_messages(
    Path(room_id): Path<Uuid>,
    Query(params): Query<PaginationParams>,
    State(state): State<AppState>,
    Extension(user): Extension<AuthUser>,
) -> Result<Json<MessagesResponse>, ApiError> {
    // 1. 校验用户是否在房间内（可选，P1 跳过，任何登录用户可查询）
    // state.room_manager.is_user_in_room(user.id, room_id)?;

    // 2. 分页查询
    let limit = params.limit.unwrap_or(50).min(100); // 默认 50，最大 100
    let offset = params.offset.unwrap_or(0);

    let messages = sqlx::query_as!(
        ChatMessageRow,
        r#"
        SELECT cm.id, cm.user_id, cm.content, cm.created_at,
               u.nickname, u.avatar_url
        FROM chat_messages cm
        LEFT JOIN users u ON cm.user_id = u.id
        WHERE cm.room_id = $1
        ORDER BY cm.created_at DESC
        LIMIT $2 OFFSET $3
        "#,
        room_id,
        limit as i64,
        offset as i64
    )
    .fetch_all(&state.db)
    .await?;

    Ok(Json(MessagesResponse {
        items: messages,
        total: count_messages(room_id, &state.db).await?,
        limit,
        offset,
    }))
}

#[derive(Deserialize)]
struct PaginationParams {
    limit: Option<i32>,
    offset: Option<i32>,
}
```

## 三、TDD 验收标准

### 功能用例

1. **U-1**（消息持久化）：用户 U1 在房间 R1 发送消息 "hello"，`chat_messages` 表新增 1 行，`content="hello"`，`user_id=U1`，`room_id=R1`。
2. **U-2**（广播与 DB 一致）：U1 发送消息，房间内 U2/U3 收到 `ChatMessage` 的 `msg_id` = DB 插入返回的 `id`。
3. **U-3**（历史查询）：调用 `GET /rooms/{R1}/messages?limit=10`，返回最新 10 条消息（按 `created_at DESC` 排序）。
4. **U-4**（分页正确）：R1 有 123 条消息，`limit=50&offset=0` 返回前 50 条，`offset=50` 返回 51-100 条，`offset=100` 返回 101-123 条（23 条）。
5. **U-5**（空房间）：新建房间 R2 无消息，查询返回 `{"items":[], "total":0}`。

### 边界用例

6. **B-1**（limit 超限）：`limit=999` 时服务端强制降为 100（或返回 400）。
7. **B-2**（offset 越界）：`offset=99999`（超过 total）返回空数组，不报错。
8. **B-3**（并发写）：10 个用户同时发送消息，所有消息均成功插入 DB，无丢失（通过 `COUNT(*)` 验证）。

### 集成用例

9. **I-1**（TC-CHAT-新增-07）：用户进房 → 发 5 条消息 → 断线 → 重连 → 调用 REST 接口拉取历史，获得全部 5 条。
10. **I-2**（与 last_msg_id 互补）：`last_msg_id` 回放窗口出窗后，客户端降级为 REST 拉取（模拟 3 分钟后重连）。

### 回归用例

11. **R-1**：迁移脚本幂等执行 3 次，`chat_messages` 表结构一致，无重复索引。
12. **R-2**：T-00016 现有测试通过（`cargo test -p voice-room-server --test chat_test`）。
13. **R-3**：性能回归：发送消息平均延迟增加 <50ms（DB 插入开销可接受）。

### 4.3 设计决策

- **D-1 `payload.msg_id` 而非 envelope.msg_id**：`broadcast_to_room` 始终覆盖 envelope.msg_id 以服务 §6.7 last_msg_id 重放窗口，因此将 DB id 放入 chat 业务负载 `payload.msg_id`，不破坏现有重放语义；现有 S01..S09 单元测试断言 `result.msg_id` 为客户端 msg_id（result 回显），与本次改动正交。
- **D-2 写库 → 广播 顺序**：先 DB 后广播，DB 失败时不广播（避免客户端看到不在历史的消息）；DB 错误统一返回 50000。
- **D-3 表结构**：迁移已带 `room_id REFERENCES rooms(id) ON DELETE CASCADE` 与 `user_id REFERENCES users(id) ON DELETE SET NULL`；房间删除时联动清理 chat_messages，用户删除时保留消息但匿名化（user_id NULL）。
- **D-4 `_sqlx_app_migrations` 注册**：迁移文件名按现有规则放入 `app/server/migrations/`，启动时由 `voice_room_shared::migrate::run_migrations_with_table` 自动登记，符合 T-0000M ADR-0001。
- **D-5 排序 tiebreak**：`ORDER BY created_at DESC, id DESC` —— 第二键 `id DESC` 仅作 deterministic tiebreak（同 created_at 下返回顺序稳定 / 不抖动），**不**主张其反映插入时序（id 为 UUID v4 随机生成）；FakeRepo 用 atomic seq 模拟同语义。如需严格插入序，需后续引入 `seq BIGSERIAL` 或迁移到 UUIDv7。

### 4.4 已知风险 / 后续

- 真实 DB 性能（R-3 50ms）依赖网络；本地 PG 实测 < 5ms，CI/远程未做强约束。Round 1 后新增 `r3_real_db_perf_smoke`（`#[ignore]`）作为真路径基线（100 次 INSERT，p95 < 50ms）。
- 历史接口未实现 `since`/`before` 时间游标，仅 limit/offset；超大回滚（offset 数千）下需后续切游标分页。Round 1 已加 offset 软上限（`MAX_OFFSET = 100_000`）兜底。
- `id` 为 UUID v4，`ORDER BY ..., id DESC` 仅做稳定排序，不反映插入时序；如未来需要严格插入序（如审计回放），需引入 `seq BIGSERIAL` 列或迁移到 UUIDv7（已在 D-5 标注）。
