## 二、方案设计
### 核心数据流
```
JoinRoom handler (现有 T-00009)
  ↓ join 成功后
NobleEntryService::on_join(user_id, room_id):
  ├─ 查 user_nobles JOIN noble_tiers → tier 信息
  ├─ 若 tier=null → 退出（普通用户）
  ├─ 序列化 NobleSnapshot { tier_id, level, badge_color, frame_url, expire_at }
  ├─ 注入 UserJoined.payload.noble（已有信令字段补充）
  ├─ 按 privileges.entry_effect.scope 决策广播：
  │   - none → 仅 UserJoined（隐身贵族也广播 UserJoined 但 noble 字段被 filter）
  │   - own_room/half/fullscreen → 房间广播 NobleEntered
  ├─ 按 privileges.invisibility.scope 过滤接收者（隐身规则见 §10.4.5）
  ├─ 若 level >= 5 (duke/king) 且 today_count(noble_global_broadcast_log) < daily_limit:
  │     INSERT noble_global_broadcast_log + Redis PUBLISH "noble:global" NobleEntranceGlobal
   ↓
其他 server 实例 SUBSCRIBE "noble:global" → 推送给所有在线 WsConnection
```

## 三、TDD 验收用例
- [ ] tier=null 用户 JoinRoom：UserJoined 不含 noble 字段
- [ ] 普通贵族 JoinRoom：UserJoined.payload.noble 含完整 5 字段
- [ ] earl 进场（mic_only invisibility）：UserJoined 正常广播；NobleEntered 半屏特效广播
- [ ] king 进场（all invisibility）：UserJoined 仍广播但 noble 字段对非 admin 接收者被剔除；NobleEntered 仅 admin 收到
- [ ] duke 第一次进场：触发 NobleEntranceGlobal Pub/Sub；多实例订阅器都收到
- [ ] duke 当日第二次进场：daily_limit 命中，不再广播全服
- [ ] privileges JSON 缺失字段 → 用 default 兜底（如 entry_effect.duration_ms 缺失 → 5000）

### 🔴 协议路径绑定
- [ ] **PROTO-1**：UserJoined.payload.noble 子对象字段严格 `tier_id/level/badge_color/frame_url/expire_at`
- [ ] **PROTO-2**：NobleEntered.payload 字段严格匹配 §10.4.5；scope 取值 `marquee|half|fullscreen`
- [ ] **PROTO-3**：NobleEntranceGlobal Pub/Sub channel 严格 `noble:global`，payload 严格匹配 §10.4.6

### Round 2（2026-05-10，Reviewer: everything:code-reviewer）
- **状态**：❌ 打回
- **打回理由**：

#### [P0] NobleEntranceGlobal 完全缺失——无 stub，无接口

**文件**：全仓库 `app/server/src/` 中搜索 `NobleEntranceGlobal`，仅在 `privileges.rs:90` 有一行文档注释。

协议 §10.4.6 是 Server 端必须实现的信令。当前实现中：
- 无 `global_pubsub` 模块或任何 pub/sub 接口 trait
- 无 Redis `PUBLISH noble:global NobleEntranceGlobal` 调用（即使是占位符/stub）
- 无 `noble_global_broadcast_log` INSERT 逻辑（每日频控记录）
- `can_trigger_global_broadcast(level)` 仅是一个检查 `level >= 5` 的布尔辅助函数，从未在 JoinRoom 流程中被调用

**延后判断标准**：「如果完全缺失（非 stub/占位符），判 P0 打回；如果有功能完整的 stub（接口已定义，Redis 调用已写但运行时降级），可接受为 P1。」

本次实现属于「完全缺失」——P0 打回。

**最低可接受修复**：新增 `global_pubsub::try_broadcast_noble_entrance_global(user_id, level, registry, redis_url)` async fn，内部：①查 `noble_global_broadcast_log` 是否已发（stub: 直接跳过查询，tracing::info! 占位）→ ②`redis.publish("noble:global", payload)` stub（可降级为仅 tracing::warn!）。接口已定义即满足 P1 条件。

#### [HIGH] NobleEntered payload 缺少 `scope` 等关键协议字段

**文件**：`app/server/src/room/handler/lifecycle.rs`（NobleEntered envelope 构造段）

协议 §10.4.5 schema 明确要求 `NobleEntered.payload` 包含：
- `scope`: `"marquee" | "half" | "fullscreen"`（源自 `privileges.entry_effect.scope`）
- `duration_ms`（源自 `privileges.entry_effect.duration_ms`）
- `entrance_animation_url`、`bgm_url`、`marquee_text_en`、`marquee_text_ar`

当前实现将 `UserNobleDto`（仅含 `tier_id/level/badge_color/frame_url/expire_at`）内嵌在 `"noble"` 字段中，缺失以上所有字段。客户端收到信令后无法确定应播放 marquee/half/fullscreen 哪种动画类型，进场特效功能不可用。

测试 `t69_noble_entered_broadcast_for_lv3_plus` 仅断言信令 `"type" == "NobleEntered"` 存在，未校验 payload 结构。

**修复方向**：将 `entry_effect` 字段（scope/duration_ms）加入 `UserNobleDto` 或专门为 NobleEntered 构造含以上字段的 payload；同步更新测试断言。
