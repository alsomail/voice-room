# TDS: 协议路径绑定审计脚本 (Task ID: T-0000T)

## 一、背景与目标

### 关联需求与根因背景

- **BUG-CHAT-WS 系统性根因复盘**：2026-05-05 Round 16 复盘揭示，`BUG-CHAT-WS` 的根本原因之一是「协议路径未落锚文档」——Android 客户端与 Server 双方在 `SendMessage` 信令的字段名称、调用路径上存在静默漂移（`wsClient.connect` 从未在生产路径被调用，但无任何自动化机制检测此缺陷），导致 Round 1~15 反复修复、反复回归，浪费约 15 轮 E2E 验证资源。
- **协议治理铁律 v2.83 落地**：`copilot-instructions.md` 红线 #7「协议路径绑定（最高优先级）」已在 v2.83 批次生效，规定每个跨端 Task 必须填写「🔌 协议路径绑定表」。T-00047（首个试跑 Task）已验证该流程，T-00048 完成了双路径等价回归测试。
- **现状缺口**：绑定表目前仅凭人工填写 + Review 目测核查，无法在 CI 阶段自动发现「TDS 声明了绑定但服务端未实现对应 Route」「客户端调用了 TDS 未声明的路径」「字段名与 protocol/ 锚点不一致」等 P0 问题。
- **价值定位**：本 Task 交付一个自动化审计脚本 `scripts/audit/protocol-binding-audit.ts`，实现 **TDS 绑定表 ↔ server 实现 ↔ client 调用** 三角对账，并接入 `npm run audit:protocol` CI 门禁，将协议路径一致性校验从「人工目测」升级为「自动化阻断」。

### 本 Task 目标

新建 `scripts/audit/protocol-binding-audit.ts`：从所有 TDS 文件中解析「协议路径绑定表」，与 server 侧路由注册、WS 信令分发、client 侧真实调用进行三角对账，输出 P0/P1 分级报告，并在发现 P0 问题时以非零退出码阻断 CI 合并。

##### 2.4 Markdown 表格解析

```
标准格式（来自 T-00047.md §二 协议路径绑定表）：
| # | 协议类型 | 入口 / 信令名 | 客户端调用方（实文件路径 + 函数）| 服务端处理函数（实文件路径 + 函数）| 广播 / 响应 | protocol/ 锚点 |
|---|---------|--------------|----------------------------------|-----------------------------------|------------|---------------|
| 1 | WS C→S  | `SendMessage` ⭐ | app/android/.../RoomViewModel.kt::sendMessage | app/server/src/room/handler/chat.rs::handle_send_message | ... | websocket_signals.md §6.8.1 |

解析步骤：
1. 找到以 | # | 协议类型 开头的表头行
2. 跳过分隔行（| --- |）
3. 逐行解析，拆分 | 分隔的单元格
4. 提取字段：rowId, protocolType, signalOrPath, clientRef, serverRef, broadcastOrResponse, protocolAnchor
5. 识别 ⭐ 标记（主路径标记）
```

#### 4. 三角对账逻辑

##### 4.1 对账流程

对每条 `BindingRow`（来自 TDS 绑定表），执行以下校验：

```
对 BindingRow[i]:
  protocolType = row.protocolType   // e.g. "WS C→S", "HTTP REST"
  signalOrPath = row.signalOrPath   // e.g. "SendMessage", "POST /api/v1/chat-messages"
  clientRef    = row.clientRef      // e.g. "RoomViewModel.kt::sendMessage"
  serverRef    = row.serverRef      // e.g. "app/server/src/room/handler/chat.rs::handle_send_message"
  isMainPath   = row.hasStarMark    // ⭐ 标记

校验 ①：server 实现存在性
  expected: serverRef 指向的文件存在，且 grep 结果中有对应函数名命中
  failure → P0 issue: "server_impl_missing"

校验 ②：client 调用存在性
  if (clientRef != "目前无客户端" && clientRef != "（目前无客户端）"):
    expected: clientRef 指向的文件存在，且 grep 结果中有对应调用命中
    failure → P0 issue: "client_call_missing"

校验 ③：主路径 ⭐ 互斥校验
  if isMainPath:
    expected: 同一客户端 × 同一业务功能下，⭐ 行只有一条
    if 该信令/路径在 grep 结果中有 "必须 0 命中" 的备路径出现:
      failure → P0 issue: "forbidden_alt_path_used"

校验 ④：protocol/ 锚点存在性
  expected: row.protocolAnchor 指向的文件（如 websocket_signals.md §6.8.1）在 doc/protocol/ 目录存在
  failure → P1 issue: "protocol_anchor_broken"

校验 ⑤：字段命名一致性（best-effort）
  在 protocolAnchor 指向的协议文件中，检查 signalOrPath 字符串是否出现
  failure → P1 issue: "signal_name_mismatch"
```

##### 4.2 缺失 / 不一致判定规则

| 场景 | 严重度 | 类型 | 触发条件 |
|------|--------|------|----------|
| TDS 声明了绑定行，但 server 函数文件不存在 | **P0** | `server_impl_missing` | `fs.existsSync(serverFile)` 为 false |
| TDS 声明了绑定行，但 grep 找不到对应函数名 | **P0** | `server_impl_not_found` | grep 结果零命中 |
| TDS 声明了有效客户端调用方，但 grep 找不到 | **P0** | `client_call_missing` | grep 结果零命中 |
| ⭐ 主路径的「禁用备路径」出现在客户端代码中 | **P0** | `forbidden_alt_path` | 备路径 grep 命中 |
| TDS 无绑定表且无 N/A 声明（潜在漏填） | **P1** | `binding_table_absent` | 无表格 + 无 NA_PATTERNS 命中 |
| protocol/ 锚点文件不存在 | **P1** | `protocol_anchor_broken` | 文件不存在 |
| 信令名在 protocol/ 中找不到 | **P1** | `signal_name_mismatch` | grep protocol 文件零命中 |
