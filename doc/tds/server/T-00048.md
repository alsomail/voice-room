## 三、TDD 验收用例

### DUAL-1（双 WS 观察者均收到等价 envelope）
- **前置**：两个 WS 观察者 A/B 同时连接同一房间（通过 register_conn），一个 WS 发送者，一个 REST 发送者
- **步骤**：
  1. WS 发送者发一条消息（handle_send_message）
  2. REST 发送者发一条消息（POST /api/v1/chat-messages）
  3. 观察者 A/B 分别收到两条 RoomMessage
- **验证**：WS 发的 envelope vs REST 发的 envelope（在同一观察者视角）除 `envelope.msg_id`/`payload.msg_id` 外逐字段相等
- **通过条件**：A/B 两个观察者均通过字段比对

### DUAL-2（payload 字段逐项断言）
- **验证字段**：
  - `payload.msg_id`：合法 UUID v4（Uuid::parse_str 不报错），WS 路径和 REST 路径的值互不相同
  - `payload.content`：两路径相等（均经 filter_content 处理）
  - `payload.timestamp`：int64 ms（值允许不同，但均 > 0）
  - `payload.user_id`：两路径相等
  - `type`：均等于 "RoomMessage"
- **通过条件**：所有字段断言均通过

### DUAL-3（死连接清理逻辑一致性）
- **步骤**：
  1. 关闭一个 WS 观察者连接（不优雅断开，直接 drop receiver）
  2. 通过 WS 路径发消息 → 确认服务器不 panic、不阻塞（有存活观察者收到消息）
  3. 通过 REST 路径发消息 → 确认服务器不 panic、不阻塞（存活观察者收到消息）
- **通过条件**：两次发消息均完成，无 panic，存活观察者均收到消息
