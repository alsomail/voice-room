# TDS: JWT 拦截器 (Task ID: T-30003)

## 一、背景与目标
- **关联需求**：所有需要认证的 C 端 API
- **本 Task 的目标**：实现 OkHttp AuthInterceptor，自动为每个请求附加 JWT token，并处理 401 未授权响应（清除 token + 跳转登录页）。
- **前置依赖**：T-00004 (JWT 中间件), T-30002

## 二、方案设计
### 核心数据流
```
OkHttp Request → AuthInterceptor → TokenManager.getToken()
  → 有 token？→ 添加 Authorization: Bearer xxx Header → 发送请求
  → 收到 401？→ TokenManager.clearToken() → 跳转登录页
  → 无 token → 直接发送请求（公开接口如 send-code/login）
```

### HIGH 问题（阻塞合并）

#### H-01：并发 401 竞态条件 — `DefaultUnauthorizedHandler` 无原子性保护

**文件：** `DefaultUnauthorizedHandler.kt:42-45`

**问题描述：**  
当应用内存在多个并发 OkHttp 请求同时收到 401 响应时（例如进入大厅后批量发起的多个 API 请求），每个 OkHttp 后台线程都会独立执行：

```kotlin
// AuthInterceptor.kt:50 — 每个 OkHttp 线程各自 runBlocking
if (response.code == 401) {
    runBlocking { unauthorizedHandler.onUnauthorized() }
}
```

而 `onUnauthorized()` 内部没有任何并发保护：

```kotlin
// DefaultUnauthorizedHandler.kt:42-45 — 无互斥，可被多线程并发执行
override suspend fun onUnauthorized() {
    tokenManager.clearToken()        // 多次调用：虽然幂等，无害
    _unauthorizedEvent.tryEmit(Unit) // ⚠️ 多次调用：可能向 UI 多次发射导航事件
}
```

`_unauthorizedEvent` 的 `extraBufferCapacity=1 + DROP_OLDEST` 仅能保证在**同一时刻缓冲区满**时丢弃旧事件。若订阅方（ViewModel）处理速度快于后续 `tryEmit` 到达速度（实际场景中极易发生），多个事件均会被消费，导致 `navController.navigate("login") { popUpTo(0) { inclusive = true } }` 被执行多次，引发导航栈异常（重复弹出 / 崩溃）。

**修复方案：** 使用 `AtomicBoolean` 实现 compare-and-set 一次性语义，或使用 `Mutex` + 布尔标记：

```kotlin
// 方案A：AtomicBoolean（非挂起，适合跨线程场景）
private val handled = AtomicBoolean(false)

override suspend fun onUnauthorized() {
    if (!handled.compareAndSet(false, true)) return  // 后续调用直接忽略
    tokenManager.clearToken()
    _unauthorizedEvent.tryEmit(Unit)
}
```

> ⚠️ 注意：`AtomicBoolean` 需在用户重新登录成功后重置（`handled.set(false)`），否则下次 Token 失效时无法再触发登出。建议将重置逻辑放入 `ITokenManager.saveToken()` 调用成功后，或通过 DI 在登录成功后重建 `DefaultUnauthorizedHandler` 实例。

#### H-02：缺少并发 401 测试 — 现有测试仅覆盖顺序调用，竞态分支无回归

**文件：** `DefaultUnauthorizedHandlerTest.kt:69-91`

**问题描述：**  
现有测试 `multiple onUnauthorized calls each emit one event` 测试的是**串行**两次调用，且断言期望收到 2 个事件（`assertEquals(2, receivedEvents.size)`）——这恰好**验证了竞态的存在**，而非验证了竞态被消除。

`AuthInterceptorTest.kt` 完全没有多线程并发场景测试。

**必须补充的测试：**

```kotlin
@Test
fun `concurrent 401 responses trigger onUnauthorized exactly once`() = runTest {
    val tokenManager = FakeTokenManager()
    val handler = DefaultUnauthorizedHandler(tokenManager)

    val receivedEvents = mutableListOf<Unit>()
    val collector = launch {
        handler.unauthorizedEvent.collect { receivedEvents.add(it) }
    }
    runCurrent()

    // 模拟多个并发 OkHttp 线程同时触发 401
    val jobs = (1..5).map {
        launch(Dispatchers.Default) { handler.onUnauthorized() }
    }
    jobs.forEach { it.join() }
    runCurrent()
    collector.cancel()

    assertEquals(
        "Concurrent 401s must trigger logout exactly once",
        1, receivedEvents.size
    )
    assertEquals(
        "clearToken() must be called exactly once",
        1, tokenManager.clearTokenCallCount
    )
}
```

### MEDIUM 问题（建议修复）

#### M-01：`DefaultUnauthorizedHandler` 未强制单例，DI 误配置可导致事件通道断开

**文件：** `DefaultUnauthorizedHandler.kt:26`

**问题描述：**  
`DefaultUnauthorizedHandler` 是普通 `class`，没有任何 Singleton 作用域注解或文档约束。若 DI 框架（Hilt/Koin）配置时以 non-singleton 方式注入，`AuthInterceptor` 持有的实例 A 发射事件，而 UI/ViewModel 订阅的是实例 B，导致 401 事件永远不会到达 UI，用户 Token 失效后无法自动跳转登录页（静默失效，严重体验 Bug）。

**修复方案：** 在类 KDoc 中明确标注单例要求，并在 DI 模块中以 `@Singleton`（Hilt）或 `single { }` block（Koin）提供：

```kotlin
/**
 * ...
 * ⚠️ **必须以应用级单例注入**（Hilt `@Singleton` 或 Koin `single { }`）。
 * AuthInterceptor 与 UI 层必须共享同一实例，否则事件通道断开。
 */
class DefaultUnauthorizedHandler(...)
```

现有测试仅覆盖 `""` (empty)，但未覆盖 `"   "` (whitespace-only)。`isNullOrBlank()` 对两者均返回 true，行为一致，但添加 `"   "` 测试用例可明确记录此约定，防止未来有人误改为 `isNullOrEmpty()`。

#### L-02：`AppHttpClientFactory` 为 `internal object`，跨模块扩展受限

**文件：** `AppHttpClientFactory.kt:19-21`

单模块内可接受，但未来拆多模块时（如独立的 `:core:network` 模块），`internal object` 无法被其他模块注入或测试替换。建议后续拆模块时改为可注入的 `class` 并通过 DI 管理。当前 MVP 阶段不阻塞。
