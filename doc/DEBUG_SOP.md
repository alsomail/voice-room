# DEBUG_SOP — AI Agent 动态排障与防御性编程操作系统

> **适用对象**：具备 MCP 工具、终端执行能力、代码读写能力的 AI 编程助手  
> **执行级别**：强制  
> **核心要求**：遇到故障时必须按本 SOP 执行；写代码时必须主动植入可观测性与防御性机制；严禁靠猜测、严禁盲改、严禁静默吞错。

---

## 0. 总纲：你的首要职责不是“尽快改代码”，而是“定位真实根因”

当你遇到编译失败、运行时崩溃、接口超时、状态错乱、弱网异常、数据不一致、第三方 SDK 行为异常时，**必须先建立证据链，再做修改**。

你必须遵守以下总原则：

1. **必须先观察，再假设，再验证，再行动。**
2. **必须优先使用终端、MCP、网络面板、数据库客户端等外部工具获取事实。**
3. **严禁只读最后一行报错就动手改代码。**
4. **严禁一次改多个地方来“碰运气”。**
5. **严禁在缺少证据时臆造 API、配置项、协议行为或框架用法。**
6. **必须默认系统是复杂的，错误可能发生在代码、配置、网络、环境、依赖、数据、权限、时序任一层。**

## 1. 科学排障第一性原理 (The Scientific Method)

面对任何问题，你必须严格执行 **OODA 循环**：**Observe -> Hypothesize -> Verify -> Act**。

### 1.1 Observe：先拿全量事实，严禁只看“报错摘要”

你必须主动收集完整上下文，至少覆盖以下维度：

1. **完整日志**
   - Android / 移动端：`adb logcat`
   - 容器：`docker logs`
   - 编译器：`rustc --explain`、构建输出、完整栈追踪
   - 服务端：应用日志、反向代理日志、网关日志。**（请注意：Rust 后端的日志已通过 `tracing` 结构化输出，查询时必须带上 `request_id` 或 `msg_id` 进行关联检索。）**
2. **真实请求/响应**
   - 请求 URL、Method、Headers、Body
   - 响应状态码、响应体、响应耗时
   - 是否发生重试、超时、重定向、CORS、TLS 失败
3. **真实数据状态**
   - 数据库当前记录
   - 缓存值
   - 队列堆积
   - 会话 / Token / 房间状态 / 余额状态
4. **真实运行环境**
   - 当前机器 IP、端口监听、环境变量、DNS、代理、证书、时钟偏差
   - 本地 / 测试 / 预发 / 生产是否一致

你必须落实以下行为：
- **必须查看完整日志上下文，而不是只看最后一行。**
- **必须确认错误首次出现在哪一层，不得先入为主。**
- **必须记录“预期行为”和“实际行为”的差异。**
- **必须识别问题是稳定复现、概率复现，还是只在特定输入/网络/账号下出现。**

### 1.2 Hypothesize：必须发散性提出至少 3 个假设

在未验证前，任何结论都只是候选假设。你必须至少提出 **3 个相互独立** 的可能性，示例：

1. **网络层问题**
   - DNS 错误、端口不通、代理拦截
   - Android 明文 HTTP 被拦、TLS 校验失败
2. **环境配置问题**
   - `.env` 错位、base URL 错误
   - dev/prod 配置串用、依赖版本不兼容
3. **数据层问题**
   - 脏数据、锁等待、事务未提交
   - 外键约束失败、缓存未失效
4. **代码逻辑问题**
   - 空值未处理、并发竞态、状态机分支遗漏
   - 超时处理缺失、错误被吞掉
5. **第三方依赖问题**
   - SDK 回调未触发、API 参数变更
   - 限流、权限不足、上游服务降级

你必须做到：
- **必须把“代码错了”视为众多假设之一，而不是默认答案。**
- **必须优先考虑网络、环境、数据、权限、时序、依赖这些非纯代码因素。**
- **严禁在未列出替代假设前直接改业务代码。**

### 1.3 Verify：修改代码前，必须先做单点验证

在改项目代码之前，你必须优先使用外部工具验证单个假设：

1. **接口问题**：用 `curl` / Postman 裸调接口，对比客户端请求差异。
2. **数据库问题**：用 `psql` / MySQL 客户端直接查真实数据。
3. **网络问题**：用 `ping`、`curl`、`nc` 等验证连通性。
4. **进程/端口问题**：检查端口是否被占用，服务是否监听正确地址。
5. **配置问题**：打印实际生效配置，验证环境变量注入。

你必须遵守：
- **必须先验证外部世界，再决定是否修改代码。**
- **必须让每一次验证只服务于一个明确假设。**
- **严禁一边猜、一边改、一边重新编译来试错。**

### 1.4 Act：单一变量原则

行动时必须遵守：
1. **每次只修改一个变量**（只改一处配置/分支/日志等）。
2. **每次修改后必须重新观察结果**。
3. **必须明确这次修改是在验证哪一个假设**。
4. **若结果无改善，必须回到 Observe，而不是连改第二处**。

严禁：
- 同时改接口地址、鉴权逻辑、重试策略。
- 一次性大面积重构后再说“应该好了”。

## 2. 精准日志与可观测性强制规范 (Precision Logging)

> **重点规则**：你必须克服“少打日志”“懒得埋点”“吞错返回默认值”的倾向。写代码时，必须默认未来排障者就是你自己。

### 2.1 Must-Log Nodes：关键节点必须记录

#### 2.1.1 跨越系统边界处
必须记录 HTTP 发出与响应、DB 事务起止、SDK 调用、WebSocket 建连与断开。
日志至少包含：目标、参数摘要、关联键（trace_id/room_id）、状态、耗时、错误对象。

#### 2.1.2 核心状态流转
必须记录鉴权结果、进出房间、麦位切换、资金变动。
日志必须体现：谁触发、什么对象变化、状态跳变轨迹、触发条件。

#### 2.1.3 异常捕获点
所有 `catch`、`Err`、失败回调分支：
- **必须记录错误堆栈与触发参数**
- **严禁静默吞没异常（如 `catch (e) {}` 或直接返回默认值）**

### 2.2 Strict Log Levels：日志级别必须严格控制

- **ERROR**：仅用于需人类介入的系统故障、资金不一致。严禁将预期内的业务阻断标为 ERROR。
- **WARN**：可恢复异常、风控拦截、弱网重试。
- **INFO**：业务关键里程碑（进房、支付成功）。
- **DEBUG**：排障追踪、详细状态机输出。

### 2.3 日志内容质量要求

关键日志必须具备：可定位（模块/函数名）、可关联（上下文 ID）、可比较、可搜索、可脱敏（严禁明文密码或完整 Token）。

### 2.4 高频防刷屏机制 (Anti-Spam & Throttling)

对于高频事件（WebSocket 心跳、音量回调、UI 滚动、**中东弱网下的批量埋点上报重试**等），**严禁使用 INFO/ERROR 级别狂打日志**。

必须采用以下机制之一：
1. 降级为 DEBUG 级别。
2. 采样输出（如每 100 次打印一次）。
3. 状态跳变输出（仅当状态真正改变时打印）。
4. 时间窗口限流或聚合摘要输出。

### 2.5 防御性代码内建要求

- 必须对外部请求设置超时。
- 仅对幂等操作进行有限重试。
- 必须有熔断/降级策略。
- 关键资源（连接、事务、锁）释放时必须可观测。

## 3. 动态故障隔离策略 (Dynamic Fault Isolation)

### 3.1 端到端断点法

必须按真实拓扑切分：`UI -> 网络层 -> 代理 -> 网关 -> 服务 -> 数据库`。
确认每一段的请求和响应，严禁跳过中间层直接下结论。

### 3.2 环境嗅探机制

遇到网络或联调问题，必须主动嗅探真实环境：
1. 检查本机 IP、网络限制。
2. **确认当前是 Dev 还是 Prod 环境，不要把架构文档中规定的 Nginx 网关地址与 Vite Proxy 本地代理地址搞混。**
3. 确认移动端的 `localhost`、`10.0.2.2` 差异。

### 3.3 复杂系统的隔离顺序
先分层，再分环境，再分账号/时间窗口，最后才进入代码行级分析。

## 4. MCP 工具的自由组合 (Autonomous Tool Chaining)

### 4.1 源码探视
遇到陌生库，必须先查 `Cargo.toml`/`package.json` 等依赖声明，再查对应版本的官方注释。严禁凭记忆编造 API。

### 4.2 全局搜索
遇到诡异错误码或异常，必须主动在工作空间全局搜索抛错源头或同类日志。

### 4.3 状态快照
处理并发、房间状态、资金变化时，必须对比触发前、触发动作、触发后的快照差异。

---

## 5. 防呆与防死循环机制 (Guardrails & Fallbacks)

### 5.1 拒绝幻觉
如果在文档、日志、网包或数据库中找不到依据，**严禁**编造隐藏参数或臆断第三方行为。必须退回基础验证。

### 5.2 三次失败停手规则
如果出现以下情况，**必须立即停手**：
1. 修改 3 次仍未解决。
2. 连续编译 3 次失败。
3. 故障现象持续变化。

### 5.3 标准《排障诊断报告》模板
触发停手规则时，必须向人类输出如下报告：
- **问题现象**（预期 vs 实际）
- **已观察到的事实**（日志/网络/数据）
- **已验证的假设与结论**
- **当前卡点与缺失的证据**
- **需要人类提供的额外协助**

### 5.4 严禁破坏性重试
严禁为了“先跑起来”而删掉鉴权或事务，严禁用默认值掩盖底层错误。

---

## 6. AI Agent 代码生成期强制附加条款

1. 所有失败路径必须有日志或错误传播。
2. 关键路径必须传递 `trace_id` / `room_id`。
3. 资金、房间状态变更必须有审计轨迹日志。
4. fallback 必须带日志并标明原因。

## 7. 执行清单（每次动手前必须默念）

1. 我是否拿到了完整日志，而不是只看报错最后一行？
2. 我是否提出了至少 3 个独立假设？
3. 我是否用外部工具验证过假设？
4. 我这次修改是否只改变了一个变量？
5. 我是否为关键路径补上了日志并避免了刷屏？
6. 如果再失败，我是否会主动停手输出诊断报告？

**如果任一答案为否，必须先补齐，再继续行动。**

## 8. 一句话总训

> **先证据，后判断；先验证，后修改；先定位根因，后提交修复。没有日志的系统不可维护，没有证据链的修复不可信，没有边界意识的 AI 不配做排障。**
