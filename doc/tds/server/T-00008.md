## 接口设计

### `GET /api/v1/rooms`

**认证**：无需 JWT（公开接口）

**查询参数**

| 参数 | 类型 | 默认 | 验证规则 |
|------|------|------|----------|
| `page` | `i64` | `1` | `>= 1` |
| `size` | `i64` | `20` | `1 – 100` |

**成功响应** `HTTP 200`

```json
{
  "code": 0,
  "message": "ok",
  "request_id": "<uuid>",
  "data": {
    "total": 42,
    "page": 1,
    "size": 20,
    "items": [
      {
        "room_id": "uuid",
        "title": "Room Name",
        "room_type": "normal",
        "member_count": 25,
        "max_members": 50,
        "owner_id": "uuid",
        "owner_nickname": "Alice",
        "owner_avatar": "https://cdn.example.com/avatar.jpg",
        "created_at": "2024-01-01T00:00:00Z"
      }
    ]
  }
}
```

**错误响应**

| HTTP | code | 场景 |
|------|------|------|
| 400 | 40003 | page < 1 / size < 1 / size > 100 |

---

## 排序规则

1. `member_count DESC`（热度优先）
2. `created_at DESC`（同热度时最新创建的在前）

**过滤条件**：`status = 'active' AND deleted_at IS NULL`

### 审查结果汇总

| ID | 级别 | 问题描述 | 建议处理方式 |
|----|------|----------|-------------|
| M-01 | MEDIUM | `page` 无上界：当 `page ≥ 92_233_720_368_547_760` 时 `(page-1)*size` 溢出 i64，Debug 模式 panic（DoS），Release 模式 OFFSET 为负 → PostgreSQL 报错 → 500 | 在 service.rs 校验区新增 `if page > 100_000 { return Err(ValidationError) }` |
| M-02 | MEDIUM | `JOIN users` 未过滤 `u.is_banned = true`：被封禁用户的活跃房间仍出现在公开列表中 | SQL 加 `AND u.is_banned = false`，或封禁用户时同步关闭其房间 |
| L-01 | LOW | `count_active_rooms` 与 `find_active_rooms` 两次查询非原子，`total` 可能与 `items` 数量不一致 | 添加注释说明此为已知设计决策，可接受 |
| L-02 | LOW | `JOIN users` 未过滤 `u.deleted_at IS NULL`，软删除用户的房间信息可能显示过期数据 | 建议 `JOIN users u ON u.id = r.owner_id AND u.deleted_at IS NULL` |

### 核查通过项

- ✅ **公开接口设计合理**：`list_rooms` 无 JWT，符合"先浏览后注册"产品逻辑，L-24 集成测试覆盖
- ✅ **SQL 过滤完整**：`count` 和 `find` 均包含 `status='active' AND deleted_at IS NULL`
- ✅ **OFFSET 公式正确**：`(page-1)*size`，page=1 时 offset=0
- ✅ **参数校验全覆盖**：page=0 / size=0 / size=101 / size=200 均返回 ValidationError（L-07~L-09 覆盖）
- ✅ **默认值正确**：`page=1, size=20`，L-05 覆盖
- ✅ **排序正确**：`ORDER BY member_count DESC, created_at DESC`，L-01/L-13 覆盖

### 结论

M-01、M-02 建议在 **T-00009 开工前**修复（不阻塞当前合并）。L-01/L-02 可在后续技术债清理中处理。
