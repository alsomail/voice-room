# TDS: 协议字段全量冻结 — WS + HTTP REST + Redis Pub/Sub 三协议层契约权威化 (Task ID: T-00100)

## 一、背景与目标

- **关联需求**：Round 16 BUG-CHAT-WS 系统性复盘 → 三协议层全部存在「server/客户端各写各的、字段失配靠人肉对账、无机器可读 schema、无 CI 守门」的根因（详见 v2.83 协议铁律仅覆盖文档层，未覆盖字段冻结层）。
- **本 Task 的目标**：把 `doc/protocol/` 从「描述性文档」升级为「机器可读的字段级单一事实源」：
  1. **WS 信令**：补齐 21+ 缺失信令（Ping/Pong/JoinRoom/JoinRoomResult/UserJoined/UserLeft/LeaveRoom/LeaveRoomResult/TakeMic/TakeMicResult/MicTaken/LeaveMic/LeaveMicResult/MicLeft/SendMessageResult/SendGiftResult/EventReportAck/KickUser/MuteUser/UserMuted/UnmuteUser/TransferAdmin/AdminChanged/ForceTakeMic/ForceLeaveMic/RoomInfoUpdated）的字段表 + JSON 示例 + JSON Schema。
  2. **HTTP REST**：所有 endpoint 的 request/response DTO 字段表 + 错误码 envelope + JSON Schema（auth_api / room_api / admin_api / ranking_api / providers）。
  3. **Redis Pub/Sub**：`admin:events` 频道事件家族（BanUser/UnbanUser/CloseRoom/BroadcastNotice 等）字段表 + JSON Schema，**首次落锚**（此前完全不在 protocol/ 中）。
  4. **conventions.md** 三章节强制铁律：§4 字段命名 snake_case 全协议强制 + §5 WS 业务字段必须放 `payload` 嵌套 + §6 envelope 双 ID（`msg_id` / `timestamp`）。
  5. **ping/pong → Ping/Pong 升级**：与其他信令 PascalCase 对齐（用户决策 #2）。
  6. **新增 `doc/protocol/schemas/*.schema.json`**：每信令/每 endpoint 一份 JSON Schema 2020-12，作为 T-00101/T-00102/T-00103/T-00105 的代码生成与运行时校验输入。

### 关键设计决策

| # | 决策 | 选择 | 理由 |
|---|------|------|------|
| D1 | 字段命名 | 全协议 snake_case + WS 业务字段嵌入 `payload` | 用户决策 #1；保留 server 现状，最小化改动面 |
| D2 | ping/pong | 升级 PascalCase `Ping`/`Pong` | 用户决策 #2；与其他 28 信令统一 |
| D3 | Schema 标准 | JSON Schema 2020-12 | 业界标准、Kotlin/TS/Rust 生态有成熟 codegen |
| D4 | Pub/Sub 锚点 | 落入 `providers.md` 而非新建文件 | 与 RTC providers 同一类「外部基础设施」，语义内聚 |
| D5 | 兼容期 | Ping/Pong 升级双发 1 个版本 | server 出栈同时发 `Ping` + `ping`；Android 双收；下个版本只发 `Ping` |

### Round 1
- **状态**：🟡 有条件通过（WARNING）— P1 问题须在下游 T-00101/T-00102 启动前修复
- **审查意见**：
  整体框架设计完整，三铁律条文清晰，34 份 WS schema 文件格式合法（JSON Schema 2020-12，`additionalProperties: false` 100% 覆盖），8 个自动化验收标准全部通过。pubsub 四事件首次落锚，mic_slots 强类型化，conventions.md 三铁律章节到位，质量属于高水平文档化工作。
  发现 3 个 P1 级别内部一致性问题（见"改进建议"），不影响当前实现，但会阻断下游代码生成任务。
- **改进建议**：
  - **[P1-1] 必修 — `UserJoined.schema.json` avatar 类型须允许 null**
    `websocket_signals.md §6.7.1` 已写 `"string | null"`，server `avatar: Option<String>` 可为 null，但 schema 仍为 `"type": "string"`。下游 T-00101/T-00102 按 schema 生成类型后，无头像用户加入房间将触发 decode 失败。
    修复：`"avatar": { "type": ["string", "null"], "format": "uri" }`
  - **[P1-2] 必记录 — timestamp 单位：conventions.md §6 写"Unix ms"但 server 大多数 handler 实际用秒**
    `mic.rs`/`lifecycle.rs` 全量用 `timestamp()`（秒），仅 `chat.rs:149` 用 `timestamp_millis()`（毫秒）。建议在 conventions.md §6 补注：「当前 server 实现待 T-00108 统一改为 ms；过渡期客户端应兼容秒/ms 双格式」，或由 T-00108 先修复 server 再冻结文档。
  - **[P1-3] 建议修改 — conventions.md §6 声称"S→Room 广播必须携带 msg_id"但 schema 和 server 均不发**
    建议在 §6 S→Room 行补注「msg_id 待 T-00108 补齐；当前实现暂未携带」，消除文档内部矛盾。
  - **[P2-1] 建议 — validate-protocol-freeze.sh PROTO-FREEZE-8 深度不足**
    当前脚本仅 grep 字段名存在，未检查类型。P1-1 的 avatar nullable 问题通过了 8/8 PASS，说明脚本强度不够。建议 T-00106 集成 AJV CLI 对 fixture 做实际类型校验。
  - **[P2-2] 建议 — index.md 文件数改为 "34 个"（28 核心 + 6 附加）**
