# TDS: BUG-MIC-ONCLICK — Android 上麦 UI 调用链断裂 (Task ID: T-30055)

## 一、背景与目标

- **关联需求**：E-05 麦位管理（T-30012 上下麦 UI）
- **本 Task 的目标**：修复 `RoomScreen` / `AppNavGraph` / `MicPermissionHandler` 三处调用链断裂，使点击自己已占麦位图标能触发 ViewModel 的 `onMicSlotClick()` → `leaveMic()` → WS `LeaveMic` 帧发出。

### 根因（静态分析）

| # | 文件 | 问题 |
|---|------|------|
| 1 | `feature/room/RoomScreen.kt` | `@Composable fun RoomScreen(...)` 缺少 `onMicSlotClick: (Int) -> Unit` 参数，`MicSlotCard` 的 onClick 只能路由到 `onMicPermissionGranted`（上麦流程） |
| 2 | `navigation/AppNavGraph.kt` | `RoomScreen(...)` 调用缺少 `onMicSlotClick = { roomViewModel.onMicSlotClick(it) }` |
| 3 | `core/mic/MicPermissionHandler.kt` | `onMicSlotClick` lambda 对**所有**点击均调用 `onPermissionGranted`，无法区分"空麦位上麦"与"自己麦位下麦" |
| 4 | `feature/room/RoomViewModel.kt` | `onMicPermissionGranted` 无条件发 `TakeMic`，未先检查该槽是否已被自己占用 |

> `RoomViewModel.onMicSlotClick(slotIndex)` 已正确实现 `leaveMic()` 逻辑，只是从未被调用。

## 二、方案设计

### 核心数据流（修复后）

```
MicSlotCard.onClick(slotIndex)
        │
        ├─ slotState == EMPTY  ─────────────→ MicPermissionHandler.requestPermission(slotIndex)
        │                                              │ granted → RoomViewModel.onMicPermissionGranted(slotIndex) → TakeMic
        │
        └─ slotState == OCCUPIED_BY_SELF ──→ RoomViewModel.onMicSlotClick(slotIndex) → leaveMic(slotIndex) → LeaveMic
```

### ⚠️ MEDIUM-1：客户端新增的 `payload.mic_index` 与现有协议文档/集成测试存在文字出入

**文件**：`app/android/app/src/main/java/.../RoomViewModel.kt:828`  
**关联**：`tests/scripts/API/TC-MIC.spec.ts:8`、`tests/cross-lang/android-server-ws/CROSS-4-leave-mic.spec.ts:95,158`、`doc/protocol/schemas/ws/LeaveMic.schema.json:12`

**问题描述**：

本次修复在 Android 端的 `LeaveMic` 帧中新增了 `payload.mic_index`，并以 `PROTO-BINDING` 注释指向 `LeaveMic.schema.json`。但横向检查发现以下三处与此相悖：

1. `TC-MIC.spec.ts` 文件头注释明确写道：
   ```
   LeaveMic: { type:'LeaveMic', msg_id }  (no seat param; releases sender's own slot)
   ```
   且测试代码中所有 `LeaveMic` 调用均不携带 `payload`（L65、L73、L90、L199）。

2. `CROSS-4-leave-mic.spec.ts`（交叉语言集成测试）发送 `{ type: 'LeaveMic' }`，同样无 `payload`（L95、L158）。

3. `LeaveMic.schema.json` 的 `payload` 定义为 `{ "type": "object", "additionalProperties": true }`，**未将 `mic_index` 声明为任何属性**——schema 对此字段既不要求也不文档化。

服务端 `handle_leave_mic` 也从未读取 `payload.mic_index`（其通过 `leave_mic_slot(user_id)` 从连接上下文推断槽位），因此 Android 多发的字段在运行时被静默忽略，**不会造成任何功能故障**。

**需要的处理**（二选一，在本 TDS 记录决策即可，不要求阻塞发布）：

- **方案 A（推荐）**：将 `mic_index` 作为**可选的信息字段**正式纳入 schema，并在协议文档（`websocket_signals.md §6.5.5`）注明"可选，服务端不读取，仅供调试/日志追踪"；同时在 `TC-MIC.spec.ts` 的文件头注释中移除"no seat param"描述，以消除误导。
- **方案 B**：认为 `mic_index` 无需在 LeaveMic 中发送（服务端确实不需要），则将 Android `leaveMic()` 恢复为 `wsClient.sendEnvelope(type = "LeaveMic")`，并更新 PROTO-1 断言。

### ⚠️ MEDIUM-2：`LeaveMic.schema.json` 未同步更新，PROTO-BINDING 注释指向的锚点缺乏完整定义

**文件**：`doc/protocol/schemas/ws/LeaveMic.schema.json:12`

**问题描述**：

PROTO-BINDING 注释的隐含语义是"此代码遵循该 schema"。但当前 schema 对 `payload` 的定义是裸 `"additionalProperties": true`，未声明 `mic_index` 的类型、范围或可选性。若下一位工程师仅看 schema 无法得知 `mic_index` 的存在，PROTO-BINDING 注释的参考价值因此下降。

本条与 MEDIUM-1 同根，一并处理即可（方案 A 需同步更新 schema）。
