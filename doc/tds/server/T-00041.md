# TDS: WS 心跳 30s 超时主动断开（ping/pong） (Task ID: T-00041)

## 一、需求背景

- **关联需求**：QA 战报 BUG-WEB-001（模拟）暴露 App Server WebSocket 心跳超时检测缺陷。
- **影响测试用例**：TC-WS-00002（心跳 30s 超时断开）当前失败，实测服务端并未在 35s 静默后主动关闭连接。
- **业务影响**：僵尸连接累积导致内存泄漏，影响并发 1000 连接压测稳定性（TC-WS-00004）。
- **现状**：`app/server/src/modules/websocket/handler.rs`（假定路径）当前实现：
  - 客户端发 `{"type":"ping"}`，服务端回复 `{"type":"pong"}`。
  - **缺陷**：未实现服务端主动超时检测，依赖 TCP keepalive 机制（默认 2h）。
- **本 Task 目标**：在 WebSocket 连接处理器中实现：
  1. 记录最后心跳时间 `last_pong_at`。
  2. 启动定时检查任务，检测 `now - last_pong_at > 30s` 时主动关闭连接（close code=1000）。
  3. 确保 TC-WS-00002 步骤 3 通过（35s 静默后断开）。

## 二、技术方案

### 改造点位

**文件**：`app/server/src/modules/websocket/handler.rs`（或 `connection.rs`，以实际项目结构为准；下同，TDD 阶段需 grep `axum::extract::ws` 精确定位）

```rust
// 伪代码示例（不是完整实现）
use tokio::time::{interval, Duration};
use std::sync::{Arc, RwLock};

struct ConnectionState {
    user_id: Uuid,
    last_pong_at: Arc<RwLock<Instant>>, // 记录最后心跳时间
}

async fn handle_websocket(
    ws: WebSocket,
    user_id: Uuid,
    room_manager: Arc<RoomManager>,
) {
    let state = Arc::new(ConnectionState {
        user_id,
        last_pong_at: Arc::new(RwLock::new(Instant::now())),
    });

    // 启动超时检测任务
    let timeout_state = Arc::clone(&state);
    tokio::spawn(async move {
        let mut ticker = interval(Duration::from_secs(5)); // 每 5s 检查一次
        loop {
            ticker.tick().await;
            let elapsed = timeout_state.last_pong_at.read().unwrap().elapsed();
            if elapsed > Duration::from_secs(30) {
                tracing::warn!(user_id = ?timeout_state.user_id, "Heartbeat timeout, closing connection");
                // 发送关闭帧（close code=1000）并 break
                break;
            }
        }
    });

    // 主消息循环
    while let Some(Ok(msg)) = ws.next().await {
        match msg {
            Message::Text(text) => {
                let envelope: Envelope = serde_json::from_str(&text)?;
                match envelope.msg_type.as_str() {
                    "ping" => {
                        // 更新心跳时间
                        *state.last_pong_at.write().unwrap() = Instant::now();
                        // 回复 pong
                        ws.send(Message::Text(json!({"type":"pong"}).to_string())).await?;
                    }
                    // 其他消息处理...
                }
            }
            _ => {}
        }
    }
}
```

**关键改动点**：
1. 在 `ConnectionState` 或等效结构中新增 `last_pong_at: Arc<RwLock<Instant>>`。
2. 在收到 `ping` 消息时更新 `last_pong_at`。
3. `tokio::spawn` 独立任务，每 5s 检查一次：
   - 若 `now - last_pong_at > 30s`，调用 `ws.close(Some(CloseFrame { code: 1000, reason: "Heartbeat timeout".into() }))` 并结束任务。
4. 关闭码参考 [RFC 6455](https://datatracker.ietf.org/doc/html/rfc6455#section-7.4)：1000=Normal Closure，1002=Protocol Error（可选）。

### 协议契约

**ping/pong 消息格式**（已定义于 `doc/protocol/websocket_signals.md §6.2`）：

- 客户端 → 服务端：`{"type":"ping"}`
- 服务端 → 客户端：`{"type":"pong"}`

**超时行为**：
- **触发条件**：连续 30s 未收到客户端 `ping`（或任何消息，视实现可扩展为"任何有效消息均重置计时器"）。
- **动作**：服务端发送 WebSocket Close Frame（code=1000，reason="Heartbeat timeout"）并关闭 socket。
- **客户端感知**：收到 `close` 事件，`event.code === 1000`。

### 配置/env 变更

可选：在 `config/default.toml` 中新增 `[websocket]` section：

```toml
[websocket]
heartbeat_timeout_sec = 30
heartbeat_check_interval_sec = 5
```

TDD 阶段如实现配置化，需同步更新 `app/server/src/infrastructure/config.rs` 增加对应字段。

### 数据迁移

无（纯内存逻辑）。

## 三、TDD 验收标准

### 功能用例

1. **U-1**（正常心跳保活）：客户端每 15s 发一次 `ping`，连续 3 分钟，连接不断开。
2. **U-2**（30s 超时断开）：客户端建立连接后完全静默 35s，服务端在 30-35s 内主动关闭连接，close code=1000。
3. **U-3**（边界值）：客户端静默 29s 后发送 `ping`，连接保持不断；静默 31s 后连接已关闭，`ping` 无法发送。
4. **U-4**（pong 延迟）：服务端收到 `ping` 后 <500ms 内回复 `pong`（TC-WS-00002 步骤 2）。
5. **U-5**（并发场景）：10 个连接同时运行心跳，其中 5 个静默超时，5 个正常保活，仅超时的 5 个被关闭。

### 回归用例

6. **R-1**：`cargo test -p voice-room-server --test websocket_test` 全部通过（包括 T-00011 已有心跳测试，若存在）。
7. **R-2**：TC-WS-00002（心跳 30s 超时断开）在 E2E 套件中通过（`npm run e2e:local -- tests/ws.spec.ts`）。
8. **R-3**：TC-WS-00004（并发 1000 连接压测）无回归，内存 60s 内增长 <20%。

### 安全与边界

9. **S-1**：恶意客户端高频发送 `ping`（100 QPS）不导致服务端 CPU 飙升或拒绝服务（Rate Limit 由其他 Task 负责，本 Task 仅验证不 panic）。
10. **S-2**：服务端重启后，所有活跃连接的 `last_pong_at` 正确初始化为连接建立时间（不会立即触发超时）。
