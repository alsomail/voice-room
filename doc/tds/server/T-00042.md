# TDS: Admin 强制断连广播事件（user_banned / room_closed） (Task ID: T-00042)

## 一、需求背景

- **关联需求**：QA 战报 BUG-ARCH-002（模拟）暴露 Admin Server 封禁用户后，App Server WebSocket 未主动断开被封禁用户连接。
- **影响测试用例**：TC-WS-00005（Redis 事件订阅 - 管理员封禁事件推送）当前失败，被封禁用户仍能在房间内发送消息。
- **业务影响**：封禁后用户体验不连贯，需客户端重连或刷新才感知封禁状态；治理操作实时性差。
- **现状**：
  - Admin Server：`POST /api/v1/admin/users/:id/ban`（T-10009）已实现，但仅写 DB + admin_logs，**未推送 Redis 事件**。
  - App Server：T-00011B（Redis 事件订阅）已有框架，订阅 `admin:events` 频道，但 **事件类型仅处理占位 `test_event`**，缺少 `user_banned` / `room_closed` 处理逻辑。
- **本 Task 目标**：
  1. Admin Server 在 `POST /ban`、`POST /rooms/:id/close`（T-10006）时，向 Redis `admin:events` 频道发布 `user_banned` / `room_closed` 事件。
  2. App Server 订阅端增加对应处理器：
     - `user_banned`：找到该用户所有 WS 连接 → 发送 `UserBanned` 通知 → 关闭连接（close code=4003）。
     - `room_closed`：广播 `RoomClosed` → 断开房间内所有成员连接。

### 改造点位 3：App Server 订阅处理器

**文件**：`app/server/src/modules/events/admin_event_handler.rs`（或 `redis_subscriber.rs`，需 grep `admin:events` 定位）

```rust
// 伪代码
async fn handle_admin_event(
    event: AdminEvent,
    room_manager: Arc<RoomManager>,
    ws_registry: Arc<WsConnectionRegistry>, // 新增：连接注册表
) {
    match event.event_type.as_str() {
        "user_banned" => {
            let user_id = event.payload["user_id"].as_str().unwrap();
            // 1. 从 ws_registry 中找到该 user 所有连接
            let connections = ws_registry.get_connections_by_user(user_id);
            for conn in connections {
                // 2. 发送通知消息
                let msg = json!({
                    "type": "UserBanned",
                    "payload": {
                        "reason": event.payload["reason"],
                        "duration_sec": event.payload["duration_sec"],
                    }
                });
                conn.send(Message::Text(msg.to_string())).await.ok();
                // 3. 关闭连接
                conn.close(Some(CloseFrame {
                    code: 4003,
                    reason: "Account banned".into(),
                })).await.ok();
            }
            tracing::info!(user_id, "Disconnected banned user");
        }
        "room_closed" => {
            let room_id = event.payload["room_id"].as_str().unwrap();
            // 1. 从 room_manager 获取房间成员列表
            let members = room_manager.get_room_members(room_id).await?;
            // 2. 广播 RoomClosed
            room_manager.broadcast(room_id, json!({
                "type": "RoomClosed",
                "payload": { "reason": event.payload["reason"] }
            })).await?;
            // 3. 断开所有成员连接
            for user_id in members {
                let connections = ws_registry.get_connections_by_user(&user_id);
                for conn in connections {
                    conn.close(Some(CloseFrame { code: 1000, reason: "Room closed".into() })).await.ok();
                }
            }
            tracing::info!(room_id, "Closed room and disconnected all members");
        }
        _ => {
            tracing::warn!(event_type = event.event_type, "Unknown admin event");
        }
    }
}
```

**新增依赖**：`WsConnectionRegistry` 结构体，维护 `user_id -> Vec<WsConnectionHandle>` 映射。

**连接注册表示例**：

```rust
// 文件：app/server/src/modules/websocket/registry.rs
use dashmap::DashMap;
use tokio::sync::mpsc::UnboundedSender;

pub struct WsConnectionRegistry {
    // user_id -> Vec<sender>
    connections: DashMap<Uuid, Vec<UnboundedSender<Message>>>,
}

impl WsConnectionRegistry {
    pub fn register(&self, user_id: Uuid, sender: UnboundedSender<Message>) {
        self.connections.entry(user_id).or_default().push(sender);
    }

    pub fn get_connections_by_user(&self, user_id: &str) -> Vec<UnboundedSender<Message>> {
        let uid = Uuid::parse_str(user_id).ok()?;
        self.connections.get(&uid).map(|v| v.clone()).unwrap_or_default()
    }

    pub fn unregister(&self, user_id: Uuid, sender: &UnboundedSender<Message>) {
        if let Some(mut conns) = self.connections.get_mut(&user_id) {
            conns.retain(|s| !std::ptr::eq(s, sender));
        }
    }
}
```

## 三、TDD 验收标准

### 功能用例

1. **U-1**（用户封禁推送）：Admin 调用 `POST /admin/users/:id/ban`，被封禁用户所有 WS 连接在 <2s 内收到 `UserBanned` 并断开（close code=4003）。
2. **U-2**（房间关闭推送）：Admin 调用 `POST /admin/rooms/:id/close`，房间内所有成员在 <3s 内收到 `RoomClosed` 并断开（close code=1000）。
3. **U-3**（多连接场景）：用户 U1 同时打开 3 个客户端（3 个 WS 连接），封禁 U1 后，3 个连接**全部**断开。
4. **U-4**（跨进程推送）：Admin Server 与 App Server 分离部署（docker compose 多容器），封禁事件通过 Redis PubSub 正确投递。
5. **U-5**（事件幂等）：Admin 重复调用封禁接口（短时间内 2 次），App Server 仅处理一次（通过 `admin_logs.id` 或事件去重机制）。

### 集成用例

6. **I-1**（TC-WS-00005 通过）：E2E 测试 `TC-WS-00005` 全部步骤通过（Admin 封禁 → WS 收到通知 → 连接断开 → 封禁用户无法重连）。
7. **I-2**（TC-GOVERNANCE-XXX）：房间关闭后，被踢用户尝试 `JoinRoom` 返回 `ROOM_CLOSED` 错误（前置：T-00012 需增加房间状态检查）。

### 回归用例

8. **R-1**：Admin Server 所有现有测试通过（`cargo test -p voice-room-admin-server`）。
9. **R-2**：App Server 所有现有测试通过（`cargo test -p voice-room-server`）。
10. **R-3**：事件处理失败（如 Redis 连接断开）不影响 Admin Server 主流程，封禁操作仍成功写 DB（日志记录错误但不返回 500）。
