# TDS: 观众席列表 API (Task ID: T-00027)

## 一、背景与目标
- **关联需求**：E-10 观众席 Bottom Sheet 数据源 ([phase1_room_governance §2.1.2](../../product/phase1_room_governance.md))
- **本 Task 的目标**：GET /api/v1/rooms/:id/members，返回带角色 + 麦位信息的房间成员分页列表，100 人房间响应 <150ms。

## 二、方案设计

### 接口
**GET /api/v1/rooms/:id/members?page=1&limit=20**（需 JWT，成员可见）

响应：
```json
{
  "code":0,
  "data": {
    "total": 87, "page":1, "limit":20,
    "items":[
      {
        "user_id":"uuid","nickname":"...","avatar":"...",
        "role":"owner|admin|member",
        "mic_slot": 0,                  // null 表示观众
        "joined_at":"RFC3339",
        "muted_mic": false, "muted_chat": false   // 管理员可见
      }
    ]
  }
}
```

### 数据来源 & 排序
1. 从 `RoomManager.get_members(room_id)` 内存读 Vec<MemberSnapshot{user_id, joined_at, slot}>
2. 批量 SELECT users (nickname, avatar, deleted_at IS NULL)
3. 计算 role：`user_id == room.owner_id → owner`；`user_id == room.admin_user_id → admin`；else member
4. 排序规则：
   - **麦上用户置顶**：`slot ASC`（主麦在最前）
   - **观众按 joined_at DESC**（最新进房者在前）
5. 分页 slice

权限：
- 房间成员（连接中）可看到 `muted_mic/muted_chat` 字段
- 非成员（HTTP 路人请求）返回 403（40301） — MVP 仅连接中用户可调

## 五、Reviewer意见
### Round 1
- **状态**：未通过（TDD）
- **审查意见**：
  1. **[HIGH] 生产适配器 N+1 查询**：`members_handler.rs` 中 `AuthServiceUserAdapter::find_users_by_ids` 通过 `for id in ids { get_user_by_id(*id).await }` 逐条查 DB，100 人房间第 1 页需 20 次 SQL，违反 TDS 要求的"1 次 SQL `WHERE id = ANY($1)`"，且 `AuthService` 目前无批量查用户接口，生产路径的 <150ms 性能目标无法保证（测试通过仅因使用了 FakeUsersRepo）。
  2. **[MEDIUM] `joined_at` 双重冗余存储**：`MemberInfo.joined_at` 与 `RoomState.member_join_times DashMap` 存储相同数据；两次独立 `Utc::now()` 导致微秒级时差；`list_members` 中的 fallback 逻辑揭示数据可能不一致，应移除 `member_join_times`，统一使用 `MemberInfo.joined_at`。
- **改进建议**：
  1. 在 `UserRepository` 新增 `find_by_ids(ids: &[Uuid]) -> Vec<UserModel>` 批量查询方法（SQL: `WHERE id = ANY($1) AND deleted_at IS NULL`），`AuthService` 暴露对应的 `get_users_by_ids`，生产适配器改为单次调用。
  2. 移除 `RoomState.member_join_times` DashMap，`list_members` 直接读 `MemberInfo.joined_at`，`do_leave_room` 删除对应清理行，保持单一数据源。
