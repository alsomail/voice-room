# TDS: Chat REST POST 广播修复（BUG-CHAT-WS-BROADCAST） (Task ID: T-00045)

## 一、需求背景

- **关联缺陷**：BUG-CHAT-WS-BROADCAST（E2E Round 14 实证）。
- **现象**：客户端走 REST `POST /chat-messages`（既有客户端实现）→ server INSERT 成功，但 INSERT 后**没有**向房间内 WS 连接广播 `RoomMessage`。Round 14 全量 server 日志（2935 行）`broadcast` 关键字仅出现 1 次，源自 BalanceBroadcaster 启动日志，与 chat 链路无关。
- **现状**：
  - WS 信令路径 `SendMessage`（`room/handler/chat.rs::handle_send_message`）已实现 INSERT + `broadcast_to_room` 广播 `RoomMessage`（T-00016 / T-00043）。
  - REST 历史查询路径 `GET /api/v1/rooms/:room_id/messages` 仅读不写广播（T-00043）。
  - **缺失**：REST 写入路径 `POST /api/v1/chat-messages` 在服务端**不存在**对应处理函数；客户端发送的请求要么被路由层拒绝（404），要么被某个旧版本路径吞掉。本 Task 首要落地"REST 写消息 → INSERT → 房间广播"完整闭环，与 WS 路径行为对等。
- **本 Task 目标**：
  1. 新增 REST 端点 `POST /api/v1/chat-messages`（`room_id` + `content` 走 JSON body）。
  2. INSERT 成功（DB 事务提交后）→ 通过既有 `ws::broadcaster::broadcast_to_room` 向房间所有 WS 连接广播 `RoomMessage` envelope（与 WS 路径形态完全一致）。
  3. 单连接断开不阻断整体广播；REST 不因广播失败返回错误。
  4. 不修复 `last_msg_id` replay 窗口相关问题（仅在第六节附记）。

## 二、技术方案

### 核心数据流

```
Client (Android)  POST /api/v1/chat-messages   { room_id, content }
   │                                           Authorization: Bearer <JWT>
   ▼
[chat::controller::send_chat_message_handler]
   │  1. JWT auth (AuthContext)
   │  2. Validate content length (1..=500 chars)
   │  3. ChatRepository::insert_message(room_id, user_id, content)  ← DB transaction commits here
   │  4. Build RoomMessage envelope { type, payload:{ msg_id=DB id, user_id, content }, timestamp }
   │  5. broadcast_to_room(registry, room_state, envelope)           ← envelope-level msg_id 由广播器分配 (UUID v4)
   │     - 房间不在内存：降级 broadcast_to_room_no_state（不写 recent_broadcasts）
   │     - 单连接 send 失败容忍（既有 broadcaster 行为）
   ▼
HTTP 201 Created  { code: 0, data: { msg_id: <DB-id> } }
```

### 接口契约

- 请求：`POST /api/v1/chat-messages`
  - Headers: `Authorization: Bearer <JWT>`, `Content-Type: application/json`
  - Body: `{ "room_id": "<uuid>", "content": "<1..=500 chars>" }`
- 响应（200/201）:
  ```json
  { "code": 0, "data": { "msg_id": "<DB UUID v4>" }, "request_id": "..." }
  ```
- 错误：
  - 400：room_id 非法 UUID / content 缺失或长度越界
  - 401：JWT 缺失或过期（既有 `AuthContext` 行为）
  - 500：DB 写入失败（不广播，REST 返回错误）

### 广播 envelope 形态（与 WS SendMessage 对齐）

```jsonc
{
  "type": "RoomMessage",
  "msg_id": "<envelope-level UUID v4，由 broadcast_to_room 注入>",
  "payload": {
    "msg_id": "<DB row id (UUID v4)>",
    "user_id": "<sender uuid>",
    "content": "<filtered content>"
  },
  "timestamp": <unix ms>
}
```

### 实现约束（红线对齐）

1. **广播必须发生在 DB 事务提交之后**：调用 `repo.insert_message().await?` 拿到 `id` 才构造 envelope。
2. **envelope 携带 `msg_id`（UUID v4）**：通过 `ws::broadcaster::broadcast_to_room` 统一注入，复用既有续传 / 去重契约（§6.7）。
3. **broadcast 失败不让 REST 失败**：现有 `broadcast_to_room` 内部对每条连接 `let _ = sender.send(...)`，单连接断开不会向上抛错；REST handler 不捕获其返回值（`String`）作为错误信号。
4. **房间状态不存在时降级**：`room_manager.get_room(room_id)` 为 `None` 时走 `broadcast_to_room_no_state`，不写 `recent_broadcasts`（房间无内存则 last_msg_id 续传本就不可用，无新增风险）。
5. **不引入新依赖**：复用 `ChatRepository`、`ws::broadcaster`、`room::manager::RoomManager`、`AuthContext`、`AppError` / `ApiResponse`。
