## 1. 战略定位

### 1.1 为什么这些能力要合并为一个 Epic
创建房间完整化、观众席、房主权限、管理员体系四件事**业务强耦合**：
- 创建房间 → 确定房主 → 任命管理员 → 管理员操作用户 → 被操作用户在用户列表里
- 拆散做会导致"房主能力"四处散落，TDD 测试无从入手
- 管理员权限需要的"踢人/禁麦/禁言"能力必须同时支持，否则房间治理不完整

### 1.2 为什么放在 E-07 之后
- **营收闭环优先**：先打通"钻石→礼物"这条主动脉
- **房间治理是"质量基础"**：不是驱动营收的核心引擎，但**影响留存与内容合规**
- **MENA 市场合规刚需**：中东女性房主比例 60%+，缺少管理工具 = 骚扰/举报高发 = App Store 下架风险

### 1.3 已具备的基础设施（复用）
本 Epic 无需新建通信基础设施，全部复用现有能力：
- ✅ WS 心跳断连机制（被踢用户 30s 内自动清理）
- ✅ `RoomManager` 内存成员表（观众席数据源）
- ✅ `admin:events` Redis Pub/Sub（踢人/禁言广播复用）
- ✅ 房间聊天链路（聊天禁言仅需加"前置权限校验"）
- ✅ `stats/service.rs` 在线检测（管理员查看谁在房间）

|------|-----|
| 任命 / 撤销管理员 | ✅ 最多 1 位管理员（MVP） |

|------|-----|
| 被踢后不能马上回来 | Redis 记录 `kicked:{room_id}:{user_id}`，TTL 10 分钟 |

| 被禁言视觉反馈 | 聊天输入框置灰 + Toast |
| 举报入口 | 观众席列表点用户→"举报"（MVP 仅写入日志，不走审核流） |

详见 [business_flows.md §2.8](./business_flows.md)，本节仅列流程清单：

| # | 流程 | 复杂度 |
|---|------|-------|
| 3.1 | 创建房间（含封面/分类/密码） | 🟢 低 |
| 3.2 | 密码房进房校验 | 🟡 中 |
| 3.3 | 观众席打开与刷新 | 🟢 低 |
| 3.4 | 房主任命管理员 | 🟡 中 |
| 3.5 | **踢出用户（重点）** | 🔴 高 |
| 3.6 | 禁麦 / 禁言 / 抱麦 | 🟡 中 |
| 3.7 | 被踢/禁用户的客户端反馈 | 🟡 中 |

### 踢出用户的关键设计（3.5）

```
房主/管理员操作 → WS SendSignal KickUser { room_id, target_user_id, reason }
   ↓
Server 权限校验（操作者角色 > 目标用户角色）
   ↓
Server 事务：
  1. Redis SET kicked:{room_id}:{user_id} EX 600（10 分钟黑名单）
  2. RoomManager 清理 target 内存状态
  3. 若 target 在麦上 → 自动下麦
   ↓
Server 广播：
  ├─ 向被踢用户：UserKicked { reason, room_id } → 客户端主动断开 WS + 弹窗
  └─ 向房间其他人：UserLeft { user_id, reason: "kicked_by_admin" }
   ↓
被踢用户 10 分钟内再次 JoinRoom：
  Server 查询 kicked:* → 返回 `KICKED_COOLDOWN` 错误
```

## 7. 技术约束

1. **单一事实源**（红线 #1）：角色信息（房主/管理员/普通）由 Server 广播，客户端**严禁**本地推断
2. **权限校验位置**：所有管理操作的权限校验**必须在 Server 层**，客户端只做 UI 隐藏/置灰
3. **强事务**：踢人/禁麦/任命管理员必须是原子操作（DB + Redis + 广播）
4. **审计日志**：所有管理操作写入 `room_kick_records` / `room_mute_records`，供后续申诉
5. **防腐层**：房间封面图上传走对象存储适配器（`core/storage/`）
6. **合规**：被踢/被禁用户必须收到明确原因，不能静默处理（MENA PDPL 要求）
