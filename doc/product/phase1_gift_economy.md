## 1. 战略定位：为什么是"礼物闭环"先行

### 1.1 营收权重决定优先级
竞品验证（详见 [competitors.md](./competitors.md)）：
- Yalla 虚拟礼物占营收 **70%**
- YoHo 打赏分成占 **65%**
- Ahlan 土豪大 R 打赏占 **80%**

**结论**：礼物打赏是语聊房赛道的第一营收支柱，必须最先打通。

### 1.2 "封闭内循环"策略（规避支付合规卡点）
真实支付涉及 Google Play Billing / 本地钱包（STC Pay、Mada）/ 财务合规，周期长、风险高。

**本 Epic 采用"运营手动充值"作为 MVP 充值通道**：
- Admin 后台可为任意用户调整钻石余额（用于测试 / 种子用户 / 补偿）
- Android 端"充值"按钮显示"即将上线"占位，但**钻石余额、送礼、扣减、榜单全部真实生效**
- E-08 接入真支付后，只需新增支付回调→调用同一个"加余额"服务即可，零重构

### 1.3 Android 端可见性承诺
本 Epic 共 **7 个 Android Task**，每 2-3 天即可看到新 UI：
1. 钱包页面（余额 + 流水）
2. 礼物面板（Bottom Sheet）
3. 礼物接收者选择器
4. 送礼全屏 Lottie 动画
5. 弹幕礼物消息样式
6. 魅力/财富榜页
7. 余额不足引导弹窗

## 2. 范围边界 (Scope)

### 2.1 In Scope（本 Epic 必做）
| 领域 | 内容 |
|------|------|
| 钱包 | 钻石（diamond）余额字段、流水表、查询 API、余额 WS 推送 |
| 礼物 | 礼物配置表（id/name/icon/price/effect_type/animation_url）、CRUD 后台、列表 API |
| 送礼 | `SendGift` WS 信令、SQLx 强事务扣减、`GiftReceived` 广播、幂等 msg_id |
| 榜单 | Redis ZSet 日榜 + 周榜（魅力值 = 收到礼物钻石数 / 财富值 = 送出钻石数） |
| 运营 | Admin Server 手动调整余额 API、Web Admin 用户详情"调整余额"按钮 |
| UI | Android 7 个新页面/组件 + 房间内礼物入口激活 |

### 2.2 Out of Scope（延后）
| 领域 | 延后到 | 原因 |
|------|-------|------|
| Google Play 真支付 | E-08 | 需独立支付回调/签名验证工程 |
| 本地支付（STC Pay/Mada） | E-08/后续 | 合规周期长 |
| 礼物合成 / 盲盒 | Phase 3 | MVP 不追求复杂玩法 |
| 全服广播横幅 | Phase 2 | 依赖推送服务 |
| 金币（coin）体系 | 后续 | MVP 仅保留钻石一种货币，降低心智负担 |

### 2.3 货币模型（简化）
- **单货币**: `diamond`（钻石），1 钻石 = 运营定价（预设 1 钻石 ≈ $0.01）
- **不区分** 充值钻 vs 赠送钻（即"钻石可以送礼也可以提现"的简化模型）
- 提现（主播→现金）延后到 E-08 完成真支付接入时一并设计

## 4. 核心业务流（简版，详细流程见 business_flows.md §2.7）

### 4.1 送礼正向流程
```
用户点击房间底部 🎁 → 礼物面板弹出 → 选礼物 → 选接收者(麦位列表)
    → 选数量(1/10/66/520/1314) → 点击"送出"
    → WS: SendGift {gift_id, receiver_id, count, msg_id}
    → Server 事务：扣发送者余额 → 加接收者魅力值 → 写流水
    → WS 广播 GiftReceived 给房间所有人
    → Android: 全屏 Lottie + 弹幕礼物消息 + 更新余额
```

### 4.2 关键异常
| 场景 | Server 行为 | Android UI |
|------|-----------|-----------|
| 余额不足 | 返回 `INSUFFICIENT_BALANCE` 错误 | 弹窗"余额不足，去充值"（跳钱包页） |
| 接收者离线/已下麦 | 返回 `RECEIVER_UNAVAILABLE` | Toast 提示 + 礼物面板不关闭 |
| 重复 msg_id | 幂等拒绝，不二次扣减 | 前端忽略（已收到广播） |
| 网络断开 | 请求失败，礼物按钮恢复可点 | Toast + 保留面板状态 |
| Server 内部错误 | 事务回滚，余额不变 | 弹窗"赠送失败，请重试" |

## 5. 关键技术约束（Plan Agent 需遵循）

1. **强事务**（红线 #2）：余额扣减 / 魅力值累加 / 流水写入必须在**同一个 SQLx Transaction** 内，任何一步失败整体回滚。
2. **幂等**（红线 #2）：`SendGift` 必须携带 `msg_id`（UUID），Server 基于 `(sender_id, msg_id)` 做去重。
3. **单一事实源**（红线 #1）：Android 余额**严禁**本地预扣减，必须等 Server 的 `BalanceUpdated` WS 广播后刷新。
4. **榜单缓存**：Redis ZSet 每日/每周独立 key，凌晨 00:00 Asia/Riyadh 切换（中东时区）。
5. **礼物资源**：Lottie JSON 文件托管在 CDN（Phase 1 可用项目 assets/），Android 通过 URL 下载缓存。

## 6. 验收指标（Exit Criteria）

- [ ] Android 端用户可完成：登录→进房间→打开礼物面板→选礼物→选接收者→送出→看到全屏动画→余额刷新

- [ ] 所有 Android 新页面含 `testTag` 自动化测试锚点
