## 1. 需求描述

提供 `POST /api/v1/rooms` 接口，允许已登录用户创建语音房间。

### 验收标准

| 编号 | 描述 |
|------|------|
| C-01 | 合法请求返回 HTTP 201 + room_id / title / room_type / created_at |
| C-02 | 无 JWT Token → HTTP 401，code 40101 |
| C-04 | 空 title → HTTP 400，code 40003 |
| C-05 | title 超 30 Unicode 字符 → HTTP 400，code 40003 |
| C-06 | title 恰好 30 Unicode 字符 → 成功 |
| C-07 | owner 已有 active 房间 → HTTP 409，code 40900 |
| C-08 | room_type=password 未提供 password → HTTP 400，code 40003 |
| C-09 | 密码房 password_hash 为 bcrypt 摘要（可 verify） |
| C-10 | room_type 不在枚举中 → HTTP 400，code 40003 |
| C-12 | 错误响应中 request_id 与 X-Request-Id 一致 |

## 2. 技术方案

### 2.1 请求 / 响应格式

**请求体**（`application/json`）

```json
{
  "title": "我的房间",
  "room_type": "normal",
  "password": "可选，password 类型时必填"
}
```

**成功响应**（HTTP 201）

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "room_id": "uuid",
    "title": "我的房间",
    "room_type": "normal",
    "created_at": "2024-01-01T00:00:00Z"
  },
  "request_id": "..."
}
```

**错误响应**（HTTP 400 / 401 / 409）

```json
{
  "code": 40003,
  "message": "Validation error: title must not be empty",
  "request_id": "..."
}
```

### 2.2 业务规则

| 规则 | 实现位置 |
|------|----------|
| JWT 认证 | `AuthContext` extractor（复用 T-00004） |
| title: 1–30 chars（chars().count()） | `RoomService::create_room` |
| room_type: normal / password / paid | `RoomService::create_room` |
| password 类型必须提供 password | `RoomService::create_room` |
| 同一 owner 最多 1 个 active 房间 | DB 唯一索引 + 服务层检查 |
| password 使用 bcrypt hash 存储 | `tokio::task::spawn_blocking` |

```sql

    ON rooms (owner_id)
```

### 问题清单

#### [HIGH] H-01：DB 唯一约束违反未转换为正确的 HTTP 409 响应

**文件**：`app/server/src/modules/room/repository.rs` + `app/server/src/common/error.rs`

**问题描述**：

TDS §2.2 明确规定"同一 owner 最多 1 个 active 房间"使用 **DB 唯一索引 + 服务层检查** 双重防护。当两个并发请求同时通过服务层的 `find_active_by_owner` 检查（TOCTOU 窗口），只有 DB 唯一索引 `idx_rooms_owner_active` 能拦截第二条 INSERT。

然而，`From<sqlx::Error> for AppError` 的实现将**所有** sqlx 错误（包括 PostgreSQL unique violation `code=23505`）统一转为 `AppError::DatabaseError`，导致：

```
并发 INSERT → sqlx::Error::Database(code=23505)
    → AppError::DatabaseError(...)       ← 错误！
    → HTTP 500 "internal server error"   ← 应为 HTTP 409 / code 40900
```

```rust
// 当前实现（error.rs）— 吞掉了 unique violation
impl From<sqlx::Error> for AppError {
    fn from(e: sqlx::Error) -> Self {
        AppError::DatabaseError(e.to_string())  // 丢失了错误类型信息
    }
}

// 正确实现应检测 unique violation：
impl From<sqlx::Error> for AppError {
    fn from(e: sqlx::Error) -> Self {
        if let sqlx::Error::Database(ref db_err) = e {
            // PostgreSQL unique violation = "23505"
            if db_err.code().as_deref() == Some("23505") {
                return AppError::ActiveRoomExists;
            }
        }
        AppError::DatabaseError(e.to_string())
    }
}
```

**影响**：并发创建同一用户房间时，第二个请求收到 HTTP 500 而非 HTTP 409，违背 TDS 验收标准 C-07 的精神（C-07 只覆盖了串行场景，未覆盖并发场景），且 DB 层防护形同虚设。

**修复要求**：在 `From<sqlx::Error> for AppError` 中检测 PostgreSQL 错误码 `23505`（unique violation），返回 `AppError::ActiveRoomExists`；并补充对应单元测试。

#### [MEDIUM] M-01：非密码房间可静默存储 `password_hash`

**文件**：`app/server/src/modules/room/service.rs:63-88`

**问题描述**：

服务层仅验证"password 类型房间**必须**提供 password"，但未拒绝/清除非 password 类型房间携带的 `password` 字段：

```rust
// 客户端发送：{"room_type": "normal", "password": "secret"}
// 当前行为：静默 bcrypt hash 后存入 password_hash 列 ← 数据污染
let password_hash = if let Some(pwd) = req.password {
    // 无论 room_type 是什么，只要 password 不为 None 就 hash
    Some(hash)
} else {
    None
};
```

普通房间（`normal`/`paid`）的 `password_hash` 列语义上应为 `NULL`。若误存了 hash，后续进入逻辑可能错误地要求密码验证。

**修复建议**：

```rust
// 只在 room_type == "password" 时 hash
let password_hash = if req.room_type == "password" {
    if let Some(pwd) = req.password {
        // spawn_blocking ...
        Some(hash)
    } else {
        None // C-08 已在上方校验，不会走到这里
    }
} else {
    None  // 非密码房间，忽略 password 字段
};
```

**文件**：`app/server/src/modules/room/service.rs:43-48`

`" "` (单空格) 的 `chars().count() == 1`，可通过验证并创建房间，但空白标题从 UX 角度无意义。现有测试 `title_single_space_is_one_char_passes` 将此行为标记为"通过 service 验证"，但没有 TDS 说明此设计意图。

**建议**：若非有意为之，在验证处补充 `title.trim().is_empty()` 检查，并在 TDS 中说明。属于低优先级，可与 M-01/M-02 一并修复。

#### [LOW] L-02：`RoomModel` 派生 `Serialize` 会包含 `password_hash` 字段

**文件**：`app/shared/src/models/room.rs`

**问题描述**：

`RoomModel` 整体 derive `Serialize`，`password_hash` 字段默认参与序列化。当前 handler 正确返回 `CreateRoomResponse`（不含 `password_hash`），安全。但若未来某个 handler 误用 `Json(room_model)` 直接响应，会泄露密码哈希。

**建议**：在 `password_hash` 字段上加 `#[serde(skip_serializing)]` 或专门的序列化配置，消除未来误操作风险。属于防御性建议。
