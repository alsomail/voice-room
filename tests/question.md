# 测试用例对齐问题清单

> 生成时间：2026-05-09
> 基于报告：`tests/report-20260509073757/Report.md`
> 用例源：仓库中不存在 `tests/cases/`，实际用例文件位于 `doc/tests/cases/`
> 分析结论：失败主因是**用例前置状态、脚本实现与应用真实行为/真实环境不对齐**，不是单一业务代码缺陷。

## 总体对齐问题

### P0：前置状态假设错误

大量 AND 用例默认 `resetAndroidToMainPage()` 之后一定落在大厅、`resetAndroidToLoginPage()` 之后一定落在登录页；但报告显示真实运行中经常仍停留在房间页、主界面或手机桌面。也就是说，测试把“启动后页面确定性”当成既成事实，而应用真实行为仍受 DataStore 导航恢复、JWT 删除不彻底、前序套件 force-stop 污染影响。

### P1：测试数据/角色/房间夹具没有真正建立

不少用例文档要求“固定房间、固定余额、固定榜单数据、固定第二用户/管理员/受害者”，但脚本实际常常只是“登录后点第一张房间卡片”或“如果有其他用户就继续，否则降级/跳过”。这导致脚本即使跑通，也未必在验证文档里定义的真实业务剧本。

### P1：步骤描述与真实 UI 文案/信息架构漂移

典型问题包括：中文按钮文案硬编码在阿语 UI 上查找、`THEME-00002` 文档要求进入 Debug Demo 页面但脚本改测生产页、`RANKING-00001` 文档要求 Top3 奖牌而脚本已经放宽到“空榜单也接受”。也就是 case doc、spec 实现、真实产品三者已经分叉。

### P2：环境能力不足却仍按强约束 E2E 验证

`RESILIENCE` 依赖网络控制与 session 观测、`GOVERNANCE/CHAT/GIFT/MIC` 依赖多用户同房、`RANKING` 依赖榜单 seed 数据；但当前环境并未稳定提供这些能力。结果是失败体现为“找不到元素/超时”，本质却是环境没有满足剧本前提。

## 逐用例分析

### AND 失败用例

#### TC-WALLET-00004
- **用例期望**：文档要求 `U1 余额=50` 且已在房间内；spec 通过 `resetAndroidToMainPage()` 后直接点第一张房间卡片，再打开礼物面板并选择最贵礼物。
- **实际行为**：报告显示未触发余额不足弹窗；根因被归到 ROOM 大厅/进房前置失败。
- **对齐问题**：脚本既没有把余额显式改成 50，也没有保证一定能进到目标房间；它把“低余额 + 已进房”当成默认真实状态。
- **建议修改**：先用 DB/API 固定余额，再进入固定测试房间；只有在确认已进房且礼物面板已打开后，才验证不足弹窗。

#### TC-PROFILE-00003
- **用例期望**：文档要求个人中心存在“💎 钻石余额 >”入口；spec 登录后进入“我的”，直接点击“钻石余额行或 >”。
- **实际行为**：报告为 `failed to locate element`，且明确提示可能是 UI 文案/布局变化，或测试账号无钻石余额导致入口隐藏。
- **对齐问题**：case/spec 把入口形态固定成“钻石余额行”，但真实页面可能展示 wallet/coins 入口，或 0 余额时视觉形态不同。
- **建议修改**：前置显式 seed 非零余额；断言从“钻石余额”放宽为“余额/钱包入口”，优先使用稳定 testTag/语义选择器。

#### TC-PROFILE-00005
- **用例期望**：文档要求个人中心底部存在红色“退出登录”；spec 进入个人中心后不滚动，直接点击底部退出按钮。
- **实际行为**：报告显示未找到“退出登录”按钮。
- **对齐问题**：脚本假设退出按钮首屏必可见、文案固定；真实 UI 可能需要滚动到底部，或因阿语/英语本地化导致 AI 描述失配。
- **建议修改**：进入个人中心后先滚动到底部；选择器改为“退出/Sign out/红色退出操作”，不要把中文文案写死。

#### TC-GOVERNANCE-00001
- **用例期望**：文档要求进入 CreateRoomScreen 后能看到房名、封面、分类、公告、密码等字段联动；spec 登录进大厅后点击 FAB 进入创建页。
- **实际行为**：报告显示“App 未在预期的创建房间页面”。
- **对齐问题**：脚本把“登录完成后一定在大厅且 FAB 可点击”视为事实，但真实启动状态不稳定；同时文档中的若干字段在当前实现里未必是必现元素。
- **建议修改**：先验证当前确实在大厅，不在大厅就强制回大厅/清理导航状态；把“可选字段存在性”与“创建流程成功”拆成两类断言。

#### TC-GOVERNANCE-00005
- **用例期望**：文档是 3 设备 / 9 角色组合矩阵；spec 实际只有一个 Android 客户端，加一个“可选 seed 麦位占用者”。
- **实际行为**：报告显示失败依赖 ROOM 大厅渲染，无法稳定进房。
- **对齐问题**：就算进房成功，当前 spec 也只是单端 smoke，不是真正的角色权限矩阵验证；case 与实现严重缩水。
- **建议修改**：重写为多端编排测试，至少固定 O/A/M 三个身份；若做不到，应下调为“单端菜单存在性”用例并同步改 case 文档。

#### TC-GOVERNANCE-00006
- **用例期望**：文档前提是“房主已经对成员 M 打开踢人原因弹窗”；spec 先创建房间，再尝试打开观众席并选择其他用户。
- **实际行为**：报告将其归为进房/大厅依赖失败。
- **对齐问题**：脚本没有稳定 seed 观众用户；即使创建房间成功，也大概率没有“可被踢的人”。
- **建议修改**：将“创建房间”和“踢人原因弹窗”拆开；通过 API/WS 先把受害者放进房间，再只测踢人弹窗行为。

#### TC-GOVERNANCE-00007
- **用例期望**：文档要求 `U_TARGET` 已在房内，由房主执行踢出，目标端收到全屏被踢弹窗；spec 则是当前账号进第一张房间卡片后，再尝试用 `validToken` 调接口踢自己。
- **实际行为**：报告显示失败同样卡在进房前置。
- **对齐问题**：case 需要明确的“房主端/受害者端”双角色；脚本却用单账号推导 roomId，再让同一运行体承担被踢对象，角色语义不成立。
- **建议修改**：使用独立 victim token / 第二设备；先确认服务端确实创建 kick 记录，再观察被踢 UI。

#### TC-GOVERNANCE-00008
- **用例期望**：文档要求 `U1` 已被 `mic_muted` / `chat_muted`，并验证被禁麦/禁言后的 UI 反馈；spec 只是随机进房后点击空麦位、聊天框看看会弹什么。
- **实际行为**：报告显示依赖 ROOM 进房失败。
- **对齐问题**：case 是“被治理后的状态验证”，script 却没有任何治理状态预置；即便进房成功，也不一定在验证禁麦/禁言。
- **建议修改**：测试开始前通过 API/Redis/后台接口显式设置 `mic_muted/chat_muted`，再进入房间看 UI。

#### TC-ROOM-00001
- **用例期望**：文档要求已有 25 个活跃房间且 U1 已在大厅；spec 直接 `resetAndroidToMainPage()` 后断言大厅网格、下拉刷新和分页。
- **实际行为**：报告明确写到 AI 看到的是“虚线圆形带加号”的房间内麦位，而不是大厅房间卡片网格。
- **对齐问题**：spec 假设 reset helper 能把应用稳定带回大厅；真实行为却会从 DataStore/导航状态恢复到 RoomScreen。
- **建议修改**：在 helper 内显式清理导航状态，或在测试开始时检测“如果在房间页则先退回大厅”。

#### TC-ROOM-00003
- **用例期望**：文档要求 U1 无活跃房间，点击大厅 FAB 后弹出创建房间 Bottom Sheet；spec 也是从 `resetAndroidToMainPage()` 起步。
- **实际行为**：报告显示该用例失败依赖大厅状态。
- **对齐问题**：case/spec 都把“已稳定落在大厅”当作前置，但真实 app 启动后不一定在大厅，因此创建房间入口并不可靠可达。
- **建议修改**：先把“回到大厅”做成单独可复用的 guard；通过后再进入创建房间验证。

#### TC-ROOM-00005
- **用例期望**：文档要求大厅首屏至少有 1 张房间卡片；spec 进入大厅后点第一张卡片进入房间再返回。
- **实际行为**：报告显示失败仍在大厅前置，根本没看到房间卡片。
- **对齐问题**：脚本默认“第一张房间卡片一定存在且可见”，但真实状态是根本没有到大厅。
- **建议修改**：先验证大厅列表存在；必要时用固定房间标题或深链/接口直达，不要依赖“第一张卡片”。

#### TC-RESILIENCE-00001
- **用例期望**：文档要求已在房间内，并且环境支持断网/重连观测；spec 通过系统命令关 Wi‑Fi/数据后，用视觉文案判断断线与重连。
- **实际行为**：报告显示当前环境不支持网络断线模拟。
- **对齐问题**：用例把“设备网络可被自动化稳定控制”和“重连横条必定出现”都当成前提，但当前环境不满足。
- **建议修改**：先做环境能力探测；若缺少网络控制，就标记为环境门控用例，并改用 logcat/WS 指标做主断言。

#### TC-RESILIENCE-00004
- **用例期望**：文档要求验证“10s 回前台不重连，35s 回前台新 session”；spec 实际只做了 10s Home 回前台，并靠 `aiBoolean` 判断是否出现重连横条。
- **实际行为**：报告显示 AI 状态检测失败。
- **对齐问题**：spec 已经比 case 文档缩减，而且把关键验证简化成脆弱的视觉提示，没有检查 `session_id` / 日志 / 事件。
- **建议修改**：拆成 10s 与 35s 两个子场景；用日志或埋点验证 session 是否复用，而不是只看横条。

#### TC-RESILIENCE-00005
- **用例期望**：文档要求“强杀后 JWT 保留，但房间不自动恢复，只回大厅”；spec 只判断 `goesToHall === true`。
- **实际行为**：报告指出内部业务断言失败，出现 `expect(false).toBe(true)`。
- **对齐问题**：脚本把“直接进大厅”压缩成一个布尔值，无法区分“回登录页 / 恢复房间 / 停在桌面 / 其它页面”等真实状态；case 与断言粒度不匹配。
- **建议修改**：拆成两个独立断言：一是“不回登录页”，二是“不自动恢复房间”；同时保留截图/日志证据。

#### TC-MIC-00001
- **用例期望**：文档要求“全新安装、无录音权限、已进入 RoomScreen”；spec 通过 `resetAndroidToLoginPage()` 期望回登录页，再走登录+进房流程。
- **实际行为**：报告显示 reset 后 app 仍显示主界面而不是登录页，说明 JWT 删除失效。
- **对齐问题**：脚本把“删 JWT = fresh install”当成等价条件，但真实 helper 并未稳定清空认证态。
- **建议修改**：用 `pm clear` 或可验证的 token 删除方案建立真正 fresh state，再验证权限拒绝路径。

#### TC-MIC-00002
- **用例期望**：文档要求 U1 已授予麦克风权限、位于固定房间 R1、麦位 3 空闲；spec 只是进入第一张房间卡片并寻找任意空麦位。
- **实际行为**：报告显示受 ROOM 大厅问题影响，无法稳定进入房间。
- **对齐问题**：即使进房成功，script 也没有验证固定 slot=3，更没有保证目标房间和 RTC 前提成立。
- **建议修改**：使用固定 E2E 房间和固定座位 seed，先确认 seat=3 空闲，再测试上下麦。

#### TC-MIC-00009
- **用例期望**：文档要求已在固定 E2E 房间里，自己已经占住麦位，再点击自己麦位弹出下麦确认；spec 通过房间排序 bump、下拉刷新、坐标点击等方式自愈。
- **实际行为**：报告仍将其归类为依赖进入房间失败。
- **对齐问题**：这个 case 的真正前提是“能稳定找到 E2E Test Room 并进入”，但它仍受大厅状态影响；此外大量固定坐标也强依赖现网布局不变。
- **建议修改**：优先绕过大厅（深链/接口 join 固定房间），其次再做麦位点击验证；不要把大厅可达性和麦位交互耦合在一起。

#### TC-GIFT-00001
- **用例期望**：文档要求 U1 在房间内且余额=10000，U2/U3 已在麦上；spec 只是登录后进入第一张房间卡片。
- **实际行为**：报告显示“找不到房间卡片”。
- **对齐问题**：case 需要稳定房间与接收者夹具，script 却完全依赖随机大厅与随机房间。
- **建议修改**：先建立固定礼物测试房和麦上用户，再打开礼物面板；否则即使进房也无法证明是目标剧本。

#### TC-GIFT-00003
- **用例期望**：文档要求已选定接收者 U2，且要验证 UUID msg_id、连送、超时；spec 仍从“进入第一张房间卡片”开始。
- **实际行为**：报告显示失败原因同样是进房前置未满足。
- **对齐问题**：脚本把“送礼对象、余额、房间、连送状态”都当作现场自然存在，和文档定义的强前置不一致。
- **建议修改**：先 seed 余额、接收者、房间，再单独测 SendGift 流程。

#### TC-CHAT-00002
- **用例期望**：文档要求 U1 与 U2 同房，并验证 U2 同步收到消息且自动滚到底部；spec 实际只有一个 Android 客户端。
- **实际行为**：报告显示其先被 ROOM 大厅问题阻断，无法进房。
- **对齐问题**：即便没有大厅问题，当前脚本也无法验证“U2 收到消息”这个多端条件；实现与 case 文档不一致。
- **建议修改**：引入第二客户端或 WS 观察者；若只能单端执行，应把 case 目标降级为“本端发送与本端渲染”。

#### TC-RANKING-00001
- **用例期望**：文档要求已 seed 魅力榜日榜 10 条数据、Top3 具备金/银/铜视觉；spec 进入榜单后其实已经放宽成“允许空状态”。
- **实际行为**：报告显示排行榜列表为空、无历史送礼数据。
- **对齐问题**：case 文档要测 Top3 奖牌，但 spec 实现已经接受“无数据也算正常 UI”，说明文档与执行目标已经漂移。
- **建议修改**：二选一：要么在环境中稳定 seed 榜单数据；要么把当前脚本改成单独的“空榜单状态”用例，不再挂靠 Top3 奖牌 case。

#### TC-SHELL-00001
- **用例期望**：文档要求冷启动时看到 Splash Logo 动画，再分流到登录页/主界面；spec 用 `pm clear` 后 `agent.launch()` 抓启动画面。
- **实际行为**：报告显示 `agent.launch()` 引发 HOME 闪屏，AI 看到桌面而非 Splash。
- **对齐问题**：自动化启动方式与真实用户冷启动路径不一致，导致测试观测到的是操作系统桌面，而不是应用启动动画。
- **建议修改**：改用 ADB `am start` + 连续截图/录屏，或使用平台级启动事件，而不是 Midscene `launch()` 去抓 800ms 内的 Splash。

#### TC-SHELL-00002
- **用例期望**：文档要求底部 3 Tab 是“房间 / 消息 / 我的”，并验证切回房间页后保留滚动位置；spec 却断言“首页 / 发现 / 我的”，并没有真正检查滚动位置。
- **实际行为**：报告为 `failed to locate element`。
- **对齐问题**：这是最明显的文档/脚本双漂移：产品 IA、case 文案、spec 断言三者不一致。
- **建议修改**：先统一当前产品的真实 Tab 命名；然后按最终命名重写用例，并真正记录/比较列表滚动位置。

#### TC-SHELL-00005
- **用例期望**：文档要求已经进入一个有主播/副麦/公屏/底栏的固定房间；spec 则从大厅点第一张房间卡片进入任意房间。
- **实际行为**：报告显示失败依赖 ROOM 进房问题。
- **对齐问题**：用例要测的是房间页面视觉骨架，但脚本把“进任意房间”混进了前置流程，导致房间壳层验证被大厅状态拖死。
- **建议修改**：固定房间夹具，必要时用深链或 join API 直接进房，再单测 RoomScreen 壳层视觉。

#### TC-THEME-00001
- **用例期望**：文档要求应用为黑金主题，并且能回到登录页观察“获取验证码”按钮样式；spec 中途 `resetAndroidToLoginPage()` 后期待登录页出现。
- **实际行为**：报告显示 reset 后仍在主界面，无法看到登录页按钮。
- **对齐问题**：测试想验证登录页主题，但前置 helper 并不能稳定带回登录页，导致 case 的目标页面根本没建立起来。
- **建议修改**：先修复 token 清除 helper；或改成 `pm clear` 后单独验证登录页主题。

#### TC-THEME-00002
- **用例期望**：文档明确要求“打开 Design System Demo 页面（Debug 菜单入口）”；spec 却在大厅、登录页、个人中心上拼凑验证 GoldButton / TextField / Avatar。
- **实际行为**：报告为 `failed to locate element`。
- **对齐问题**：这不是单纯元素没找到，而是 case 文档和脚本根本在测不同页面。
- **建议修改**：要么恢复/暴露 Demo 页面并按文档执行；要么把文档改写为“在生产页验证这些组件的落地样式”。

#### TC-THEME-00003
- **用例期望**：文档要求系统语言切为阿语并验证 RTL 镜像；spec 用 `am broadcast ... SET_LOCALE ar` 后直接去点榜单页。
- **实际行为**：报告为 `failed to locate element`。
- **对齐问题**：脚本没有先验证“阿语 locale 已真实生效”；而这个广播在不同设备上并不可靠，导致整个 RTL 前提本身不成立。
- **建议修改**：先读取系统/应用 locale 并确认已切成阿语，再做镜像断言；优先使用 app-specific locale API。

#### TC-AUTH-00003
- **用例期望**：文档要求新用户、Mock SMS 固定验证码、无 cooldown 污染；spec 也在开始前删用户和删 Redis 键。
- **实际行为**：报告显示同手机号被前序测试发过验证码，60s 冷却未过，导致无法再次获取。
- **对齐问题**：case 把手机号视为独占资源，但整个套件复用了固定号码，导致环境不隔离。
- **建议修改**：改成号码池/动态号码，或在 suite 级统一管理 cooldown 生命周期。

### E2E 失败用例

#### TC-AUTH-E2E-00001
- **用例期望**：文档要求启动后在 LoginScreen 点击“获取验证码”；spec 也直接 `aiTap("获取验证码")`。
- **实际行为**：报告显示界面是阿拉伯语，AI 只能看到阿语的发送验证码按钮，找不到中文文案。
- **对齐问题**：脚本把中文文案当稳定选择器，但产品本来就是中东市场多语言/RTL 应用。
- **建议修改**：改用“发送/获取验证码的按钮”这类语义选择器，或先在前置中统一 locale。

#### TC-ANALYTICS-E2E-00001
- **用例期望**：文档要求 Android 端已登录、full 同意、余额足、接收者在麦；spec 则从 `resetAndroidToLoginPage()` 后重新走 UI 登录，再进入房间送礼。
- **实际行为**：报告显示执行时 Android 停在手机主屏幕，连验证码输入框都看不到。
- **对齐问题**：case 假设 E2E 套件拥有独立、可控的 Android 初始状态；真实运行却继承了 AND 套件的 force-stop/桌面污染，前置 setup 不足以恢复。
- **建议修改**：E2E 每个用例独立做设备 reset 与 App 启动确认；不要依赖前一套件结束后的设备状态。

#### TC-GOVERNANCE-E2E-00002（spec 内部 ID：TC-GOVERNANCE-00002）
- **用例期望**：文档要求管理员与目标用户都已在房间里，目标用户已在 slot=2；spec 实际从登录页开始单端走 UI，再点“大厅第一张房间卡片”。
- **实际行为**：报告显示等待手机号输入框超时，Android 实际处于主屏幕（桌面）。
- **对齐问题**：这里有两层不对齐：一是 AND→E2E 的设备隔离失败；二是 case 需要 A/B 多端多角色，但 spec 只是单端 smoke，根本没有稳定建立 `U_ADMIN / U_TARGET / slot=2` 的前置。
- **建议修改**：用双设备/双账号编排该 case；在开始前先确认 App 已启动且角色、房间、麦位状态均已 seed 完成。

## 建议的修复方向

1. **先修 reset helper 的确定性**：`resetAndroidToMainPage/resetAndroidToLoginPage` 必须附带“结果页校验”，失败就继续清理导航状态/JWT，而不是盲目进入断言。
2. **把房间/余额/榜单/多用户夹具前置化**：不要再依赖“第一张房间卡片”“可能有其他用户”“可能榜单有数据”这类随机现场。
3. **统一 case doc 与 spec**：像 `SHELL-00002`、`THEME-00002` 这类明显漂移的文档，需要先定唯一事实源，再重写脚本。
4. **多语言选择器全部去文案硬编码**：中/英/阿 UI 统一改成语义选择器或稳定 testTag，不再写死“获取验证码”“退出登录”等单语言文本。
5. **环境门控测试单独标识**：网络控制、多设备、多角色、榜单 seed 等能力不满足时，应明确 skip/block，而不是跑成元素缺失。
6. **把“大厅可达性”从业务 case 中剥离**：ROOM 大厅一旦不稳定，会级联拖死 GIFT/CHAT/MIC/GOVERNANCE/SHELL/WALLET；应先有独立的“回到大厅 guard”。
