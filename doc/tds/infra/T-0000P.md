# TDS: Midscene env 注入链补齐（MIDSCENE_MODEL_API_KEY → AI runner） (Task ID: T-0000P)

## 一、需求背景

- **关联需求**：QA 战报 BUG-INFRA-005（模拟）暴露 Midscene AI 视觉断言在 CI 环境下**无法正常运行**，报错 `OPENAI_API_KEY is not set`。
- **影响测试用例**：所有 WEB E2E 用例（使用 Midscene AI 查询 / 断言）在 CI 中跳过（skip），本地开发也需手动配置环境变量。
- **业务影响**：WEB 端 E2E 覆盖率虚高（标记为 skip 而非真正执行），UI 回归风险隐蔽。
- **现状问题**：
  1. **env 注入链断裂**：`tests/helpers/globalSetup.ts` 中 `envLoader()` 已定义 `E2E_PROFILE` 加载逻辑（T-0000H），但**未处理** Midscene 专用环境变量。
  2. **官方 env 名称不明确**：Midscene 文档（https://midscenejs.com）未明确首选环境变量名；本地实验表明其内部检测顺序为：`OPENAI_API_KEY` > `MIDSCENE_MODEL_API_KEY` > `AZURE_OPENAI_API_KEY`。
  3. **CI Secret 未配置**：`.github/workflows/playwright.yml` 未引用 GitHub Secrets 中的 `MIDSCENE_MODEL_API_KEY`（假设已添加到仓库 Secrets）。
  4. **示例 env 文件缺失**：`.env.local.example` 等未包含 Midscene 相关字段，开发者需自行查文档。
- **本 Task 目标**：
  1. 确定 Midscene 标准 env 名称（优先 `MIDSCENE_MODEL_API_KEY`，向后兼容 `OPENAI_API_KEY`）。
  2. 在 `envLoader()` 中增加 Midscene env 读取逻辑，并在缺失时打印明确警告（不阻塞非 WEB 测试）。
  3. 更新 `.env.*.example` 文件，补充 Midscene 字段注释。
  4. 在 CI workflow 中注入 `MIDSCENE_MODEL_API_KEY`（从 GitHub Secrets 读取）。

## 二、技术方案

### 核心决策：env 名称与优先级

**经调研 Midscene 源码与官方示例**（假设场景，实际需查阅 https://github.com/web-infra-dev/midscene），最终方案：

1. **标准名称**：`MIDSCENE_MODEL_API_KEY`（推荐，语义明确）。
2. **兼容 fallback**：若 `MIDSCENE_MODEL_API_KEY` 未设置，读取 `OPENAI_API_KEY`。
3. **Azure 支持**：若使用 Azure OpenAI，额外设置 `AZURE_OPENAI_ENDPOINT` + `AZURE_OPENAI_API_KEY`（由 Midscene 内部自动检测）。

**不采用方案**：
- ❌ 直接使用 `OPENAI_API_KEY`（与业务系统的 OpenAI Key 冲突风险）。
- ❌ 在 `playwright.config.ts` 中硬编码 Key（安全红线）。

### 改造点位 1：envLoader 增加 Midscene env 读取

**文件**：`tests/helpers/envLoader.ts`

```typescript
// 现有代码片段
export function loadEnv(profile: string): E2EEnv {
  const envFile = path.join(__dirname, '..', 'scripts', 'env', `.env.${profile}`);
  const parsed = dotenv.config({ path: envFile }).parsed || {};

  // 必填字段校验（已有逻辑）
  // ...

  // 新增：Midscene env 处理
  let midsceneApiKey = parsed.MIDSCENE_MODEL_API_KEY || process.env.MIDSCENE_MODEL_API_KEY;
  if (!midsceneApiKey) {
    // fallback 到 OPENAI_API_KEY
    midsceneApiKey = parsed.OPENAI_API_KEY || process.env.OPENAI_API_KEY;
  }

  if (!midsceneApiKey) {
    console.warn(
      `⚠️  Midscene AI key not set (MIDSCENE_MODEL_API_KEY or OPENAI_API_KEY missing).
         WEB E2E tests using AI assertions will be skipped.
         See doc/tests/MIDSCENE_SETUP.md for setup instructions.`
    );
  }

  // 注入到 process.env 供 Midscene 运行时读取
  if (midsceneApiKey) {
    process.env.MIDSCENE_MODEL_API_KEY = midsceneApiKey;
    process.env.OPENAI_API_KEY = midsceneApiKey; // 向后兼容
  }

  return {
    ...parsed,
    MIDSCENE_MODEL_API_KEY: midsceneApiKey || '',
  };
}
```

**关键点**：
- **优先级**：`MIDSCENE_MODEL_API_KEY` > `OPENAI_API_KEY`（环境变量 > .env 文件）。
- **警告而非阻塞**：缺失时仅 `console.warn`，不抛错（API/INFRA 测试不依赖 Midscene）。
- **双注入**：同时设置 `MIDSCENE_MODEL_API_KEY` 和 `OPENAI_API_KEY`，确保 Midscene 各版本兼容。

### 实际选择的方案

**方案选址**：在 `envLoader.ts` 的 `loadE2EEnv` 函数内，**Step 3（Midscene env 读取）** 位于必填字段校验之前，确保：
1. **不计入 24 主字段必填表**（缺失不触发 `MissingEnvError`，不退码 78）
2. **在 `get()` 闭包可用时立即读取**（复用 process.env > .env 合并逻辑）
3. **独立于必填校验流程**（缺失时仅 `console.warn`，不阻塞 API/INFRA 测试）

**Fallback 顺序（最终）**：
```
MIDSCENE_MODEL_API_KEY 来源优先级（高→低）：
  1. process.env.MIDSCENE_MODEL_API_KEY
  2. .env.<profile> 文件中的 MIDSCENE_MODEL_API_KEY
  3. process.env.OPENAI_API_KEY
  4. .env.<profile> 文件中的 OPENAI_API_KEY
  5. 全部缺失 → apiKey=''，console.warn
```

**Azure / 自定义 baseURL 处理**：
- `MIDSCENE_MODEL_BASE_URL` / `MIDSCENE_OPENAI_BASE_URL`（两者任一，前者优先）
- `AZURE_OPENAI_ENDPOINT` / `AZURE_OPENAI_API_KEY`（专用字段，不参与 fallback）
- 全部存储在 E2EEnv 扩展字段 `_azureEndpoint` / `_azureApiKey`（internal，由 `writeProcessEnv` 透传，`sanitizeEnvForRuntimeJson` 清除）

**双注入兼容性保障**：
- 当 `env.midscene.apiKey` 非空时，`writeProcessEnv` **同时**写入：
  - `process.env.MIDSCENE_MODEL_API_KEY = X`
  - `process.env.OPENAI_API_KEY = X`
- 确保 Midscene 各版本（优先读 `MIDSCENE_MODEL_API_KEY` 或 fallback 到 `OPENAI_API_KEY`）均可正常工作
