# TDS: 观众席 Bottom Sheet (Task ID: T-30039)

## 一、背景与目标
- **关联需求**：E-10 观众席查看 ([设计文档](../../design/android/T-30039.md))
- **本 Task 的目标**：`AudienceBottomSheet` 展示房间内所有成员（麦上置顶 + 观众），支持分页与实时更新。

### State
```kotlin
data class AudienceUiState(
    val onMic: List<RoomMember> = emptyList(),
    val audience: List<RoomMember> = emptyList(),
    val total: Int = 0,
    val loading: Boolean = false,
    val currentPage: Int = 0,  // 初始值为 0，表示"尚未加载任何页"；首次 loadMoreMembers 请求 page=1
    val hasMore: Boolean = true
)
```

### 布局
- `ModalBottomSheet` 70% 高度
- Header "观众席 ($total)"
- `LazyColumn`：
  - Section header "麦上 (N)" + onMic 列表
  - Section header "观众 (N)" + audience 列表
- 每行 `MemberRow`：头像 + 昵称 + 角色徽章（owner 👑 / admin 🛡️）+ 在线时长
- 滚动触发 loadMore

### WS 集成
- `UserJoined`：state 追加到 audience 尾
- `UserLeft`：移除对应 user
- `MicTaken`：从 audience 移入 onMic
- `MicLeft`：从 onMic 移入 audience
- `AdminChanged`：更新 role 字段

### 点击回调
`onMemberClick(member) -> openUserActionSheet(member)`（T-30040）

### testTag
`Key('audience_sheet')`, `Key('audience_item_${userId}')`, `Key('audience_header_on_mic')`, `Key('audience_header_observers')`

### 性能
- 100 人房间：LazyColumn + 稳定 key (`userId`) 保持滚动流畅
- 头像用 Coil 缓存

## 五、Reviewer意见
### Round 1
- **状态**：FAIL
- **审查意见**：
  - **[HIGH-01]** `AudienceUiState.currentPage` 初始值为 1，`loadMoreMembers()` 第一次调用计算 `nextPage = 1 + 1 = 2`，直接请求第 2 页（API 1-indexed，见 `IRoomMemberRepository.kt` 注释）。`joinRoom()` 中无任何初始成员加载逻辑。结果：用户进入房间打开观众席时，所有已在房间的成员均不可见，仅未来通过 WS `UserJoined` 到达的用户才会出现。这是静默数据缺失的功能性 Bug，直接违反 A39-05 的语义正确性。A39-extra-4 测试因 `FakeRoomMemberRepository` 对任意页码返回相同数据而掩盖了该问题。
  - **[HIGH-02]** `NoOpRoomMemberRepository` 硬编码 `hasMore = true`，同时是 `RoomViewModel` 的默认构造参数。在未接入 DI 的集成测试或手动构造 ViewModel 的场景下，每次触发 `loadMoreMembers()` 都会成功（返回空列表但 `hasMore = true`），导致 `currentPage` 无限累积。应改为 `hasMore = false`。
  - **[MEDIUM-01]** `MicTaken` 兜底逻辑（`RoomViewModel.kt:454`）在用户未经 `UserJoined` 直接上麦时，仅用 `id + nickname` 创建 `RoomMember`，丢失 `role` 和 `avatarUrl` 字段。owner 上麦将无 👑 徽章，违反单一数据源原则。
  - **[MEDIUM-02]** A39-01「双空状态文案」测试仅验证 ViewModel 状态，未验证 UI 渲染。当前 UI 实现 onMic 区空时完全隐藏（不显示单独空态文案），两区均空时只渲染一条「暂无成员」，与「双空」规格存在歧义，应明确 UI 规格并补充 Compose UI 层断言。
  - **[LOW-01]** `RoomViewModel` 类级 KDoc WS 消息映射表未涵盖 T-30039 新增的 `audienceState` 更新逻辑（`UserJoined/UserLeft/MicTaken/MicLeft/AdminChanged`），文档滞后于实现。
- **改进建议**：
  1. 修复 `currentPage` 初始值（改为 0）或在 `joinRoom()` 中新增 `loadInitialMembers()` 加载第 1 页，并修复 A39-05 测试补充对 `fakeMemberRepo.calls.first().second == 1` 的断言。
  2. 将 `NoOpRoomMemberRepository` 的 `hasMore` 改为 `false`。
  3. `MicTaken` 兜底 RoomMember 创建处加 FIXME 注释，后续协商 WS 补字段或 API 按需补全。
  4. 明确「双空状态文案」UI 规格并补充对应 Compose 测试断言。
