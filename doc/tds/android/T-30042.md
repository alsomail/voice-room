### UserKicked 流程
```
WS UserKicked 到达
   ↓
RoomViewModel.onKicked(reason, cooldown_sec)
   ↓
显示全屏 UserKickedDialog
   ├ 文案 "你已被移出房间，原因：XXX，10 分钟后可再次进入"
   ├ [知道了] 按钮
   ↓ 点击后
KickCooldownStore.save(roomId, until = now + 600s)
离开 RoomScreen，返回 Hall
   ↓ Hall 点击同一房间卡片前
if (KickCooldownStore.get(roomId).isFuture) {
    Toast("你暂时不能进入此房间，还剩 N 秒")
    return
}
```

### UserMuted 流程
```kotlin
// RoomViewModel 收到 UserMuted 且 target_user_id == self
when (type) {
    "mic"  -> muteViewModel.startMicCountdown(expiresAt)
    "chat" -> muteViewModel.startChatCountdown(expiresAt)
}
// duration_sec=0 则解除
```

`MuteStatusChip` 在房间顶部显示 "你已被禁{麦|言} N:NN" 每秒刷新，到期自动消失。

## 五、Reviewer意见
### Round 1
- **状态**：未通过（BLOCKED）
- **审查意见**：
  1. **[CRITICAL] RoomScreen 未接入任何 T-30042 UI 组件**：`kickedState` 未被收集（`UserKickedDialog` 从不显示）；`events.collect{}` 仅处理 `ClearInput`，`RoomEvent.UserMuted` 分支缺失（`MuteCountdownViewModel` 永远不会被调用）；`MuteStatusChip` 未出现在 RoomScreen 中。整个功能从用户视角完全不可见，属于死代码集成。
  2. **[HIGH] KickCooldownStore 两 ViewModel 各持独立实例**：`RoomViewModel` 与 `HallViewModel` 均以默认参数各自 `new InMemoryKickCooldownStore()`，`HallViewModel.Factory` 也不传入 store。`acknowledgeKick()` 写入实例 A，`enterRoom()` 读取实例 B，cooldown 拦截在生产环境完全失效（K42-03 验收标准形同虚设）。
  3. **[HIGH] Clock 未注入 RoomViewModel / HallViewModel**：`acknowledgeKick()`（L526）与 `enterRoom()`（L187）均直接调用 `System.currentTimeMillis()`，违反 TDS 审查要点 #8 的可注入 Clock 设计要求，亦导致 K42-02 测试依赖真实系统时间而脆弱。
  4. **[LOW] `cooldownMin` 整数除法截断**：`UserKickedDialog` L86 `state.cooldownSec / 60` 遇非 60 倍数时显示偏低，建议向上取整。
- **改进建议**：
  - 在 `RoomScreen` 中收集 `kickedState`，条件渲染 `UserKickedDialog`；补全 `events.collect{}` 的 `UserMuted` 分支；显示 `MuteStatusChip`。
  - 在 Application 层提供 `KickCooldownStore` 单例（或通过 DI），`HallViewModel.Factory` 增加 `kickCooldownStore` 构造参数，确保两个 ViewModel 共享同一实例。
  - `RoomViewModel` 和 `HallViewModel` 各自增加 `clock: Clock = SystemClock()` 构造参数，替换所有 `System.currentTimeMillis()` 硬调用。
