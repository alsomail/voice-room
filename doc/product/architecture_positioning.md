## 四端职责划分

| 端 | 定位 | 核心用户 | 主要功能 | 部署方式 |
|----|------|----------|----------|----------|
| **Android** | C 端用户应用 | 普通用户 + 主播 | 语音聊天、送礼、充值、社交 | Google Play |
| **App Server** | C 端业务后端 | - | 用户认证、房间管理、实时通信 (WebSocket)、支付回调 | 公网部署 (AWS/阿里云) |
| **Admin Server** | B 端管理后端 | - | 后台管理 API、数据统计、报表生成 | **内网部署** (VPN 访问) |
| **Web** | B 端管理后台 | 运营人员 + 客服 | 用户管理、内容审核、数据统计、素材配置 | 内网部署 |

---

## 为什么要分离 App Server 和 Admin Server？

| 维度 | App Server | Admin Server | 原因 |
|------|-----------|--------------|------|
| **安全性** | 公网暴露 | **内网隔离 + VPN** | 管理后台涉及敏感操作（充值/封禁），必须隔离 |
| **并发要求** | **极高** (10万+ QPS) | 低 (< 100 QPS) | 用户端需要抗住大流量，后台只有运营人员使用 |
| **数据库** | 读多写少 (主从分离) | **读写权限** | 后台需要直接修改数据（充值/扣款） |
| **实时性** | WebSocket 长连接 | HTTP 短连接 | 用户端需要实时推送，后台不需要 |
| **部署策略** | 多区域部署 + CDN | 单区域内网 | 用户端需要低延迟，后台访问量小 |
| **监控** | APM + 全链路追踪 | 基础日志 | 用户端需要精细监控，后台简单即可 |

## 数据库设计

**Phase 0 共享同一个 PostgreSQL**，通过不同账号权限隔离：
- `app_server_user`: 只读权限（SELECT）+ 部分写权限（INSERT users, rooms 等）
- `admin_server_user`: 全部读写权限 + DDL 权限

**Phase 1 后可考虑读写分离**：
- App Server → 读库（主从复制）
- Admin Server → 主库（可写）

## 重要说明

- 🚫 Web 端**不是**用户端，普通用户无法访问
- ✅ Web 端是**运营管理系统**，需要通过 VPN 访问
- ✅ 两个 Server 共享数据库，但 API 完全独立
- ✅ 未来可能增加 iOS 端（与 Android 功能一致，复用 App Server）
