# TDS: 密码房进房校验 + 锁定机制 (Task ID: T-00026)

## 一、背景与目标
- **关联需求**：E-10 密码房进房流程 ([phase1_room_governance §2.1.1 + §3.2](../../product/phase1_room_governance.md))
- **本 Task 的目标**：提供 POST `/rooms/:id/verify-password`，签发 60s 短期 token；失败 5 次锁定 30min；WS JoinRoom 读 token 才允许进入。

## 二、方案设计

### 接口：POST /api/v1/rooms/:id/verify-password
Request: `{ "password": "123456" }`

成功 200:
```json
{ "code":0, "data": { "access_token":"<jwt-60s>" } }
```
token claim：
```json
{ "sub": user_id, "room_id": xxx, "iat": now, "exp": now+60, "iss":"voiceroom-room-access" }
```

错误：
| HTTP | code | 说明 |
|------|------|------|
| 400 | 40003 | password 非 6 位数字 |
| 404 | 40400 | 房间不存在/已关闭 |
| 400 | 40014 | 非密码房 |
| 401 | 40103 | 密码错误（未达锁定） |
| 429 | 42910 | 已锁定 (payload `{ locked_remaining_sec }`) |

### Redis Key
| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `pwd_fail:{user_id}:{room_id}` | Int | 1800s | 失败计数 |
| `pwd_lock:{user_id}:{room_id}` | String | 1800s | 锁定标记 |

流程：
```
1. 若 pwd_lock 存在：返回 42910 + TTL
2. bcrypt 验证 password
   - 成功 → DEL pwd_fail → 签发 room_access token
   - 失败 → INCR pwd_fail; 若 ==5 → SETEX pwd_lock 1800 + INCR 保留
           返回 40103 payload `{ remaining_attempts: 5-count }`
```

### WS JoinRoom 读 token
- `JoinRoom` payload 增 `access_token?: string`
- 对密码房：
  - 无 token → 返回 `PASSWORD_REQUIRED(40104)`
  - 有 token：解析验证 iss='voiceroom-room-access' && sub==user_id && room_id 匹配 && 未过期
  - token 过期 → `TOKEN_EXPIRED(40105)`

### 并发安全
失败计数 `INCR` 是原子的，锁定 key 的 SETEX 用 `SET ... NX EX 1800` 防止已锁定后重复 set，失败计数到 5 时同一批并发仅一条触发 SETEX。

## 五、Reviewer意见
### Round 1
- **状态**：通过 ✅
- **审查意见**：
  - [CRITICAL-✅] JWT iss 验证：`decode_room_access_token` 通过 `validation.set_issuer(&[ROOM_ACCESS_ISS])` 强制校验 `iss == "voiceroom-room-access"`，session JWT 无法被复用进密码房，满足 PR26-12 防护要求。
  - [CRITICAL-✅] SET NX EX 并发锁：`RealRoomPasswordRedis::set_nx_ex` 使用 `ExistenceCheck::NX + SetExpiry::EX`（单条原子命令），`FakeRoomPasswordRedis` 内存实现同步 NX 语义，PR26-10 并发测试已验证。
  - [CRITICAL-✅] WS JoinRoom room_id 校验：handler.rs 第 118 行明确比较 `claims.room_id != room_id.to_string()`，返回 40106 INVALID_TOKEN，PR26-12 单元测试覆盖。
  - [HIGH-✅] 失败计数触发锁：代码使用 `count >= MAX_FAIL_COUNT`（而非 `== 5`），差异说明：一旦计数 ==5 锁被创建，后续请求在步骤 1（get_ttl 检查）即被拦截，count 不会超过 5+并发数量；并发超量时 set_nx_ex NX 防止重复写，功能等价且更健壮。
  - [HIGH-✅] remaining_sec 来源正确：初次锁定返回常量 LOCK_TTL_SECS（刚创建时等于实际 TTL），后续锁定请求在步骤 1 通过 `get_ttl` 读取 Redis 真实 TTL，PR26-06 覆盖验证。
  - [HIGH-✅] 成功后清除 pwd_fail：`redis.del(&fk).await?` 在 bcrypt 验证通过后即执行，PR26-11 测试验证 key_exists 返回 false。
  - [MEDIUM] `incr_with_ttl` 生产实现中 INCR + EXPIRE 非原子（两条命令），极端情况下进程崩溃于两命令之间导致 pwd_fail key 无 TTL 永久存在，实际影响极低（仅造成该用户该房间的失败计数无法自动清除，可接受）。
  - [LOW] handler.rs 步骤注释编号重复（第 150/152 行均标注"步骤 7"），不影响功能。
- **改进建议**：
  - [MEDIUM/可选] 生产实现可考虑用 Lua 脚本将 INCR + EXPIRE 合并为单条原子操作，消除崩溃窗口，但非阻断性问题，可在后续迭代改进。
