### AdjustBalanceModal 设计
- Props：`userId`, `currentBalance`, `open`, `onClose`, `onSuccess`
- 表单字段：
  - `amount`: InputNumber（可负数），必填非零，绝对值 ≤10,000,000
  - `reason`: Input.TextArea 2-200 字符，必填
- 当 `amount < 0` 时，Modal 头部显示红色警示"扣减操作"，二次 `Modal.confirm`
- 调用 `POST /api/v1/admin/users/:id/wallet/adjust`，成功后 `onSuccess(newBalance)` 让父 Drawer 刷新
- `isConfirming` 防重复提交

### GiftManagementPage 设计
- `Table` 列：图标（预览）/ code / 名称(双语) / 价格 / tier / 上架状态 Switch / 操作（编辑/删除）
- 筛选：tier 下拉 + 状态筛选（全部/上架/下架/已删）
- 顶部"新增礼物"按钮打开 `GiftEditModal`
- `Switch` 本地状态 optimistic update + 失败回滚

### GiftEditModal
- 表单字段对齐 T-10014
- 图片上传：`Upload` 组件调 `/api/v1/admin/gifts/upload`，返回 URL 回填
- 预览区实时显示图标
- 动效 URL 可选（Lottie JSON）

  **HIGH（1 个）**：

  1. **[HIGH] `statusFilter === 'inactive'` 逻辑错误，验收标准 #4 未实现**
     文件：`app/web/src/features/gift/GiftManagementPage.tsx:85-91`
     问题：当用户选择「已下架」筛选时，代码进入 `else` 分支并设置 `include_inactive = true`，该参数会让 API 返回**全部礼物（上架 + 下架）**，与"已下架"语义完全相反——与「全部」选项效果相同，用户实际看不到仅下架过滤的结果。这直接违反验收标准 4「礼物列表可筛选 tier/状态」。

     ```ts
     // BAD: 'inactive' 与 'all' 逻辑完全相同
     } else {                        // statusFilter === 'inactive'
       params.include_inactive = true; // 返回全部礼物，非仅下架
     }
     ```

     修复方案：后端 `GET /admin/gifts` 目前只有 `include_inactive` boolean 参数，无法直接过滤「仅下架」。需在前端请求 `include_inactive=true` 后，对结果进行客户端过滤：

     ```ts
     // GOOD: 拉取全量后客户端过滤
     const result = await adminListGifts(params); // include_inactive=true
     const items = statusFilter === 'inactive'
       ? result.items.filter((g) => !g.is_active)
       : result.items;
     setGifts(items);
     setTotal(items.length); // total 需同步修正
     ```

     同时，测试 `G08` 也需补充对 `inactive` 过滤结果的断言。

  **MEDIUM（2 个）**：

  2. **[MEDIUM] 新增 API 函数均缺少 `AbortSignal` 参数，存在竞态风险**
     文件：`app/web/src/core/network/apiClient.ts:567-618`
     问题：`adminListGifts` / `adminCreateGift` / `adminUpdateGift` / `adminDeleteGift` 四个函数均未接受 `signal?: AbortSignal` 参数，而项目既有模式（`adminGetUserDetail`、`adminGetRoomDetail`）一致支持 signal 以防竞态。`GiftManagementPage` 在用户快速切换 tier/状态筛选时，旧请求结果可能覆盖新结果。

     ```ts
     // BAD: 无法取消
     export async function adminListGifts(params?: AdminListGiftsParams): Promise<AdminGiftsData>

     // GOOD: 与项目既有模式一致
     export async function adminListGifts(
       params?: AdminListGiftsParams,
       signal?: AbortSignal,
     ): Promise<AdminGiftsData>
     ```

     同时，`GiftManagementPage.fetchGifts` 应在 `useEffect` 的 cleanup 中 `controller.abort()`，与 `useUserDetail` / `useRoomsPage` 保持一致。

  3. **[MEDIUM] "调整余额"按钮无前端 RBAC 控制，`cs` 角色可见按钮但 API 必定返回 403**
