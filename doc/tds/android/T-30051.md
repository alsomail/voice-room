# T-30051 Android WS 接收链路可观测性增强

> **关联 Bug**：BUG-CHAT-WS-ANDROID-SILENT
> **关联前序**：T-00046（Server 侧广播可观测性）已闭环

## §一 问题背景

Round 18 E2E 实证：Server 端 WS 广播链路完全正常（`broadcast_to_room_inner` 日志 `DELTA_SENT=+2, DELTA_DROPPED=0, AFTER_RECIPIENTS_TOTAL>=2`），但 Android logcat 中**无任何** OkHttp/WebSocket/RoomMessage/ChatMessage 相关日志，公屏依然不渲染。

这意味着 Android 客户端 WS 接收链路是**完全黑盒**的——无法判定故障点究竟在：
- TCP/WS 帧到达 OkHttp 之前？
- OkHttp `onMessage` 收到了但 JSON 解析失败？
- 解析成功但路由分发未命中 `RoomMessage` 分支？
- ViewModel 处理了但 UI 收集器没拿到？

**本任务只做可观测性增强（注入日志），不修改任何业务逻辑。** 目的是让下一轮 E2E 能在 logcat 中拿到决策证据，定位真根因。

## §二 链路分析

通过实读现有代码，Android WS 接收链路的 5 个关键节点如下：

| # | 节点 | 文件 | 当前位置 | 当前日志状态 |
|---|------|------|---------|-------------|
| 1 | WebSocket 帧接收入口 | `core/ws/OkHttpWebSocketClient.kt` | `WebSocketListener.onMessage(text)` 第 118-124 行 | ❌ 无任何接收日志 |
| 2 | Envelope JSON 解析 | `feature/room/RoomViewModel.kt` | `handleWsMessage(raw)` 第 892-897 行 (`JsonParser.parseString`) | ❌ 解析失败被静默 catch |
| 3 | 消息类型路由分发 | `feature/room/RoomViewModel.kt` | `when (type)` 第 912 行起 | ❌ 无 dispatch 日志 |
| 4 | RoomViewModel WS 入口 | `feature/room/RoomViewModel.kt` | `observeWsMessages` 第 876-884 行（订阅 `wsClient.state` 中的 `Message` 状态） | ❌ 订阅 callback 无日志 |
| 5 | UI 收集器 | `feature/room/ChatMessageList.kt` | `LaunchedEffect(deduplicated.size)` 第 50-54 行 | ❌ 列表更新无日志 |

> 备注：OkHttp `onMessage` 仅有 `onFailure` 写了 `Log.w`（line 143），正常接收路径无任何 `Log.*` 调用。

## §三 本轮目标

**只补日志，不动业务逻辑。**

1. 在 5 个关键节点注入 `android.util.Log` 调用（项目未引入 Timber，沿用现有 `android.util.Log`，与 `OkHttpWebSocketClient` 已有的 `Log.w`/`Log.e` 风格一致）。
2. 日志前缀采用统一约定 (`ws: received` / `ws: parse start` / `ws: parse ok` / `ws: parse failed` / `ws: dispatch` / `rvm: onWsMessage` / `ui: chatMessages collected`)，便于 e2e 自动化抓取与决策。
3. 日志内容**只截断头部 80 字符**，不打印完整正文（PII 保护，详见 §六 决策树）。

## §四 日志方案

### 节点 1 — `OkHttpWebSocketClient.onMessage` 入口

文件：`app/android/app/src/main/java/com/voice/room/android/core/ws/OkHttpWebSocketClient.kt`

```kotlin
override fun onMessage(webSocket: WebSocket, text: String) {
    Log.i(TAG, "ws: received text len=${text.length}, head=${text.take(80)}")
    if (text.contains("\"type\":\"pong\"")) {
        startHeartbeat()
    }
    _state.value = WebSocketState.Message(text)
}
```

### 节点 2 — `RoomViewModel.handleWsMessage` 解析点

文件：`app/android/app/src/main/java/com/voice/room/android/feature/room/RoomViewModel.kt`

```kotlin
private fun handleWsMessage(raw: String) {
    Log.d(TAG, "ws: parse start len=${raw.length}")
    val json = try {
        JsonParser.parseString(raw).asJsonObject
    } catch (e: Exception) {
        Log.e(TAG, "ws: parse failed head=${raw.take(80)}", e)
        return
    }
    val type = json.get("type")?.asString ?: run {
        Log.w(TAG, "ws: parse ok but type missing head=${raw.take(80)}")
        return
    }
    Log.d(TAG, "ws: parse ok type=$type")
    // ...
}
```

### 节点 3 — 路由分发

在 `when (type)` 入口前插入：

```kotlin
Log.d(TAG, "ws: dispatch type=$type roomId=${state.roomId}")
```

### 节点 4 — `observeWsMessages` 入口（rvm 视角）

```kotlin
wsClient.state.collect { wsState ->
    if (wsState is WebSocketState.Message) {
        Log.i(TAG, "rvm: onWsMessage len=${wsState.text.length}")
        handleWsMessage(wsState.text)
    }
}
```

### 节点 5 — UI 收集器

文件：`app/android/app/src/main/java/com/voice/room/android/feature/room/ChatMessageList.kt`

```kotlin
LaunchedEffect(deduplicated.size) {
    Log.d("ChatMessageList", "ui: chatMessages collected size=${deduplicated.size}")
    if (deduplicated.isNotEmpty()) {
        listState.animateScrollToItem(deduplicated.lastIndex)
    }
}
```

### TAG 约定

- `OkHttpWebSocketClient`：沿用现有 `private const val TAG = "OkHttpWebSocketClient"`
- `RoomViewModel`：新增 `private const val TAG = "RoomViewModel"`（companion object）
- `ChatMessageList`：直接字面量 `"ChatMessageList"`（Compose 函数无 companion）

## §五 测试计划

### 单元测试（JVM 可运行）

新增 `app/android/app/src/test/java/com/voice/room/android/feature/room/RoomViewModelLoggingTest.kt`：

1. **测试 1**：FakeWS 推送合法 `RoomMessage` JSON → ViewModel 应进入 `handleWsMessage` 分支并触发 `messages` 状态更新（已有逻辑覆盖，本测试只确认日志路径不抛异常）。
2. **测试 2**：FakeWS 推送非法 JSON（如 `"not-a-json"`）→ ViewModel 应安全 swallow 并保留 `_uiState` 为原 Success（验证 `parse failed` 分支不崩溃）。
3. **测试 3**：FakeWS 推送 `{"type":null}` 之类缺 type 字段的 JSON → ViewModel 安全 swallow（验证 `type missing` 分支）。

> 单元测试无法直接断言 `android.util.Log`（默认 mock 抛异常），因此采用 **Robolectric/ShadowLog** 或 **行为回归**（确保新增日志不引入崩溃）。本仓库已有 JVM 单测（如 `RoomViewModelChatTest`），沿用 Robolectric-free 模式 + try-catch 规避 `Log` 在 JVM 单测中抛 `RuntimeException("not mocked")` 的风险——必要时通过依赖注入 logger 接口；本轮先采用 **直接 try-catch 包装**或 **行为断言**（不直接断言 Log 调用次数，只断言 ViewModel 状态正确流转）。

## §六 决策树（供下一轮 E2E 使用）

| 现象（logcat 抓取） | 真根因 | 后续 Task |
|---------------------|-------|-----------|
| **a1**：完全没有 `ws: received` | TCP/WS 握手层未收到帧（防火墙 / DNS / TLS / 路由 NAT 等基础设施层） | 补 `OkHttpClient` interceptor 日志 / 抓 `tcpdump` |
| **a2**：有 `ws: received` 但出现 `ws: parse failed` 或仅有 `ws: parse ok type=Other` 没 `RoomMessage` | JSON schema 不匹配，server 实际 type ≠ "RoomMessage"，或字段大小写错位 | 对比 server `chat.rs` 出网 payload，更新 `handleWsMessage` 分支 |
| **a3**：`ws: parse ok type=RoomMessage` 但无 `rvm: onWsMessage` 或没进 `dispatch` | `wsClient.state` 收集断流（Flow 取消、scope 销毁） | 检查 `observeWsMessages` 启动时机与 viewModelScope 生命周期 |
| **a4**：`rvm: onWsMessage` + `ws: dispatch type=RoomMessage` 都有但 `ui: chatMessages collected size` 不增长 | `_uiState.value` 更新未触发 Compose 重组（state 引用未变 / Success 包装层未透传） | 修复 ViewState immutable copy 链路 |
| **a5**：`ui: chatMessages collected size` 增长但屏幕仍空白 | LazyColumn 渲染 / 主题 / Modifier 问题 | UI 层独立 task |
