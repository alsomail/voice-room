# TDS: Dev/Staging Mock 充值通道 [Dev-only] (Task ID: T-00055)

## 一、背景与目标
- **关联需求**：[phase1_payment_billing.md §9.2 Dev Mock 通道](../../product/phase1_payment_billing.md) — 让 CI / 自动化测试 / 内部 demo 在不调起 BillingClient 的前提下完成入账闭环。
- **本 Task 的目标**：实现 `POST /api/v1/_dev/mock_recharge`，仅在 `feature="dev_payment_mock"` + 配置 `payment.mock_enabled=true` 双开关激活；production profile 启动时若 mock_enabled=true 直接 panic 阻止误发布。

## 二、方案设计

### 核心数据流
```
Android dev/staging flavor (T-30065) → POST /api/v1/_dev/mock_recharge { user_id, sku_id, force_outcome }
   ↓
[启动期] cfg(feature="dev_payment_mock") && config.payment.mock_enabled → 注册路由
[启动期] config.profile=production && mock_enabled=true → panic!("dev_payment_mock cannot run in production")
   ↓
DevMockRechargeService::handle
  ├─ force_outcome="success" → 注入 MockBillingPort 走与 T-00052 完全相同的 verify 链路
  │     · order INSERT PENDING → VERIFIED → CREDITED → ACKED
  │     · wallet_transactions.source='dev_mock'（隔离财务报表）
  │     · WS BalanceUpdated reason='dev_mock_recharge'
  ├─ force_outcome="fail"    → state=FAILED 不入账
  └─ force_outcome="pending" → state=VERIFYING；二次同参调用且 force_outcome=success → 推进至 ACKED
   ↓
1s 内同 (user_id, sku_id) 重复请求 → 幂等 200 返回首次结果
```

## 三、TDD 验收用例
- [ ] **production 构建**：`cargo build --release --no-default-features` 产物中 grep `mock_recharge` 应为空
- [ ] **production+enabled 启动**：env=production, mock_enabled=true → 启动 panic
- [ ] **success**：发起 mock → 200，余额 += sku.diamonds，wallet_transactions.source='dev_mock'，BalanceUpdated 推送
- [ ] **fail**：state=FAILED；余额不变；不广播
- [ ] **pending**：首次 → state=VERIFYING；同参第二次 force_outcome=success → 推进至 ACKED
- [ ] **幂等**：1s 内重复同参 → 200 返回首次结果，DB 不新增订单
- [ ] **未认证**：401
- [ ] **隔离**：财务报表 SQL 过滤 `source != 'dev_mock'` 时 dev 流水不出现

### 🔴 协议路径绑定的强制验收
- [ ] **PROTO-1**：dev 集成测试断言 URL 严格 `/_dev/mock_recharge`；production 构建产物 grep 为空
- [ ] **PROTO-2**：MockBillingPort 与 T-00052 共用 `GooglePlayBillingPort` trait；不允许另起入账分支
