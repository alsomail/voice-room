# TDS: 手动赠送/撤销贵族 API (Task ID: T-10031)

## 一、背景与目标

- **关联需求**：[phase1_nobility.md §3.5 贵族特权细则](../../product/phase1_nobility.md)；[phase1_nobility.md §6 运营管理](../../product/phase1_nobility.md)
- **本 Task 的目标**：在 Admin Server 提供 super_admin 专属的手动赠送/撤销贵族接口，写 `user_nobles` 强事务；赠送时合并有效期（重复赠送叠加）；撤销时立即生效并触发 `NobleExpired`；全程通过 Redis Pub/Sub `admin:events` 通知 App Server 发送 WS 信令 `NobleChanged`；所有操作落 `admin_logs` + `noble_history`。

## 二、方案设计

### 核心数据流

```
POST /api/v1/admin/users/:id/noble/grant
  { "tier_id": "duke", "duration_days": 30, "reason": "VIP补偿" }
  ↓ JWT + RBAC(super_admin only — 其他角色 403)
  ↓ 参数校验：tier_id 存在且 is_active=true / duration_days ∈ [1,365] / reason 必填非空
  BEGIN TX
    SELECT user_nobles WHERE user_id=:id FOR UPDATE
    ① 无贵族记录 → INSERT user_nobles {
        tier_id, start_at=now(), current_period_start=now(),
        expire_at = now() + duration_days,
        renew_channel='admin_grant', auto_renew=false }
    ② 已有贵族（同级或不同级）→ UPDATE user_nobles SET
        tier_id=:tier_id,
        expire_at = MAX(expire_at, now()) + duration_days   ← 合并有效期
        renew_channel='admin_grant'
    INSERT noble_history { event='admin_grant', from_tier=old_tier, to_tier=tier_id,
                           actor='admin:<admin_uuid>', payload={duration_days, reason} }
    INSERT admin_logs { action='noble_grant', target_type='user', target_id=user_id,
                        detail={tier_id, duration_days, reason, expire_at} }
  COMMIT
  ↓ Redis PUBLISH admin:events {
      type: "noble_grant",
      payload: { user_id, from_tier_id, to_tier_id, expire_at, reason: "admin_grant" },
      admin_id, ts }
    App Server 收到后：
      → 单播 NobleChanged 给目标用户
      → 若用户在房间，向房间广播 NobleChanged
  ↓ 200 { user_noble: {...} }

POST /api/v1/admin/users/:id/noble/revoke
  { "reason": "违规处罚" }
  ↓ JWT + RBAC(super_admin only)
  ↓ 参数校验：reason 必填
  BEGIN TX
    SELECT user_nobles WHERE user_id=:id (不存在 → 404)
    记录 revoked_tier = tier_id
    DELETE FROM user_nobles WHERE user_id=:id      ← 立即生效
    INSERT noble_history { event='admin_revoke', from_tier=revoked_tier, to_tier=null,
                           actor='admin:<admin_uuid>', payload={reason} }
    INSERT admin_logs { action='noble_revoke', target_type='user', target_id=user_id,
                        detail={revoked_tier, reason} }
  COMMIT
  ↓ Redis PUBLISH admin:events {
      type: "noble_revoke",
      payload: { user_id, from_tier_id: revoked_tier, to_tier_id: null,
                 expire_at: null, reason: "admin_revoke" },
      admin_id, ts }
    App Server 收到后：
      → 单播 NobleChanged(to_tier=null) 给目标用户
      → 单播 NobleExpired 给目标用户
      → 若用户在房间，向房间广播 NobleChanged
  ↓ 200
```

**Redis 事件 Payload 格式**（Admin Server 发出）：

```jsonc
// noble_grant
{
  "type": "noble_grant",
  "payload": {
    "user_id": "<uuid>",
    "from_tier_id": "knight",   // 可空（首次赠送）
    "to_tier_id": "duke",
    "expire_at": "2026-07-09T00:00:00Z",
    "reason": "admin_grant"
  },
  "admin_id": "<admin_uuid>",
  "ts": 1746788688000
}

// noble_revoke
{
  "type": "noble_revoke",
  "payload": {
    "user_id": "<uuid>",
    "from_tier_id": "duke",
    "to_tier_id": null,
    "expire_at": null,
    "reason": "admin_revoke"
  },
  "admin_id": "<admin_uuid>",
  "ts": 1746788688000
}
```

App Server 订阅并将上述事件转换为 `NobleChanged` WS 信令，payload.reason 对应值 `admin_grant` / `admin_revoke`。

**admin_logs 格式**：

```jsonc
// noble_grant
{ "action": "noble_grant", "target_type": "user", "target_id": "<user_uuid>",
  "detail": { "tier_id": "duke", "duration_days": 30, "reason": "VIP补偿",
              "expire_at": "2026-07-09T00:00:00Z" } }

// noble_revoke
{ "action": "noble_revoke", "target_type": "user", "target_id": "<user_uuid>",
  "detail": { "revoked_tier": "duke", "reason": "违规处罚" } }
```

## 三、TDD 验收用例

- [ ] **GR01** super_admin 赠送成功：`user_nobles` +1 / `noble_history` +1 / `admin_logs` +1 / Redis publish 命中
- [ ] **GR02** 非 super_admin（operator/cs/finance）→ 403
- [ ] **GR03** reason 为空字符串 → 400（必填校验）
- [ ] **GR04** reason 缺省（body 无 reason key）→ 400
- [ ] **GR05** tier_id 不存在 → 404
- [ ] **GR06** tier_id 已下架（is_active=false）→ 409（40914 TIER_INACTIVE）
- [ ] **GR07** 重复赠送合并有效期：用户已有 duke expire_at=T+10d，再赠 30d → expire_at=T+10d+30d（合并，非覆盖）
- [ ] **GR08** 赠送不扣用户钻石余额（diamond_balance 不变）
- [ ] **GR09** `renew_channel` 字段值为 `admin_grant`
- [ ] **GR10** revoke 成功：`user_nobles` 记录删除，`noble_history` +1（event=admin_revoke），Redis publish 命中
- [ ] **GR11** revoke 对无贵族的用户 → 404
- [ ] **GR12** revoke 不退钻石（产品规则）
- [ ] **GR13** 事务原子性：模拟 `insert noble_history` 阶段抛错 → `user_nobles` 无变更，`admin_logs` 无新记录

### 🔴 协议路径绑定的强制验收

- [ ] **PROTO-1**：grant URL 严格 `POST /api/v1/admin/users/:id/noble/grant`；revoke URL 严格 `POST /api/v1/admin/users/:id/noble/revoke`；集成测试实际请求 URL 断言匹配
- [ ] **PROTO-2**：Redis publish payload `reason` 字段值为 `"admin_grant"` / `"admin_revoke"`（不是其他值），与 `NobleChanged.payload.reason` 枚举一致（[nobility_api.md §10.4.1](../../protocol/nobility_api.md#1041-noblechangedsc-单播--sroom-广播)）；Fake publisher 断言 payload 结构
- [ ] **PROTO-3**：revoke 事件 payload `to_tier_id` 为 `null`，触发 App Server 发 `NobleExpired`（集成测试通过 FakeEventPublisher 验证 publish 参数）

### Round 1

- **状态**：🔴 未通过（BLOCKED）
- **审查意见**：
  - **[P0-1] 赠送/撤销实现文件完全缺失**：`grant_controller.rs`、`grant_service.rs` 均未创建（nobility 模块本身也不存在，见 T-10030 P0-1）。
  - **[P0-2] 仅 super_admin 校验缺失**：`require_role("super_admin")` 门卫无任何实现代码路径；非 super_admin 角色无法被拦截（403）。
  - **[P0-3] 赠送原子 upsert 缺失（红线）**：`BEGIN TX → SELECT user_nobles FOR UPDATE → ① INSERT 或 ② UPDATE expire_at = MAX(expire_at, now()) + duration_days → INSERT noble_history → INSERT admin_logs → COMMIT` 全部缺失。重复赠送有效期合并逻辑（GR-07）无代码实现。
  - **[P0-4] Redis Pub/Sub noble_grant / noble_revoke 事件缺失**：赠送/撤销后应 PUBLISH `admin:events { type: "noble_grant"/"noble_revoke", payload.reason: "admin_grant"/"admin_revoke" }`，完全缺失，App Server 无法收到通知触发 `NobleChanged` / `NobleExpired` WS 信令。
  - **[P0-5] 路由未注册**：`POST /api/v1/admin/users/:id/noble/grant` 与 `POST /api/v1/admin/users/:id/noble/revoke` 均未在 bootstrap 注册。
  - **[P0-6] 集成测试缺失**：GR01～GR13 / PROTO-1/2/3 均未实现，`tests/grant_test.rs` 文件不存在。
- **改进建议**：
  1. 依赖 T-10030 先创建 nobility 模块骨架后，在 `grant_service.rs` 实现原子事务：SELECT FOR UPDATE → 判断已有贵族记录 → upsert（合并 expire_at）→ INSERT noble_history → INSERT admin_logs → COMMIT。
  2. 重点验证 GR-07：`expire_at = MAX(expire_at, now()) + interval '?' day`，不可覆盖式替换。
  3. Redis publish payload 中 `reason` 字段必须严格为 `"admin_grant"` / `"admin_revoke"`（字符串完全匹配，见 nobility_api.md §10.4.1 枚举值），PROTO-2 为验收红线。
  4. 集成测试用 `FakeEventPublisher` 断言 publish 参数（GR01、GR10、PROTO-3）。
