# TDS: 事件表 Schema + 分区 + HTTP 接收 API (Task ID: T-00022)

## 一、背景与目标
- **关联需求**：E-07.5 埋点基建 ([phase1_observability.md](../../product/phase1_observability.md))
- **本 Task 的目标**：建立 `events` 分区表 + 每日自动建分区 + HTTP 批量接收接口（兼容未登录）；提供统一 `EventWriter` 供 WS 通道（T-00023）复用。

## 二、方案设计

### 数据流
```
Android POST /api/v1/events/batch { events:[...] }  (可匿名)
   or  WS ReportEvent (T-00023)
   ↓
EventWriter.persist(batch)
   ├── 校验每条 event（size / required fields）
   ├── 截断 properties >8KB（记日志）
   ├── 以 server_ts=now() 统一写入
   └── COPY FROM STDIN 批量写入（sqlx-core bulk insert）
   ↓
每日 00:00 Asia/Riyadh：PartitionScheduler 创建次日分区
```

### 迁移脚本 007_create_events.sql
```sql
CREATE TABLE IF NOT EXISTS events (
    id            UUID DEFAULT gen_random_uuid(),
    user_id       UUID,
    device_id     VARCHAR(64) NOT NULL,
    event_name    VARCHAR(64) NOT NULL,
    properties    JSONB NOT NULL DEFAULT '{}'::jsonb,
    session_id    VARCHAR(64),
    client_ts     TIMESTAMPTZ,
    server_ts     TIMESTAMPTZ NOT NULL DEFAULT now(),
    app_version   VARCHAR(16),
    os_version    VARCHAR(32),
    locale        VARCHAR(16),
    network_type  VARCHAR(16),
    PRIMARY KEY (id, server_ts)  -- 分区键需含 server_ts
) PARTITION BY RANGE (server_ts);

-- 首日分区（示例：部署当日）
CREATE TABLE IF NOT EXISTS events_20260421 PARTITION OF events
    FOR VALUES FROM ('2026-04-21 00:00:00+03') TO ('2026-04-22 00:00:00+03');

CREATE INDEX idx_events_user_ts ON events (user_id, server_ts DESC) WHERE user_id IS NOT NULL;
CREATE INDEX idx_events_name_ts ON events (event_name, server_ts DESC);
```

### PartitionScheduler
- `tokio-cron-scheduler`，Cron `0 0 23 * * *`（Riyadh 23:00 = UTC 20:00，留 1h 提前窗口）
- 读取 Redis `events:partition:last_created` → 补偿创建所有缺失分区
- 函数签名：`create_next_partition(date: NaiveDate) -> Result<()>`

### HTTP 接口
**POST /api/v1/events/batch** （**JWT 可选**）
```json
{
  "events": [
    {
      "event_name": "gift_send_success",
      "device_id": "uuid",
      "user_id": "uuid|null",
      "session_id": "uuid",
      "client_ts": 1720000000000,
      "properties": { "gift_id": "uuid", "count": 1, "total_price": 520 },
      "app_version": "1.2.0", "os_version":"Android 14",
      "locale":"ar-SA", "network_type":"wifi"
    }
  ]
}
```
- 单请求 ≤100 events；超限返回 `BATCH_TOO_LARGE(40204)` 但**仍写前 100 条**
- 单 event properties JSON 序列化后 >8KB 截断为 `{"_truncated": true}` 并 log warn
- 若 JWT 存在，server 覆盖 user_id（即使请求体带不同 user_id 也以 JWT 为准，并 log warn）
- 若无 JWT，则允许 `user_id=null`，`device_id` 必填

**响应 200**
```json
{ "code":0, "data":{ "received": 98, "rejected_indices":[12, 45] } }
```

### EventWriter 核心
```rust
pub struct EventWriter { pool: PgPool }

impl EventWriter {
    pub async fn persist(&self, batch: Vec<EventInput>, jwt_user_id: Option<Uuid>)
        -> Result<PersistResult>;
}

pub struct EventInput { /* 来自 HTTP / WS 统一结构 */ }
pub struct PersistResult { pub received: usize, pub rejected_indices: Vec<usize> }
```
- 内部采用 `sqlx::QueryBuilder` 构造多行 INSERT 批量写入
- properties 超 8KB 截断后 `event_name` 加后缀 `_truncated` 便于过滤

## 五、Reviewer意见
### Round 1
- **状态**：未通过
- **审查意见**：
  - **[HIGH] `scheduler.rs` 分区时间边界计算错误（UTC 偏移方向反转）**
    - 问题文件：`app/server/src/core/analytics/scheduler.rs` 第 56–77 行
    - Asia/Riyadh = UTC+3，Riyadh 时间 00:00 对应 UTC 前一天 21:00。`events_20260421` 正确范围应为 `[2026-04-20 21:00+00, 2026-04-21 21:00+00)`，但代码生成 `[2026-04-21 21:00+00, 2026-04-22 21:00+00)`（实为 Riyadh 04-22），整体偏差一天。
    - 生产后果：Scheduler 创建的所有分区与其名称不匹配，Riyadh 当日存在分区空隙，事件 INSERT 将抛出 "no partition found for row"，造成当日数据丢失。
    - `tests::s04` 同样断言了错误值，注释 "00:00 Riyadh = 21:00 UTC" 方向颠倒，掩盖了该 Bug。
  - **[MEDIUM] Scheduler 使用 24h interval 而非 TDS 规定的 Cron（`0 0 23 * * *` UTC）**
    - 无法保证每日固定窗口执行，建议后续用 `tokio-cron-scheduler` 替换。
- **改进建议**：
  1. 修复 `create_partition_if_not_exists` 中的时间边界计算，将 `date` 改为 `date - 1 day` 作为 `from_ts` 的日期部分，`date` 作为 `to_ts` 的日期部分（或直接使用 `chrono_tz::Asia::Riyadh` 进行严格时区转换）。
  2. 同步修正 `tests::s04` 的断言值，改为 `from_ts = "2026-04-20 21:00:00+00"`，`to_ts = "2026-04-21 21:00:00+00"`。
  3. `writer.rs` 第 64 行 `DateTime::from_timestamp()` 建议替换为 `DateTime::from_timestamp_millis(ts_ms)` 消除 deprecated warning。
