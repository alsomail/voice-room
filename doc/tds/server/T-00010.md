## 1. 功能描述

房主关闭（结束）自己拥有的语音房间。关闭后，房间 status 变为 `closed`，
后续对该房间发起的 `GET /api/v1/rooms/:id` 将返回 404。

---

## 2. 接口规范

### 请求

```
DELETE /api/v1/rooms/:id
Authorization: Bearer <JWT>
```

| 参数   | 类型   | 说明                   |
|--------|--------|----------------------|
| `:id`  | string | 房间 UUID（路径参数） |

### 响应

#### 成功（200）

```json
{
  "code": 0,
  "data": null,
  "request_id": "<uuid>"
}
```

#### 错误

| HTTP | code  | 场景                              |
|------|-------|----------------------------------|
| 400  | 40003 | `:id` 不是合法 UUID               |
| 401  | 40101 | 未携带 Authorization header       |
| 401  | 40102 | JWT 已过期                        |
| 403  | 40301 | 当前用户不是该房间的房主           |
| 404  | 40400 | 房间不存在或已软删除               |
| 409  | 40901 | 房间已经处于 closed 状态           |

## 4. 实现细节

### 4.1 Repository 新方法

#### `find_room_any_status(room_id: Uuid) -> Result<Option<RoomModel>, AppError>`

- **过滤条件**：`deleted_at IS NULL`（不过滤 status）
- **PG SQL**：
  ```sql
  SELECT id, owner_id, title, room_type, member_count, status, password_hash,
         max_members, created_at, updated_at, deleted_at
  FROM rooms
  WHERE id = $1 AND deleted_at IS NULL
  ```
- **Fake**：从 `HashMap` 查找，filter `deleted_at.is_none()`

#### `set_room_closed(room_id: Uuid) -> Result<bool, AppError>`

- **PG SQL**：
  ```sql
  UPDATE rooms SET status = 'closed', updated_at = NOW()
  WHERE id = $1 AND deleted_at IS NULL
  ```
- **返回**：`rows_affected > 0` → `true`
- **Fake**：修改 `HashMap` 中对应 room 的 status 字段为 `"closed"`

### 4.2 Service 方法（`close_room`）

```
1. find_room_any_status(room_id)
   └─ None → NotFound
2. room.owner_id != current_user_id → Forbidden
3. room.status == "closed" → RoomAlreadyClosed
4. set_room_closed(room_id)
5. Ok(())
```

### 4.3 Controller Handler（`close_room`）

- 提取 `Path<String>` → 手动 `Uuid::parse_str`，失败 → `ValidationError` 400/40003
- 需要 `AuthContext` extractor（JWT 鉴权）
- 调用 `room_service.close_room(room_id, ctx.user_id)`
- 成功 → `200 + data: null`

### 4.4 Routes

```
/api/v1/rooms/{id}  →  get(get_room).delete(close_room)
```

### 7.3 发现的问题

#### 🟡 MEDIUM — M-01：`set_room_closed` 的 `bool` 返回值被静默丢弃

**位置**：`app/server/src/modules/room/service.rs` 第 226 行

```rust
// 当前实现：bool 返回值被 ? 运算符隐式丢弃
self.room_repo.set_room_closed(room_id).await?;
```

**问题**：  
`set_room_closed` 的契约是"返回 `rows_affected > 0`"（TDS §4.1），用于感知是否真正写入成功。  
在极低概率的 TOCTOU 场景下（`find_room_any_status` 与 `UPDATE` 之间房间被软删除），`UPDATE` 影响 0 行，方法返回 `Ok(false)`，但 `service` 忽略该值直接返回 `Ok(())`，导致客户端收到假成功的 200 响应，且缺乏可观测性。

**建议修复**（保留 MVP 可接受的语义，仅增加可观测性）：

```rust
let affected = self.room_repo.set_room_closed(room_id).await?;
if !affected {
    // TOCTOU: 极低概率——房间在 find 与 update 之间被软删除
    tracing::warn!(
        room_id = %room_id,
        "set_room_closed affected 0 rows, possible TOCTOU soft-delete race"
    );
}
```

#### 🟢 LOW — L-01：`set_room_closed` SQL 缺少 `AND status = 'active'` 防御条件

**位置**：`app/server/src/modules/room/repository.rs` 第 187-195 行

当前 SQL：
```sql
UPDATE rooms SET status = 'closed', updated_at = NOW()
WHERE id = $1 AND deleted_at IS NULL
```

添加 `AND status = 'active'` 后，`rows_affected` 的语义更精确（只有真正从 active 转为 closed 才算 1 行），对 M-01 的可观测性修复也更有意义。不影响当前逻辑正确性，属可选优化。

---

#### 🟢 LOW — L-02：UUID 解析逻辑在 `get_room` 与 `close_room` 之间重复

**位置**：`app/server/src/modules/room/controller.rs`

两个 handler 均含相同的 6 行 UUID 解析 + 错误返回代码。可提取为模块内 inline 辅助函数减少重复：

```rust
fn parse_room_id(id: &str, request_id: &str) -> Result<Uuid, axum::response::Response> {
    uuid::Uuid::parse_str(id).map_err(|_| {
        err_response(AppError::ValidationError(format!("invalid room id format: {id:?}")), request_id)
    })
}
```
