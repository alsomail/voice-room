#### §1 前置依赖

| 项 | 要求 | 说明 |

#### §2 冷启动 5 步

```
1. git clone + npm install
   $ git clone <repo>
   $ cd voice-room
   $ npm install        # 根 + tests/scripts 依赖

2. 复制三档 env 模板（5 端 env 一键就位）
   $ cp .env.example .env
   $ cp tests/scripts/env/.env.local.example tests/scripts/env/.env.local
   # （staging/prod 不在本步，见 §5）

3. 启动 docker 依赖（Postgres + Redis）
   $ docker compose up -d postgres redis

4. 启动业务服务（按需开 3 个终端）
   终端 A: APP_PROFILE=dev cargo run -p server
   终端 B: ADMIN_PROFILE=dev cargo run -p admin-server
   终端 C: npm --prefix app/web run dev

5. 跑 E2E（首次推荐 smoke 子集 ~30s 验证链路）
   $ npm run preflight                              # 5 端健康检查 ≤1s
   $ npm run e2e:prod-smoke -- --list               # 列出 @prod-safe 用例
   $ npm run e2e:local                              # 本地全量 ~5min
```

> **冷启动时间预算**：步骤 1（npm install ~2min，依网速）+ 步骤 2~4（~1min）+ 步骤 5 preflight（~1s）+ 首条 smoke 用例（~30s）≈ **5 分钟达成首条用例全绿**。

#### §5 staging / prod-safe 远端凭据流程

> **占位 SOP**（具体凭据渠道由团队 SRE/QA Lead 在落地时填入）：

1. **凭据获取**：
   - 在 1Password 团队保险库 `voice-room-e2e` 找 `staging-runner` 条目，或开 SRE 工单（标签 `e2e/credentials`）请求；
   - 凭据包含：`DATABASE_URL`、`REDIS_URL`、`E2E_VALID_TOKEN`、`E2E_ADMIN_TOKEN`、`MIDSCENE_MODEL_API_KEY`（如启用 WEB 用例）等（完整字段表见 T-0000F §2.3 + T-0000K §1）。
2. **写入 env**：
   ```bash
   cp tests/scripts/env/.env.staging.example tests/scripts/env/.env.staging
   # 用编辑器填入（绝对禁止 echo / 粘贴到聊天工具 / commit）
   ```
3. **跑用例**：
   ```bash
   # macOS / Linux
   npm run e2e:staging
   # 仅跑 @prod-safe 标签（read-only smoke 验证远端连通性）
   npm run e2e:staging -- --grep "@prod-safe"
   ```
4. **prod-safe 子集**（生产环境只读探针）：
   ```bash
   cp tests/scripts/env/.env.prod.example tests/scripts/env/.env.prod
   npm run e2e:prod-smoke
   ```

> **⚠️ Windows shell 双引号警示**：
> - PowerShell / cmd 下 `--grep '@prod-safe'`（单引号）会被解释为字面 `'@prod-safe'`（含引号本身），**永远 0 命中** → 静默通过假绿；
> - **统一使用双引号** `--grep "@prod-safe"`，与 T-0000I `package.json` `e2e:prod-smoke` 字面 `\"@prod-safe\"` 严格对齐；
> - 推荐 Windows 用户走 WSL2 或 Git Bash 规避 shell 差异。

> **🔒 安全红线**（来自 T-0000E §2.9 + T-0000K §3.1）：
> - `.env.{local,staging,prod}` 全量 gitignore，永不入库；
> - 凭据永不进 `.e2e-runtime.json`（T-0000K `sanitizeEnvForRuntimeJson` 已强制脱敏 `midscene.apiKey`）；
> - 错误日志中 Key 自动 `***` 替换（envLoader 已实现）。

#### §7 故障 FAQ（≥ 6 条）

1. **端口被占（5173 / 3000 / 3001 / 5432 / 6379）** → `lsof -nP -iTCP:<port>`，杀进程或改 `.env` 对应字段；docker 起的服务用 `docker compose down` 全部回收。
2. **Docker 拉镜像慢** → 配置国内镜像源（`/etc/docker/daemon.json` registry-mirrors），或预先 `docker pull postgres:16 redis:7`。
3. **envLoader fail-fast `MissingEnvError`** → 对照 `tests/scripts/env/.env.local.example` 补齐缺失字段；切勿用 `process.env.X ?? '默认值'` 绕过（T-0000J 已禁止该模式）。
4. **Windows shell 单引号假绿** → `--grep '@prod-safe'` 在 PowerShell/cmd 下匹配 0 条且静默通过；统一双引号 `"@prod-safe"`，详 §5。
5. **Midscene 401 / 429 / 超时** → 参考 [`./MIDSCENE_SETUP.md §5 限流回退表`](./MIDSCENE_SETUP.md) + §6 FAQ；Key 配额耗尽**不会自动 skip**（避免静默通过），需更换 Key 或减并发。
6. **`.e2e-runtime.json` 脏残留导致 worker 复用旧值** → 删除 `tests/scripts/.e2e-runtime.json` 重跑；该文件由 globalSetup 写入（T-0000H），不应手工编辑或 commit（已 gitignore）。
7. **`@prod-safe` 标签 0 命中** → 检查 fuzzy 拼写（`@prod_safe` / `@prodsafe` / `@prod-save`，T-0000J U-11 守护），统一字面 `@prod-safe`（中划线 + 全小写）。
8. **首次 npm install 慢** → 跨平台依赖 `cross-env`、`@playwright/test`、`@midscene/web` 体积较大，建议配置 npm 镜像（`npm config set registry https://registry.npmmirror.com`）。
