# TDS: 封禁对话框 (Task ID: T-20008)

## 一、背景与目标

- **关联需求**：用户管理模块——运营人员对违规用户执行封禁操作，需要选择封禁时长、封禁原因并填写备注
- **本 Task 的目标**：实现 `BanModal` 封禁对话框，集成到 `UserDetailDrawer`（T-20007）中的封禁/解封按钮，调用 T-10009 后端接口

## 二、方案设计

### §2.1 数据流

```
UserDetailDrawer.onBanClick(userId)
    │
    ▼
UsersPage.banUserId = userId
    │
    ▼
BanModal(userId=banUserId)
    ├── Form（封禁时长/原因/备注）
    └── 确认 → Modal.confirm 二次确认
              └── useBanUser.ban(userId, req) → adminBanUser(userId, req)
                        └── POST /api/v1/admin/users/:id/ban
                                    │
                                    ▼
                              onSuccess(userId)
                              → 关闭 Drawer + 刷新用户列表 + 成功 Toast

UserDetailDrawer.onUnbanClick(userId)
    │
    ▼
UsersPage: Modal.confirm 二次确认
    └── adminBanUser(userId, {action:'unban'}) → onSuccess
```

### §2.3 接口规格

```
POST /api/v1/admin/users/:id/ban
Authorization: Bearer <admin-jwt>
Content-Type: application/json

{
  "action": "ban",
  "duration": 1440,        // 分钟，null=永久
  "reason": "违规内容",
  "remark": "备注（可选）"
}
```

成功响应（200）：{ code: 0, data: null }
错误码：40401（用户不存在）、40901（已是该状态，幂等）、40301（无权限）

### §2.4 类型定义

```typescript
export interface AdminBanUserRequest {
  action: 'ban' | 'unban';
  duration?: number | null; // 分钟，null=永久
  reason?: string;
  remark?: string;
}

export async function adminBanUser(
  userId: string,
  req: AdminBanUserRequest,
  signal?: AbortSignal,
): Promise<void>
```

### §2.5 BanModal 组件设计

Props：
```typescript
interface BanModalProps {
  userId: string | null;  // null=关闭
  onClose: () => void;
  onSuccess: (userId: string) => void;
}
```

表单字段：
- duration: Select（24h=1440/72h=4320/7天=10080/30天=43200/永久=null）
- reason: Select（违规内容/骚扰他人/刷单行为/其他）
- remark: TextArea（最大 200 字，可选）

点击确认 → Modal.confirm 二次确认 → 调用 useBanUser.ban() → onSuccess

### §2.6 解封流程

在 UsersPage 的 onUnbanClick 中：
```typescript
Modal.confirm({
  title: t('users.ban.confirmUnban'),
  onOk: async () => {
    await adminBanUser(userId, { action: 'unban' });
    // 刷新用户列表
    onSuccess(userId);
  }
})
```

#### NEW HIGH（必须修复，阻塞合并）

**[NEW-HIGH] `handleClose` 未重置 `isConfirming` — 可导致提交按钮永久 disabled**
- 文件：`BanModal.tsx:98-102`
- 问题：`handleClose` 仅调用 `form.resetFields() + setBanError(null) + onClose()`，**缺少 `setIsConfirming(false)`**。
  复现路径：① 打开 BanModal → 点「提交」→ `isConfirming=true` → `Modal.confirm` 弹出；② 用户点击外层 Modal 的 ❌ 或按 ESC 关闭 BanModal → `handleClose` 执行，`isConfirming` 保持 `true`；③ BanModal 变为 `open=false`，但**组件未 unmount**，`Modal.confirm` 仍作为 portal 悬浮在页面；④ 运营立刻点击另一个用户重新打开 BanModal → 提交按钮 `disabled={true}` 永久失效，直到旧 `Modal.confirm` 被操作触发 `afterClose`。
  `destroyOnHidden` 不能解决此问题：该 prop 仅销毁 antd Modal 内部渲染内容，BanModal 组件自身未 unmount，`isConfirming` state 不被重置。
- 修复：在 `handleClose` 中补充 `setIsConfirming(false)`：
  ```typescript
  const handleClose = () => {
    form.resetFields();
    setBanError(null);
    setIsConfirming(false); // 防止外层 Modal 先于 confirm 关闭导致 state 残留
    onClose();
  };
  ```

- **改进建议**：修复 NEW-HIGH 后重新提交 Round 3 Review；LOW-1/LOW-2（useMemo + React.memo）可在 Round 3 一并处理。
