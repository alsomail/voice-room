- [ ] **缺陷 1**：[级别 P1] **密码房锁定字段名与协议不一致 — Android 客户端永远拿不到真实剩余锁定时长**
  - **文件与行号**：`app/android/app/src/main/java/com/voice/room/android/data/room/RetrofitRoomRepository.kt:146-148`
  - **问题说明**：服务端按 TDS T-00026 与 `app/server/src/modules/room/controller.rs:273-281` 在 HTTP 429 响应体中返回 `data.locked_remaining_sec`（**秒**）；但 Android 解析时读取的字段是 `errBody.data?.get("remaining_minutes")`，并以"分钟"单位包装为 `PasswordLockedException(remainingMinutes)`，再向 UI（`HallViewModel:160`）暴露 `PasswordDialogState.Locked(remainingMinutes)`。结果：
    1. 字段名不匹配 → 永远 fallback 到默认值 `30`；
    2. 单位不匹配 → 服务端的"秒"被当作"分钟"展示给用户。

  - **修复建议**：改为读取 `locked_remaining_sec`（Double→Int），并将 `PasswordLockedException` / `PasswordDialogState.Locked` 字段语义统一改为秒（或在 ViewModel 内 `secs / 60` 显式换算）。同时为该错误路径补一个 Repository 单元测试断言新字段。
  - **TDD 修复记录**（2026-04-25）：

- [ ] **缺陷 2**：[级别 P1] **`bcrypt::verify` 同步调用阻塞 Tokio 运行时**
  - **文件与行号**：`app/server/src/modules/room/password.rs:135-136`
  - **问题说明**：`verify_password` 是 async 函数，但内部直接调用同步 `bcrypt::verify(input_password, &hash)`，单次耗时随 cost 因子达到 100–300ms。同模块的 `service.rs:109` 创建房间时已正确使用 `tokio::task::spawn_blocking` 包装 `bcrypt::hash`，本处实现不一致。当多个用户并发触发密码房进入时，Tokio 工作线程会被占满，导致 WS 心跳、接口 P99 严重退化（与 LLM_RULES "禁止在 async 路径阻塞 runtime" 红线冲突）。
  - **修复建议**：
    ```rust
    let is_valid = tokio::task::spawn_blocking(move || bcrypt::verify(&input, &hash))
        .await
        .map_err(...)?
        .map_err(...)?;
    ```
    并注意 `input_password`、`hash` 需 `move` 进闭包（克隆为 owned String）。
  - **TDD 修复记录**（2026-04-25）：
    - 修复方式：`bcrypt::verify` 调用包入 `tokio::task::spawn_blocking`（克隆 input/hash 为 owned `String`），`JoinError` / `BcryptError` 分别 map 到 `AppError::Internal`。
    - 改动文件：`app/server/src/modules/room/password.rs`。
    - 新增/调整测试：`pr26_12_bcrypt_does_not_block_tokio_runtime`（multi_thread runtime + worker_threads=2，验证并发 verify 时间不超过单次 bcrypt 串行耗时阈值，确保 runtime 不阻塞）。

- [ ] **缺陷 3**：[级别 P2] **关闭/PATCH 房间 TOCTOU 非原子 — 并发关闭会出现误报 200 而非 409**
  - **文件与行号**：
    - App Server：`app/server/src/modules/room/service.rs:253-277` + `repository.rs:198-207`
    - Admin Server：`app/adminServer/src/modules/room/service.rs:111-139` + `repository.rs:117-128`
  - **问题说明**：`close_room` / `force_close_room` 均采用"先 SELECT `find_room_any_status` 校验 status，再 UPDATE `set_room_closed`"两步式，无事务无悲观锁。`set_room_closed` 的 SQL 仅 `WHERE id = $1 AND deleted_at IS NULL`，未带 `AND status = 'active'` 守卫。两个并发关闭请求（如：房主关闭 + 管理员强制关闭）都能通过预检并都会 UPDATE 成功，第二次本应返回 `40901 RoomAlreadyClosed`，实际返回 `200 ok`，违反 protocol §3.4 错误码契约。
  - **修复建议**：将状态收敛到 SQL 层：`UPDATE rooms SET status='closed', updated_at=NOW() WHERE id=$1 AND status='active' AND deleted_at IS NULL`，根据 `rows_affected()` 区分返回值——`= 0` 时再做一次 `find_room_any_status` 决定 404/409。Fake 实现需同步更新断言。
  - **TDD 修复记录**（2026-04-25）：

    - 改动文件：`app/server/src/modules/room/{repository.rs,service.rs}`、`app/adminServer/src/modules/room/{repository.rs,service.rs}`。
    - 新增/调整测试：Server `uc08_concurrent_close_only_one_succeeds`（并发关闭，断言一次 200 + 一次 RoomAlreadyClosed），Admin `fr04_concurrent_force_close_only_one_succeeds`（同语义），现有 FCR 用例同步更新签名。
