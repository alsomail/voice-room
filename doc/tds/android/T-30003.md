# TDS: JWT 拦截器 (Task ID: T-30003)

## 一、背景与目标
- **关联需求**：所有需要认证的 C 端 API
- **本 Task 的目标**：实现 OkHttp AuthInterceptor，自动为每个请求附加 JWT token，并处理 401 未授权响应（清除 token + 跳转登录页）。
- **前置依赖**：T-00004 (JWT 中间件), T-30002

## 二、方案设计
### 核心数据流
```
OkHttp Request → AuthInterceptor → TokenManager.getToken()
  → 有 token？→ 添加 Authorization: Bearer xxx Header → 发送请求
  → 收到 401？→ TokenManager.clearToken() → 跳转登录页
  → 无 token → 直接发送请求（公开接口如 send-code/login）
```

### HIGH 问题（阻塞合并）

#### H-01：并发 401 竞态条件 — `DefaultUnauthorizedHandler` 无原子性保护

**文件：** `DefaultUnauthorizedHandler.kt:42-45`

**问题描述：**  
当应用内存在多个并发 OkHttp 请求同时收到 401 响应时（例如进入大厅后批量发起的多个 API 请求），每个 OkHttp 后台线程都会独立执行：

```kotlin
// AuthInterceptor.kt:50 — 每个 OkHttp 线程各自 runBlocking
if (response.code == 401) {
    runBlocking { unauthorizedHandler.onUnauthorized() }
}
```

而 `onUnauthorized()` 内部没有任何并发保护：

```kotlin
// DefaultUnauthorizedHandler.kt:42-45 — 无互斥，可被多线程并发执行
override suspend fun onUnauthorized() {
    tokenManager.clearToken()        // 多次调用：虽然幂等，无害
    _unauthorizedEvent.tryEmit(Unit) // ⚠️ 多次调用：可能向 UI 多次发射导航事件
}
```

`_unauthorizedEvent` 的 `extraBufferCapacity=1 + DROP_OLDEST` 仅能保证在**同一时刻缓冲区满**时丢弃旧事件。若订阅方（ViewModel）处理速度快于后续 `tryEmit` 到达速度（实际场景中极易发生），多个事件均会被消费，导致 `navController.navigate("login") { popUpTo(0) { inclusive = true } }` 被执行多次，引发导航栈异常（重复弹出 / 崩溃）。

**修复方案：** 使用 `AtomicBoolean` 实现 compare-and-set 一次性语义，或使用 `Mutex` + 布尔标记：

```kotlin
// 方案A：AtomicBoolean（非挂起，适合跨线程场景）
private val handled = AtomicBoolean(false)

override suspend fun onUnauthorized() {
    if (!handled.compareAndSet(false, true)) return  // 后续调用直接忽略
    tokenManager.clearToken()
    _unauthorizedEvent.tryEmit(Unit)
}
```

> ⚠️ 注意：`AtomicBoolean` 需在用户重新登录成功后重置（`handled.set(false)`），否则下次 Token 失效时无法再触发登出。建议将重置逻辑放入 `ITokenManager.saveToken()` 调用成功后，或通过 DI 在登录成功后重建 `DefaultUnauthorizedHandler` 实例。

### MEDIUM 问题（建议修复）

#### M-01：`DefaultUnauthorizedHandler` 未强制单例，DI 误配置可导致事件通道断开

**文件：** `DefaultUnauthorizedHandler.kt:26`

**问题描述：**  
`DefaultUnauthorizedHandler` 是普通 `class`，没有任何 Singleton 作用域注解或文档约束。若 DI 框架（Hilt/Koin）配置时以 non-singleton 方式注入，`AuthInterceptor` 持有的实例 A 发射事件，而 UI/ViewModel 订阅的是实例 B，导致 401 事件永远不会到达 UI，用户 Token 失效后无法自动跳转登录页（静默失效，严重体验 Bug）。

**修复方案：** 在类 KDoc 中明确标注单例要求，并在 DI 模块中以 `@Singleton`（Hilt）或 `single { }` block（Koin）提供：

```kotlin
/**
 * ...
 * ⚠️ **必须以应用级单例注入**（Hilt `@Singleton` 或 Koin `single { }`）。
 * AuthInterceptor 与 UI 层必须共享同一实例，否则事件通道断开。
 */
class DefaultUnauthorizedHandler(...)
```

### 第二轮 HIGH 问题（阻塞合并）

#### H-01（新）：`resetUnauthorized()` 从未在生产代码中被调用 — 修复链路断裂

**文件：** `LoginViewModel.kt:130-141` / `DefaultUnauthorizedHandler.kt:73`

**问题描述：**  
`resetUnauthorized()` 方法虽然在接口和实现中均已正确定义，但在整个 `main/` 生产代码中**从未被调用**。全量搜索结果显示，调用方仅存在于测试 Fake 的 no-op 实现中：

```
grep -rn "resetUnauthorized" app/android/app/src/main/  → 0 结果（仅定义，无调用）
grep -rn "resetUnauthorized" app/android/app/src/test/  → 仅 Fake stub 和测试方法本身
```

`LoginViewModel` 登录成功后的流程如下：

```kotlin
// LoginViewModel.kt:131-141
authRepository.login(phone, code)
    .onSuccess { result ->
        runCatching { tokenManager.saveToken(result.token) }
            .onSuccess {
                _uiState.update { it.copy(isLoading = false, isLoginSuccess = true, ...) }
                _navEvent.emit(NavEvent.NavigateToHall)
                // ⚠️ 此处应调用 unauthorizedHandler.resetUnauthorized()
                // 但 LoginViewModel 根本未注入 UnauthorizedHandler
            }
    }
```

`LoginViewModel.Factory` 也只接受 `IAuthRepository` 和 `ITokenManager`，完全没有 `UnauthorizedHandler` 的注入路径：

```kotlin
class Factory(
    private val authRepository: IAuthRepository,
    private val tokenManager: ITokenManager   // UnauthorizedHandler 缺失
) : ViewModelProvider.Factory
```

**实际影响（用户可见的严重 Bug）：**

```
① 用户正常使用 → Token 过期
② 并发 401 → handled.compareAndSet(false, true) 成功 → 登出，跳转登录页 ✅
③ 用户重新登录 → saveToken 成功 → navEvent 导航到大厅
④ resetUnauthorized() ← 永远不会被调用 → handled 永远为 true
⑤ 用户继续使用 → 新 Token 又过期（例如 7 天后）
⑥ 新的 401 → compareAndSet(true, true) → false → 直接 return
⑦ clearToken() 不执行，导航事件不发射 → 用户被困在大厅，所有 API 静默 401 失败
```

一个 token 周期后，用户将永远无法被自动登出，AtomicBoolean 修复变成一次性的、不可恢复的状态机。

**修复方案：**

1. 在 `LoginViewModel` 构造注入 `UnauthorizedHandler`
2. `saveToken` 成功后调用 `resetUnauthorized()`

```kotlin
// LoginViewModel.kt — 修复示例
class LoginViewModel(
    private val authRepository: IAuthRepository = NoOpAuthRepository,
    private val tokenManager: ITokenManager = NoOpTokenManager,
    private val unauthorizedHandler: UnauthorizedHandler = NoOpUnauthorizedHandler  // 新增
) : ViewModel() {

    fun onLogin() {
        // ...
        runCatching { tokenManager.saveToken(result.token) }
            .onSuccess {
                unauthorizedHandler.resetUnauthorized()   // ← 新增：重置，使下次 401 可再次触发登出
                _uiState.update { ... }
                _navEvent.emit(NavEvent.NavigateToHall)
            }
    }

    private object NoOpUnauthorizedHandler : UnauthorizedHandler {
        override suspend fun onUnauthorized() = Unit
        override fun resetUnauthorized() = Unit
    }
}

// LoginViewModel.Factory — 同步更新
class Factory(
    private val authRepository: IAuthRepository,
    private val tokenManager: ITokenManager,
    private val unauthorizedHandler: UnauthorizedHandler   // ← 新增
) : ViewModelProvider.Factory {
    override fun <T : ViewModel> create(modelClass: Class<T>): T =
        LoginViewModel(authRepository, tokenManager, unauthorizedHandler) as T
}
```

3. 补充测试：`LoginViewModelTest` 中验证登录成功后 `resetUnauthorized()` 被调用

```kotlin
}
```

### R2 H-01 终态确认：`resetUnauthorized()` 调用时机

**检查路径（LoginViewModel.kt:133–154）：**

```kotlin
authRepository.login(phone, code)
    .onSuccess { result ->
        runCatching { tokenManager.saveToken(result.token) }
            .onSuccess {
                unauthorizedHandler.resetUnauthorized()   // ✅ saveToken 成功后立即调用
                _uiState.update { it.copy(isLoading = false, isLoginSuccess = true, isNewUser = result.isNew) }
                _navEvent.emit(NavEvent.NavigateToHall)
            }
            .onFailure {
                // ✅ saveToken 失败 → 不调用 reset（token 未保存，无需重置）
                _uiState.update { it.copy(isLoading = false, error = "登录失败，Token 存储异常，请重试") }
            }
    }
    .onFailure { error ->
        // ✅ 登录 API 失败 → 不调用 reset（用户未成功登录）
        _uiState.update { it.copy(isLoading = false, error = mapLoginError(error)) }
    }
```

调用时机正确：仅在 `saveToken` 成功的唯一路径上触发，两个失败路径均不触发。
