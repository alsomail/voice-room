# T-30044: 禁麦/禁言 UI 反馈 + 抱麦集成

> **归属端**: Android
> **模块**: Governance / Mic / Chat
> **前置依赖**: T-00029 (MuteUser 信令), T-00030 (ForceTakeMic/ForceLeaveMic), T-30042 (被禁 Chip)
> **参考**: [phase1_room_governance.md §2.1.5](../../product/phase1_room_governance.md), [business_flows.md §2.8.6~2.8.8](../../product/business_flows.md)

## 需求描述

将房间治理的"被动反馈"真正落到 UI：输入框禁用、麦位按钮禁用、被抱上麦自动申请权限。

## A. 被禁麦反馈

| UI 点 | 反馈 |
|------|------|
| 空麦位 "+" 按钮 | `enabled=false` + 灰 |
| 点灰按钮 | Toast "你已被禁麦，N 分钟后恢复" |
| 已在麦上时 | Server 自动下麦（T-00029），客户端收 MicLeft → 麦位动画消失 |

### 状态来源

- RoomViewModel.selfMuteState.micMutedUntil: Long
- 每秒刷新判断当前是否仍处于禁麦期

## B. 被禁言反馈

| UI 点 | 反馈 |
|------|------|
| 聊天输入框 | `enabled=false` |
| 输入框 placeholder | "你已被禁言，N 分钟后恢复" |
| 发送按钮 | 不可见/灰 |
| Server 双重拦截 | 即使绕过前端发送，Server 返回 CHAT_MUTED（兜底） |

## C. 被抱上麦（ForceTakeMic 到达自己）

```
收到 WS MicTaken { user_id: self, slot_index, forced_by }
   ↓
1. 全屏 Toast "你已被 {forced_by.nickname} 抱上 N 号麦位"
2. 检查麦克风权限
   ├─ 已授权 → 直接 RTC pushStream，进入推流状态
   └─ 未授权 → 弹系统权限 Dialog
       ├─ 同意 → 同上
       └─ 拒绝 → 自动 WS MicLeave（自己下麦）+ Toast "未授权无法推流"
```

## D. 被抱下麦（ForceLeaveMic 到达自己）

```
收到 WS MicLeft { user_id: self, forced_by }
   ↓
1. Toast "你已被 {forced_by.nickname} 抱下麦位"
2. RTC stopStream，UI 回到观众状态
```

## 通用组件 / 扩展点

- `ChatInputField(muteUntil)` — 聊天输入框封装，自动读 Mute 状态
- `MicSlotButton(selfMuted, ...)` — 麦位"+"按钮，统一处理 disabled
- `ForceMicHandler` — 监听 MicTaken/MicLeft 事件的 Singleton，自动处理权限与推流
- RoomViewModel.selfMuteState — 全房仅 1 份，所有 UI 订阅

## TDD 验收

1. 禁麦 5min 内点"+"按钮 0 网络请求
2. 禁麦用户原在麦上 → MicLeft 到达后本地 UI 同步下麦
3. 禁言用户输入框 disabled
4. 被抱上麦已授权麦克风 → 1s 内本地麦位显示自己
5. 被抱上麦未授权 → 权限弹窗；拒绝后自动 MicLeave
6. 被抱下麦 → RTC stop 被调用一次
7. 禁麦/禁言倒计时结束 UI 自动恢复（订阅 selfMuteState）
8. 收到解除广播 `UserMuted { duration_sec: 0 }` → 本地状态立即清空
9. 自己不是被操作对象的 ForceTakeMic 事件不触发权限申请
