# TDS: 双服务共库 Migration 表隔离 (Task ID: T-0000M)

## 一、背景与目标

- **关联需求**：模块 9 E2E 测试基建后置缺陷修复（M1 本地全栈一键起服务跑通）。
- **本 Task 的目标**：消除 AppServer 与 AdminServer 共用同一 PostgreSQL 库 `voiceroom` 时，对默认 `_sqlx_migrations` 登记表互相覆盖、互相校验失败的架构级阻断。

### 1.1 实测阻断现象（来自 2026-04-27 e2e:up 联调）

冷启动 `docker compose down -v && npm run e2e:up` 必现：

| 服务 | 现象 | 根因 |
|------|------|------|
| AppServer | `permission denied for table _sqlx_migrations`（即便补 `GRANT CREATE ON SCHEMA`） | AdminServer 已抢先创建该表并归属 `admin_server_user`，AppServer 用 `app_server_user` 写入 |
| AdminServer | `migration 5 was previously applied but is missing in the resolved migrations` | AppServer 已写入 `version=5..9`，AdminServer `migrations/` 只有 1..4，sqlx 视为「迁移消失」拒绝启动 |

**核心结论**：只要双服务对同一库执行 `sqlx::migrate!("./migrations")`，默认共享 `_sqlx_migrations` 表 → 版本号 / checksum 必然互掐，docker `down -v` 全清也无解。

### 1.2 为何不选「双库分离（方案 A）」

经对 AdminServer 源码盘点，`app/adminServer/src/modules/**` 共 13 处直接 `SELECT ... FROM users / rooms / events / wallets`：

- [`modules/user/repository.rs`](../../../app/adminServer/src/modules/user/repository.rs) 3 处
- [`modules/room/repository.rs`](../../../app/adminServer/src/modules/room/repository.rs) 3 处
- [`modules/wallet/repository.rs`](../../../app/adminServer/src/modules/wallet/repository.rs) 2 处
- [`modules/stats/repository.rs`](../../../app/adminServer/src/modules/stats/repository.rs) 2 处
- [`modules/event/query_repo.rs`](../../../app/adminServer/src/modules/event/query_repo.rs) 3 处

「物理双库」会要求把上述 13 处全部改写为 RPC 调 AppServer 或启用 `postgres_fdw`，对一个 MVP 阶段的「模块化单体 + 双进程」是过度工程，且偏离当前 [`doc/architecture/directory_and_ddd.md`](../../architecture/directory_and_ddd.md) 中「共享 DB、按 schema/table 权限隔离」的既定架构方向。**故否决方案 A**。

### 1.3 为何不直接上「方案 C（启动期跳 migrate + sqlx-cli）」

方案 C 的「跳过 runtime migrate、改由 CI/脚本 `sqlx migrate run` 一次性执行」必须先解决「同一 `_sqlx_migrations` 表里两份历史并存」的语义问题——其唯一手段仍然是 `--migrations-table` 自定义表名。换言之 **C 是 B 的超集**，B 是必经路径。本 Task 先收敛 B，把 C 作为后续上线 staging/prod 时的演进项（参见第六节演进路线）。

### 1.4 决策

采纳 **方案 B：自定义 sqlx migration 表名**。

| 服务 | 自定义迁移表名 |
|------|---------------|
| AppServer (`voice-room-server`) | `_sqlx_app_migrations` |
| AdminServer (`voice-room-admin-server`) | `_sqlx_admin_migrations` |

## 二、方案设计

### 2.1 核心数据流

```
┌─────────────────────────────────────────────────────────────────┐
│ Postgres voiceroom (single DB)                                  │
│                                                                 │
│  ┌───────────────────────┐   ┌──────────────────────────┐       │
│  │ _sqlx_app_migrations  │   │ _sqlx_admin_migrations   │       │
│  │  v=001..009           │   │  v=001..004              │       │
│  │  owner=app_server_user│   │  owner=admin_server_user │       │
│  └───────────▲───────────┘   └─────────────▲────────────┘       │
│              │                              │                   │
│  ┌───────────┴────────────┐    ┌────────────┴────────────┐      │
│  │ AppServer  main.rs     │    │ AdminServer main.rs     │      │
│  │  Migrator               │    │  Migrator              │      │
│  │   .set_table_name(...)  │    │   .set_table_name(...) │      │
│  │   .run(pool)            │    │   .run(pool)           │      │
│  └────────────────────────┘    └─────────────────────────┘      │
│                                                                 │
│  共享业务表（users / rooms / wallets / events / ...）          │
└─────────────────────────────────────────────────────────────────┘
```

启动期两个进程独立 migrate，互不感知对方版本；业务表保持共享与原有读写权限。

### 2.2 sqlx API 选型与降级路径

`sqlx::migrate!("./migrations")` 宏返回 `sqlx::migrate::Migrator`（编译期嵌入迁移文件）。`Migrator` 提供 `set_table_name`（≥ sqlx 0.7）。但默认情况下宏返回的是 `&'static Migrator`，**不可变借用**——直接 `.set_table_name(...)` 在 sqlx 0.8 会编译失败。

**TDD 实施时必须按以下顺序验证**：

1. **首选**：检查 `sqlx::migrate!` 是否提供 mut 入口（如 `let mut m = sqlx::migrate!(...).clone()` 或 0.8 新增的 `MigrationSource` API），优先使用宏 + 克隆。
2. **降级**：若克隆方案不可行，则改用 `Migrator::new(std::path::Path::new("./migrations")).await?` 运行时加载（牺牲编译期校验）。
3. **保底**：仍不可行时，绕过 Migrator，直接执行 SQL：先 `CREATE TABLE IF NOT EXISTS _sqlx_app_migrations (...)` + 自管 hash 比对（不推荐，仅作兜底）。

> Plan 阶段已通过快速代码检索确认 sqlx 0.8.x 的 `Migrator` 字段 `pub table_name: Cow<'static, str>` 在 `migrate!` 宏返回值上确实是 const 静态。**TDD 阶段须先做 1 行 spike**：写 `let m: sqlx::migrate::Migrator = sqlx::migrate!("./migrations"); /* 是否可 .clone() 后 set_table_name */`，红→定方案；20 分钟内未得绿则切到降级方案 2。

| AppServer | `_sqlx_app_migrations` | `app_server_user` |

### 2.5 安全与权限红线

- **必须**通过 init-db.sh（仅 dev / docker-compose 起的本地 PG）授予 `app_server_user` 在 `public` schema 上的 `CREATE` 权限，否则它无法建立自有的 `_sqlx_app_migrations` 表。
- **不得**让 `app_server_user` 写入 `_sqlx_admin_migrations`（默认无权限即可，无需显式 REVOKE）。
- staging / prod 环境本 Task 不直接覆盖；切 C 方案后由 sqlx-cli 一次性执行迁移，运行账号无需 schema CREATE 权限。

- **关键决策与坑点**：
  - **sqlx Migrator 改写采用的具体方案**：**保底方案（自管登记表 SQL）**。
    - 放弃首选「`migrate!().clone() + set_table_name`」与降级「`Migrator::new(Path)`」的事实依据：
      - sqlx 0.8.6 的 `Migrator` 结构体仅有 `migrations / ignore_missing / locking / no_tx` 四个字段，**无 `table_name` 字段**；`impl Migrator` 中**无 `set_table_name` / `with_table_name` 方法**。
      - `Migrator::new(Path)` 返回同构体，`<PgConnection as Migrate>` 的 `ensure_migrations_table / list_applied_migrations / apply / revert` 全部硬编码 SQL 字面量 `_sqlx_migrations`（见 `~/.cargo/registry/src/.../sqlx-postgres-0.8.6/src/migrate.rs` L119/139/154/236/287/310）。
      - sqlx 0.9 仍处 alpha.1，本期不引入。
    - 实际实现：`voice_room_shared::migrate::run_migrations_with_table(pool, migrator, table_name)` 复用 `Migrator::iter()` 暴露的 `Migration { version, description, checksum, sql, no_tx }`，自管 `CREATE TABLE / SELECT / INSERT` SQL；表名经 `validate_table_name` 严格白名单（`^[A-Za-z_][A-Za-z0-9_]{0,62}$`），杜绝拼接注入；advisory lock 用 FNV-1a(table_name) 派生 `lock_id`，避免与 sqlx 默认按 database name 派生的锁冲突。
  - **既有库残留 `_sqlx_migrations` 的处理**：保留不动（N-1 集成测试已证不污染新表）；如需清理，仅 dev 库可手动 `DROP TABLE _sqlx_migrations`。
  - **E-1 / E-2 受限说明**：`docker compose down -v && npm run e2e:up` 实测下，AppServer / AdminServer 进程**全部干净启动并完成迁移**（U-1 实测：`_sqlx_app_migrations`=9、`_sqlx_admin_migrations`=4，确认 T-0000M 解决方案生效），但 `wait-on http-get://127.0.0.1:3000/health` 与 `npm run preflight` 因两端**只暴露 `/ping`、未暴露 `/health`** 而失败。该 `/health` 端点缺失是 T-0000H 起就存在的预存在缺陷，**不属于 T-0000M 工作范围**，已记入「残余风险与后续 TODO」。
  - **N-2 实测**：手动 `REVOKE CREATE ON SCHEMA public` 后启动 AppServer，错误消息为 ``sqlx migrate (table=`_sqlx_app_migrations`): error returned from database: permission denied for schema ...``，含表名 `_sqlx_app_migrations`，符合排障可识别性要求。
  - **N-3 实测**：构造一个仅含 v1..3 的 `Migrator`（模拟删除 v4..9 的迁移文件）后启动，错误为 `MigrateTableError::VersionMissing { table_name: "_sqlx_app_migrations", version: 4 }`，消息含 `missing in the resolved migrations` 与表名。

#### 亮点

- **SQL 注入面闭合**：`validate_table_name` 白名单 `^[A-Za-z_][A-Za-z0-9_]{0,62}$` 严格按 PG 标识符上限，且唯一拼接表名的 3 处（`CREATE TABLE` / `SELECT … FROM` / `INSERT INTO …`）均共享同一个已校验的 `table_name` 形参，无遗漏。`validate_rejects_special_chars` 用例覆盖 `; " ' . - 空格` 6 类典型注入字符串。
- **advisory lock 设计正确**：FNV-1a 64-bit 的 offset basis（`0xcbf29ce484222325`）与 prime（`0x100000001b3`）常量正确；不同表名派生不同 `lock_id`，与 sqlx 默认按 database name 派生的 lock 完全错开；`pg_advisory_unlock` 用 `let _ =` 包裹，错误路径与 panic 路径下连接归还 pool 时 PG 会自动释放 session 级锁，双保险。
- **checksum / 缺失迁移检测对齐 sqlx 语义**：`run_inner` step 4 同时检查「applied 集合 ⊄ known 集合」（→ `VersionMissing`）与「同 version 的 checksum 字节相等」（→ `VersionChecksumMismatch`），错误消息均含表名 + 具体 version，可观测性满足 N-3 排障需求。
- **事务边界清晰**：每条迁移单独 `conn.begin()`，DDL 与登记表 INSERT 同事务提交；中途失败回滚不留半应用状态。
- **测试隔离严谨**：`migration_isolation_test` 每例 `CREATE SCHEMA t0m_<uuid16>` + `PgConnectOptions::options([("search_path", schema)])`，避免与产线表撞名，也避免 5 条用例之间互污染。U-1/U-2/U-3/N-1/N-3 五条用例与 TDS §3 验收点一一对应；`wallet_schema_test`（9 处）、`send_gift_test`（13 处）替换数与 commit 实际改动吻合，源代码 `grep "MIGRATOR\.run" / "sqlx::migrate!.*\.run"` 在两文件内 0 命中。
- **脚本侧收口干净**：`grep "GRANT CREATE" scripts/dev/e2e-up.sh` 0 命中，inline workaround 已撤；`init-db.sh` 的 `GRANT CREATE ON SCHEMA public TO app_server_user` 是普通 `GRANT`（PG 重复授权天然幂等）；`e2e-up.sh` 用子进程作用域 `DATABASE_URL="$ADMIN_DATABASE_URL" cargo run -p voice-room-admin-server` 覆盖，不污染父 shell。

#### 建议（不阻塞通过，留待后续迭代）

- **[建议] helper 未透传 `Migration::no_tx` 字段**：`run_inner` 对每条迁移无差别 `conn.begin()`。当前 13 条迁移文件均不带 `-- no-transaction` 前置注释，所以行为正确；但一旦未来引入 `CREATE INDEX CONCURRENTLY` 等不能在事务内执行的 DDL，本 helper 会与 sqlx 原生行为分叉并直接报 `25001`。建议在下一次小迭代加一条分支：`if m.no_tx { sqlx::raw_sql(m.sql).execute(&mut *conn).await?; INSERT 也直接走 conn; } else { tx 路径 }`，并补一条单元用例。
- **[建议] `migration_isolation_test.rs` 顶部注释关于 `app_server_user` 可跑此测试的描述不准确**：`GRANT CREATE ON SCHEMA public` 仅允许在 `public` schema 内建表，**不**赋予 `CREATE SCHEMA` 权限（后者需要 `GRANT CREATE ON DATABASE voiceroom`）。当 `DATABASE_URL` 指向 `app_server_user` 时，`create_isolated_schema` 的 `CREATE SCHEMA` 会因权限不足直接 panic。实际可跑场景仅限 superuser DATABASE_URL（CI 与本地 postgres 账号）。建议把注释改为「仅 superuser DATABASE_URL 可跑」，或在 init-db.sh 追加 `GRANT CREATE ON DATABASE voiceroom TO app_server_user`（dev-only）。
- **[建议] 其余 server 集成测试仍直连 `sqlx::migrate!().run(&pool)`**：`gift_list_test.rs`、`governance_real_repos_test.rs`、`wallet_api_test.rs`、`ranking_test.rs`、`events_batch_test.rs` 仍写默认 `_sqlx_migrations` 表。本 Task TDS §2.3 范围只覆盖 `wallet_schema_test` / `send_gift_test`，故不阻塞通过；但全仓库 grep 仍有约 12 处遗留调用，会让同一 dev 库里同时存在 `_sqlx_migrations`（旧）+ `_sqlx_app_migrations`（新）两张表，长期维护语义不一致。建议作为 T-0000M 的 follow-up 小任务统一收敛。
- **[建议] 测试 schema 残留**：`drop_schema` 仅在断言全部通过后调用，用例 panic 时 `t0m_<uuid>` schema 会残留。建议用 `scopeguard::defer!` 或在 `setup_pool` 上挂 RAII guard，避免反复跑红测试后 PG 中累积孤儿 schema。
- **[备注（非问题）] advisory lock 在并发 cargo test 下会串行化迁移阶段**：所有共用 `_sqlx_app_migrations` 表名的测试 lock_id 相同，并发时按锁排队。这是 advisory lock 应有的语义（对外是「同库 migrate 互斥」），不是缺陷；仅需关注后续测试运行时长是否拉长。
