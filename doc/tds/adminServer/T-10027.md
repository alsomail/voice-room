# TDS: SKU CRUD API (Task ID: T-10027)

## 一、背景与目标

- **关联需求**：[phase1_payment_billing.md §2.1 In Scope — SKU 配置](../../product/phase1_payment_billing.md)；[payment_api.md §9.7 Admin REST](../../protocol/payment_api.md#97-admin-restadmin-server)；[payment_api.md §9.2.1 payment_skus 表](../../protocol/payment_api.md#921-payment_skus-表)
- **本 Task 的目标**：为运营人员提供 SKU 全生命周期管理（列表/创建/更新/软删），含价格钻石数变更的二次确认机制与全操作审计，确保 SKU 与 Google Play Console productId 对齐并在软删后不影响历史订单入账。

## 二、方案设计

### 核心数据流

```
── GET /api/v1/admin/payments/skus（列表）─────────────────────────

Web Admin (T-20032)
  │
  ▼
[JWT + PaymentWrite or PaymentRead]  ← GET 仅需 PaymentRead
  ▼
list_skus_handler()
  SQL: SELECT * FROM payment_skus ORDER BY sort_order ASC, created_at ASC
  （包含 is_active=false 的下架 SKU，区别于 App 端只返回 is_active=true）
  ▼
Response 200 { code:0, data:{ skus:[…] } }

── POST /api/v1/admin/payments/skus（创建）──────────────────────────

Body: {
  sku_id: "diamond_9999",       // 必须与 Google Play Console productId 一致
  provider: "google_play",
  diamonds: 9999,
  display_price_usd: "149.99",
  display_price_local: null,
  display_currency: null,
  sort_order: 60,
  tag: "super_value",
  is_active: true
}
  ▼
[JWT + PaymentWrite]
  ▼
create_sku_handler(Json<CreateSkuRequest>)
  ├─ display_price_usd > 0 校验 → 否则 400
  ├─ diamonds > 0 校验 → 否则 400
  ├─ sku_id 格式检查：若不符合 "diamond_NNN" 模式
  │     → 响应体携带 warning: "sku_id 格式可能与 Google Play Console productId 不一致，请人工确认"
  │     （不阻断写入，仅 warning 字段提示）
  ├─ INSERT payment_skus (…)
  │     UNIQUE 约束冲突 (sku_id) → 409 (SKU_ALREADY_EXISTS)
  ├─ INSERT admin_logs (action='sku_create', detail={sku_id, diamonds, display_price_usd})
  ▼
Response 201 { code:0, data:{ sku }, warning?: "…" }

── PUT /api/v1/admin/payments/skus/:sku_id（更新）───────────────────

Query: ?confirm=true   ← 价格/钻石变更时必须携带，否则返回 422
Body: {
  diamonds?: 1200,
  display_price_usd?: "19.99",
  is_active?: false,
  sort_order?: 50,
  tag?: "hot"
}
  ▼
[JWT + PaymentWrite]
  ▼
update_sku_handler(Path<String>, Query<UpdateSkuQuery>, Json<UpdateSkuRequest>)
  ├─ SELECT current FROM payment_skus WHERE sku_id = :sku_id
  │     不存在 → 404
  ├─ 检测 diamonds / display_price_usd 是否变更
  │     若变更 AND confirm != true → 422 {code:42201, message:"PRICE_CHANGE_REQUIRES_CONFIRM"}
  ├─ display_price_usd > 0 / diamonds > 0（若提供）校验
  ├─ UPDATE payment_skus SET (变更字段), updated_at = now()
  ├─ INSERT admin_logs (action='sku_update', detail={sku_id, diff:{before, after}})
  ▼
Response 200 { code:0, data:{ sku } }

── DELETE /api/v1/admin/payments/skus/:sku_id（软删）───────────────

  ▼
[JWT + PaymentWrite（super_admin）]
  ▼
delete_sku_handler(Path<String>)
  ├─ SELECT FROM payment_skus WHERE sku_id = :sku_id
  │     不存在 → 404
  ├─ UPDATE payment_skus SET is_active = false, updated_at = now()
  │     （不物理删除；payment_orders.sku_id FK 仍有效；历史订单 verify 链路不受影响）
  ├─ INSERT admin_logs (action='sku_delete', detail={sku_id})
  ▼
Response 200 { code:0, data:{ sku_id, is_active: false } }
```

> **软删说明**：`payment_orders.sku_id` 是 FK 指向 `payment_skus.sku_id`，软删只置 `is_active=false`，不 DELETE 行，历史订单验证链路（T-00052 verify_service）查询 `payment_skus` 时不过滤 `is_active`，因此已下单的 SKU 入账不受影响。

### 接口 / 信令定义

路径字面量与表字段详见 [payment_api.md §9.2.1](../../protocol/payment_api.md#921-payment_skus-表) 与 [payment_api.md §9.7](../../protocol/payment_api.md#97-admin-restadmin-server)。

**二次确认机制**（价格/钻石变更）：

- PUT 请求若 `diamonds` 或 `display_price_usd` 有变更，且未携带 `?confirm=true` → 422 + `code: 42201 PRICE_CHANGE_REQUIRES_CONFIRM`，响应体含 `before/after` diff 供前端展示确认弹窗。
- 携带 `?confirm=true` 时正常执行，无需二次请求。

**sku_id 格式 warning（非阻断）**：

- 检查规则：不符合 `^[a-z][a-z0-9_]{2,63}$` 的 sku_id → 响应 `warning` 字段提示与 Google Play Console 一致性。
- 不阻断写入（正式校验由运营在 Google Play Console 对照）。

## 三、TDD 验收用例

### 正向用例
- [ ] **SK-01**：GET 列表返回全部 SKU（含 is_active=false），字段含 `diamonds`、`display_price_usd`、`is_active`
- [ ] **SK-02**：POST 创建 SKU，`diamonds=600, display_price_usd=9.99` → 201，DB 可查
- [ ] **SK-03**：POST 创建时 sku_id 不符合 `diamond_NNN` 格式 → 201（成功）但响应含 `warning` 字段
- [ ] **SK-04**：PUT 仅更新 `sort_order`（不变价格/钻石）→ 无需 confirm，200 直接成功
- [ ] **SK-05**：PUT 变更 `display_price_usd` 且携带 `?confirm=true` → 200，DB 已更新，`admin_logs.detail` 含 before/after diff
- [ ] **SK-06**：DELETE 软删 → `is_active=false`，行仍存在，历史 order 的 sku_id FK 未断
- [ ] **SK-07**：软删后对应 sku_id 的历史 `payment_orders` 记录 `sku_id` FK 仍正常关联（JOIN 可查）
- [ ] **SK-08**：所有写操作（POST/PUT/DELETE）均写 `admin_logs`（action=`sku_create`/`sku_update`/`sku_delete`）

### 异常用例
- [ ] **SK-09**：`display_price_usd <= 0` → 400 (40003)
- [ ] **SK-10**：`diamonds <= 0` → 400 (40003)
- [ ] **SK-11**：PUT 变更价格但未携带 `?confirm=true` → 422 (42201)，响应含 before/after diff
- [ ] **SK-12**：POST 重复 sku_id → 409
- [ ] **SK-13**：role=`cs` 或 `finance` 调用 POST → 403 (40301)
- [ ] **SK-14**：DELETE 不存在的 sku_id → 404
