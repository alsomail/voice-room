# TDS: 麦克风权限请求 (Task ID: T-30012)

## 一、背景与目标

- **关联需求**：房间核心体验——麦位上麦交互流程（Mic 模块）
- **前置依赖**：T-30011（MicSlotCard 三态麦位组件，已完成）。MicSlotsGrid 已有 `onMicSlotClick: (index: Int) -> Unit = {}` 回调，T-30012 在此回调上挂载权限守卫。
- **本 Task 目标**：当用户点击空麦位时，通过 Accompanist Permissions 发起 `RECORD_AUDIO` 运行时权限请求；根据授权状态分别展示理由对话框（首次拒绝）或跳转系统设置对话框（永久拒绝），权限授予后透传上麦意图至上层。

---

## 二、方案设计

### §2.1 核心数据流

```
用户点击空麦位
    ↓  onMicSlotClick(index)
MicPermissionHandler（Composable 守卫）
    ├─ GRANTED            → onPermissionGranted(index) → RoomViewModel.onMicPermissionGranted(index)
    ├─ DENIED (rationale) → 显示 MicPermissionRationaleDialog
    │       ├─ 确认       → permissionState.launchPermissionRequest()
    │       └─ 取消       → dismiss，丢弃 pendingSlotIndex
    └─ DENIED (permanent) → 显示 MicPermissionSettingsDialog
            ├─ 去设置     → startActivity(ACTION_APPLICATION_DETAILS_SETTINGS)
            └─ 取消       → dismiss，丢弃 pendingSlotIndex
```

### §2.2 MicPermissionHandler 组件签名

```kotlin
@Composable
fun MicPermissionHandler(
    onPermissionGranted: (slotIndex: Int) -> Unit,
    content: @Composable (onMicSlotClick: (Int) -> Unit) -> Unit,
)
```

内部状态：
- `permissionState`: `PermissionState` (accompanist)
- `pendingSlotIndex`: `MutableState<Int?>`
- `showRationaleDialog`: `MutableState<Boolean>`
- `showSettingsDialog`: `MutableState<Boolean>`

## 五、Reviewer意见

> 状态枚举：🟢 通过 | 🔴 未通过

### Round 1: ❌ 未通过

- **状态**：🔴 未通过
- **审查日期**：2025-01
- **审查意见**：

#### [HIGH] `else` 分支首次权限请求时错误显示 Settings Dialog
**File**: `MicPermissionHandler.kt:102-107`  
**Issue**: `onMicSlotClick` 的 `else` 分支注释写"永久拒绝"，但 Android 在**首次请求**时 `shouldShowRationale` 同样为 `false`（权限从未被请求过）。导致用户首次点击麦位就看到"麦克风权限已被永久拒绝，请前往系统设置手动开启"——系统权限弹窗从未弹出，核心功能完全失效。  
**Fix**: `else` 分支应调用 `helper.launchPermissionRequest()`（系统弹窗会处理首次请求；永久拒绝时系统静默忽略）。永久拒绝的判断需在请求返回后结合 `!isGranted && !shouldShowRationale` 二次判断，可通过额外的 `permissionRequested: Boolean` 状态标记区分两种场景。正确结构示意：
```kotlin
// else 分支（首次 OR 永久拒绝均走此路）
else -> {
    pendingSlotIndex = slotIndex
    helper.launchPermissionRequest()          // 系统处理首次请求
    // 返回后：若仍未授予且 !shouldShowRationale → 永久拒绝 → 再显示 Settings Dialog
}
```

#### [MEDIUM] `LaunchedEffect(helper.isGranted)` 触发路径缺少测试覆盖
**File**: `MicPermissionHandlerTest.kt`  
**Issue**: `LaunchedEffect(helper.isGranted)` 是"系统权限弹窗返回后自动透传回调"的核心路径，但 MP-01~MP-07 的 fake helper 均使用常量 `override val isGranted = true/false`，无法模拟 `denied→granted` 状态转变。该路径（用户在系统弹窗中点"允许"→ `isGranted` 从 false 变 true → `onPermissionGranted(idx)` 被调用）完全没有测试覆盖。  
**Fix**: 新增 MP-01B：使用可变 fake helper（`isGranted` 初始为 false），触发 click 设置 `pendingSlotIndex`，然后更新 helper 状态为 `isGranted=true` 并触发重组，验证 `onPermissionGranted` 被正确调用。

#### [MEDIUM] MP-08 断言精度不足，未满足 TDS 验收要求
**File**: `RoomViewModelTest.kt:299`  
**Issue**: TDS 要求验证 ShowToast 消息包含 **"座位 3"**，但测试只断言 `.contains("3")`。消息 "准备上麦：座位 13"（如 slotIndex=12）也能通过该断言，测试未精确覆盖 TDS 规格。  
**Fix**: 改为 `it.message.contains("座位 3")`。

#### [LOW] `LocalContext.current` 在条件块内读取
**File**: `MicPermissionHandler.kt:126`  
**Issue**: `val context = LocalContext.current` 在 `if (showSettingsDialog)` 条件块内读取，不符合 Compose 惯例（CompositionLocal 建议在 composable 顶层读取，避免潜在的重组时序问题）。  
**Fix**: 将 `val context = LocalContext.current` 提升到 `MicPermissionHandler` 函数体顶部。

- **改进建议**：
  1. **必须修复（阻塞）**：修正 `else` 分支逻辑，确保首次权限请求调用 `launchPermissionRequest()` 而非直接显示 Settings Dialog。
  2. **需要补充**：补充 `LaunchedEffect` 状态转变路径的 UI 测试用例（MP-01B 或类似）。
  3. **需要修复**：MP-08 断言改为精确匹配 "座位 3"。
  4. **建议优化**：`LocalContext.current` 提升到函数顶部。
