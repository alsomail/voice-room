# TDS: Android WS 客户端 sealed class 化 + RoomViewModel 27+ 野生反序列化全量迁移 (Task ID: T-00101)

## 一、背景与目标

- **关联需求**：BUG-CHAT-WS 复盘 R2 → Android `RoomViewModel.handleWsMessage`（920-1080 行）使用 `json.get("slotIndex").asInt` 顶层 camelCase 野生取字段，与 server 嵌套 snake_case 完全失配，且 `?: return` 静默吞错；表面"在跑"，实则关键 UI 信令（MicTaken/MicLeft/UserJoined/UserLeft/UserMuted/AdminChanged/RoomInfoUpdated）在生产路径上**从未真正生效**，依赖 HTTP `JoinRoomResult` 快照初始化掩盖了问题。
- **本 Task 的目标**：基于 T-00100 输出的 schemas/ws/*.schema.json，在 Android 侧建立**类型安全的 WS 反序列化层**：
  1. 生成 `WsServerMessage` / `WsClientMessage` Kotlin sealed class（手写或 codegen，二选一，先用手写在本 Task 收口；codegen 留 T-00106 自动化）。
  2. 替换 `RoomViewModel.handleWsMessage` 27+ `json.get(...)` 野生分支为 `when (msg) { is WsServerMessage.MicTaken -> ... }`。
  3. 同步 `OkHttpWebSocketClient` 心跳 ping/pong → Ping/Pong（与 T-00108 协调；本 Task 仅在 Android 侧落地）。
  4. 接入 Gson 严格模式：未知顶层信令打 ERROR + 上报埋点（不再静默 `?: return`）。

## 二、方案设计

### 核心数据流

```
WS Frame (JSON, snake_case + payload 嵌套)
      │
      ▼
OkHttpWebSocketClient.onMessage  ──→  Gson.fromJson(json, WsServerMessage::class.java)
                                                │
                                                ▼  (sealed class with @JsonAdapter or RuntimeTypeAdapterFactory)
                                       WsServerMessage.MicTaken {
                                           type: "MicTaken",
                                           payload: MicTakenPayload(mic_index, user_id, forced_by?),
                                           timestamp: Long
                                       }
                                                │
                                                ▼
                          RoomViewModel.handleWsMessage(msg: WsServerMessage)
                                                │
                                                ▼
                              when (msg) { is MicTaken -> updateMicSlot(msg.payload) ... }
```

## 五、Reviewer意见

### Round 1
- **状态**：❌ REQUEST_CHANGES
- **审查意见**：发现 3 个 P0 问题：(1) UserMuted 字段结构使用旧版平铺格式而非 payload-nested（与 UserMuted.schema.json 不符）；(2) handleWsMessage 内 MicTaken/RoomInfoUpdated 分支各有 1 处 bare `?: return` 无日志；(3) when(msg) 全部处理分支缺少 PROTO-BINDING 注释。
- **改进建议**：UserMuted 迁移为 payload-nested；`?: return` 改为带 Log.w 的显式处理；补全 PROTO-BINDING 注释。
- **🔴 Changelog 行**：Round1-P0-3项（UserMuted结构/bare return/PROTO-BINDING注释）退回修复。

### Round 2
- **状态**：❌ REQUEST_CHANGES
- **审查意见**：P0-1/P0-2/P0-3 修复均通过验证。但迁移 UserMuted 时新引入 1 处 bare `?: return`（`targetUserId ?: return`，无 Log.w）。另外 WsClientMessage.kt 中 @SerializedName 注解因绕过 Gson 序列化路径实际无效（低影响，建议标注）。
- **改进建议**：UserMuted 分支 targetUserId null check 补 Log.w。
- **🔴 Changelog 行**：Round2-HIGH（UserMuted bare return）退回补日志。

### Round 3
- **状态**：✅ APPROVE
- **审查意见**：T-00101 Android WS sealed class 经三轮审查后达到合并标准：Round 1 消除了 handleWsMessage 函数内多处裸吞错的 `?: return`，将 RoomState 替换为类型安全的 sealed class 并补全了日志可观测性节点；Round 2 针对遗漏的 UserMuted 分支进行了同等修复，为 targetUserId 和 muteType 两个可空字段各补充了带上下文信息的 Log.w + 显式 return，彻底消除了静默忽略的隐患；最终代码在安全性、可观测性（全链路 D/W/E 日志覆盖）、类型安全（sealed class dispatch + proto-binding 注释）三个维度均符合项目规范。
- **改进建议**：无（可选：后续 T 将 RoomViewModel 1410行拆分为 WsMessageHandler 独立类）。
