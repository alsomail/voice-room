## 一、背景与目标
- **关联需求**：[phase1_payment_billing.md §6.3 / §0 权威源](../../product/phase1_payment_billing.md) — E-08 营收闭环核心。所有钻石入账必须经 Google `purchases.products.get` 真签验证 + obfuscatedAccountId 双向锁定 + 强事务推进 PENDING→VERIFIED→CREDITED→ACKED。
- **本 Task 的目标**：实现 `POST /api/v1/payments/google/verify`，按 [Google Play Billing 8.3.0 官方文档](https://developer.android.com/google/play/billing/integrate) 的标准流程完成验签 + 入账 + acknowledgePurchase 三步原子推进。

## 二、方案设计

### 核心数据流
```
Android (T-30063) onPurchasesUpdated → POST /payments/google/verify { order_id, purchase_token }
        ↓
PaymentVerifyService::verify_and_credit (一次 SQLx Tx)
  ├─ SELECT FOR UPDATE payment_orders WHERE order_id=? AND state IN (PENDING, VERIFYING)
  │     └─ 状态非法 → 40905；purchase_token 已被其他订单认领 → 幂等返回 ACKED 结果
  ├─ UPDATE state=VERIFYING
  ├─ COMMIT (放 Google 调用前避免长事务)
  ↓
GooglePlayBillingPort::get_product_purchase(package_name, sku_id, purchase_token)  [防腐层]
  ├─ purchaseState != 0 → 40901
  ├─ obfuscatedAccountId != order_id → 40901 + Sentry
  └─ 成功返回 ProductPurchase
        ↓
新事务 (强事务) 推进 VERIFIED → CREDITED：
  ├─ UPDATE payment_orders SET state=CREDITED, purchase_token=?, verified_at=now, credited_at=now
  ├─ UPDATE users SET diamond_balance = diamond_balance + sku.diamonds
  ├─ INSERT wallet_transactions (type='recharge', source='google_play', payment_order_id=?)
  └─ COMMIT
        ↓
GooglePlayBillingPort::acknowledge(purchase_token)
        ↓
UPDATE state=ACKED, acked_at=now
        ↓
WS 推送 BalanceUpdated { reason='payment_credit', balance_after, delta }（[websocket_signals.md §6.13.1](../../protocol/websocket_signals.md)）
```

## 三、TDD 验收用例
- [ ] **沙箱跑通**：5 档 SKU（60/300/600/1980/6480）每档跑一次完整链路成功
- [ ] **重复 purchase_token**：第二次相同 token 调 verify → 200 幂等，返回最终 ACKED 状态；DB 仅一条 wallet_transactions
- [ ] **purchaseState=0 之外**（如 PENDING/CANCELED）→ 422 + code=40901，订单 state=FAILED，余额不变
- [ ] **obfuscatedAccountId 不匹配**：mock Google 返回 obfuscatedAccountId="fake" → 422 + code=40901 + Sentry breadcrumb 含 `mismatch_obfuscated_account_id`
- [ ] **强事务断电**：在 CREDITED 之后 acknowledge 之前进程 kill；重启后 cron（T-00054）能继续推进至 ACKED；balance 不被重复 +
- [ ] **同 token 并发 10 次**：仅一个事务成功 credit，其余幂等返回，余额仅 +1 次
- [ ] **state 非法转换**：order 已 ACKED 时再 verify → 200 幂等，不写流水
- [ ] **未认证**：401
- [ ] **WS 推送**：CREDITED 后 1s 内同 user_id 的 WsConnection 收到 `BalanceUpdated.reason='payment_credit'` envelope
