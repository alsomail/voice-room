## 五、Reviewer意见
### Round 1
- **状态**：❌ 未通过
- **审查意见**：
  - **[HIGH-1] mute_type 字段名不匹配（功能 Bug）**：`GovernanceFilters.mute_type` 与 `MuteListParams.type` 字段名不一致，`GovernanceLogsPage` 使用 `filters as MuteListParams` 强转但不重命名字段，运行时实际发送 `mute_type=mic` 给服务端，服务端期望 `type=mic`，导致禁言类型筛选条件静默失效。需在传给 `MuteLogsTab` 时显式映射 `type: filters.mute_type`。
  - **[HIGH-2] 路由层缺少角色守卫**：TDS §二 明确要求「路由层 Guard：`roles ∩ [super_admin, operator, cs]` 非空才进入；否则 Redirect to 403」。当前 `AuthGuard` 仅校验 JWT 有效性，不做角色检查。finance 角色用户可绕过菜单限制，直接通过 URL `/rooms/governance` 访问页面，与规格不符。
  - **[HIGH-3] 测试 G14-05 覆盖不完整**：G14-05 仅验证了「finance 角色无菜单项」，未测试「finance 角色直接访问路由被重定向 403」。这是 HIGH-2 的测试盲区，直接导致路由守卫缺失未被自动化测试捕获。
  - **[MEDIUM-1] 非活跃 Tab 发起多余 API 请求**：Ant Design `<Tabs>` 默认不销毁非活跃 Tab 子组件，页面初始化时 `KickLogsTab` 和 `MuteLogsTab` 同时挂载，导致 `listKicks` 和 `listMutes` 在初始加载时同时被调用。建议添加 `destroyInactiveTabPane`。
  - **[MEDIUM-2] 双 useEffect 竞态**：`KickLogsTab`/`MuteLogsTab` 用两个独立 effect 分别处理 reset-page 和 fetch，`filters` 变化时会先以旧 page 触发一次无效 fetch（被 AbortController 取消），再以 page=1 触发正确请求，建议合并为单一 effect。
  - **[LOW-1] TDS §二 语言文件规格与实现不一致**：§二 涉及文件列表写 `ar/en.json`，实际实现为 `en.ts`/`zh.ts`，§二 未同步更新。
- **改进建议**：
  1. `GovernanceLogsPage.tsx`：传给 `MuteLogsTab` 时补充字段映射 `type: filters.mute_type`（修复 HIGH-1）
  2. 新增 `RoleGuard` 组件或扩展 `AuthGuard` 支持 `allowedRoles` prop，在 `/rooms/governance` 路由上校验角色（修复 HIGH-2）
  3. `GovernanceLogsPage.test.tsx` G14-05 补充直接 URL 访问被 403 重定向的测试用例（修复 HIGH-3）
  4. `GovernanceLogsPage.tsx` `<Tabs>` 添加 `destroyInactiveTabPane`（修复 MEDIUM-1）
