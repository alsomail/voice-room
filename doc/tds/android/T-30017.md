# TDS: 接收消息逻辑 (Task ID: T-30017)

## 一、背景与目标

- **关联需求**：语聊房实时聊天体验——接收并展示其他用户发送的文字消息
- **前置依赖**：
  - T-00016（App Server 文本消息广播 ✅）：服务端接收 `SendMessage` 后广播 `MessageReceived` 给房间内所有用户
  - T-30014（聊天列表 ✅）：`ChatMessageList` 已实现，`LaunchedEffect(deduplicated.size)` 驱动自动滚动
- **本 Task 目标**：
  1. 在 `RoomViewModel.handleWsMessage` 中处理 `MessageReceived` 广播消息，将其追加到 `RoomUiState.messages`
  2. 基于 `seenMsgIds`（T-30010 已有）对 `msgId` 去重，防止重复展示
  3. 内容缺失（`content` 字段为 null）的非法消息静默忽略，不崩溃
  4. 自动滚动由 T-30014 的 `LazyColumn` `LaunchedEffect(deduplicated.size)` 自动触发，ViewModel 层无需额外处理

## 二、方案设计

### §2.1 服务端广播格式（MessageReceived）

基于 `protocol.md §6` 与 `RoomViewModel.kt` 现有代码约定：

| 方向 | type | 字段 |
|------|------|------|
| Server → Client (广播) | `MessageReceived` | `msgId`（UUID）、`senderNickname`（string）、`content`（string）、`timestamp`（long） |

JSON 示例：
```json
{
  "type": "MessageReceived",
  "msgId": "uuid-abc",
  "senderNickname": "Alice",
  "content": "Hello world",
  "timestamp": 1713312000
}
```

### §2.2 handleWsMessage 中 MessageReceived 分支

```kotlin
"MessageReceived" -> {
    val msgId = json.get("msgId")?.asString ?: return
    if (seenMsgIds.contains(msgId)) return   // 去重：已见 msgId 直接忽略
    seenMsgIds.add(msgId)

    val senderNickname = json.get("senderNickname")?.asString ?: ""
    val content = json.get("content")?.asString ?: return  // content 缺失 → 静默忽略
    val timestamp = json.get("timestamp")?.asLong ?: 0L

    val newMsg = ChatMessageUi(
        messageId = msgId,
        senderNickname = senderNickname,
        content = content,
        timestamp = timestamp,
    )
    _uiState.value = RoomViewState.Success(
        state.copy(messages = state.messages + newMsg)
    )
}
```

**关键设计决策**：
- `content` 使用 `?: return`（不是 `?: ""`）— 空内容消息在业务上无意义，静默丢弃，避免空白气泡出现在聊天列表
- `senderNickname` 使用 `?: ""`  — 昵称可以为空字符串（系统消息场景）
- `seenMsgIds` 检查使用 `contains` + `add` 两步（等价于 `add` 返回 false 时 return），复用 T-30010 已有逻辑

### §2.3 RoomUiState.messages（已存在）

T-30014 已在 `RoomUiState` 中定义：
```kotlin
val messages: List<ChatMessageUi> = emptyList()
```

`ChatMessageUi` 数据类也已存在：
```kotlin
data class ChatMessageUi(
    val messageId: String,
    val senderNickname: String? = null,
    val content: String,
    val timestamp: Long,
    val messageType: MessageType = MessageType.USER_TEXT,
)
```

无需修改 `RoomUiState.kt`。

### §2.4 自动滚动机制

由 T-30014 的 `ChatMessageList` Composable 内部处理：
```kotlin
LaunchedEffect(deduplicated.size) {
    if (deduplicated.isNotEmpty()) {
        listState.animateScrollToItem(deduplicated.size - 1)
    }
}
```

当 `messages` 列表追加新消息导致 `size` 变化时，`LaunchedEffect` 自动触发滚动到最新消息，ViewModel 层无需额外操作。

### Round 6 Follow-up（BUG-CHAT-WS，2026-04-30）

**结论：🟢 修复**

- **根因**：服务端 `app/server/src/room/handler/chat.rs` 实际广播的信封为
  `{"type":"RoomMessage","payload":{"msg_id","user_id","content"},"timestamp"}`，
  但 `RoomViewModel.handleWsMessage` 只处理了协议早期版本的 `type=="MessageReceived"` 顶层字段
  形态，导致客户端收到的所有聊天广播全部被 `when` 落入默认分支丢弃，公屏不渲染。
- **修复**：在 `handleWsMessage` 中新增 `"RoomMessage"` 分支：
  - 解析 `payload.msg_id / payload.user_id / payload.content` 与顶层 `timestamp`；
  - 缺 `msg_id` 或 `content` 时静默忽略（与 RM-05 行为一致）；
  - 通过 `_audienceState`（`onMic` / `audience`）与 `state.micSlots` 反查 `senderNickname`，
    若匿名用户暂未在场景内则回退为 `user_id` 短串；
  - 复用既有 `seenMsgIds` 去重，发送方自身回流也会被正常渲染。
  保留旧 `MessageReceived` 分支以兼容历史/单测。
- **测试**：`app/android/app/src/test/java/com/voice/room/android/feature/room/RoomViewModelChatTest.kt`
  新增 4 个用例：基础追加、msg_id 去重、content 缺失忽略、自身回流渲染。本地 `:app:testLocalDebugUnitTest --tests RoomViewModelChatTest` 全部通过。
