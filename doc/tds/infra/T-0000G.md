# TDS: Seed/Reset/Preflight 三件套脚本 (Task ID: T-0000G)

> **主设计**：[T-0000E E2E 多环境分层与切换器总设计](./T-0000E.md) — 本 TDS 严格继承其 §2.5（Seed 数据契约）+ §2.6（健康预检契约）+ §2.10（涉及文件清单）+ §2.11 Step 2 验收锚点 + §2.12 R2/R7/R9 风险来源。
> **上游依赖**：[T-0000F 三档 .env 模板](./T-0000F.md) — 本 TDS 复用其 §2.3 字段表（`E2E_VALID_TOKEN` / `E2E_*_ID` 等 24 个字段）作为 Seed 输出口与 Preflight 入参口。
> **下游消费方**：[T-0000H envLoader/globalSetup/globalTeardown](./T-0000H.md) — 本 TDS §2.7 定义 globalSetup 调用本三脚本的接口契约（命令行入参、退出码、stdout 协议）。

## 一、背景与目标

- **关联需求**：模块 9（E2E 测试基建）P0 起点之一，落地 T-0000E §2.11 Migration Path **Step 2：Seed/Reset/Preflight 三件套就绪**。
- **本 Task 的目标**：交付 3 个可执行脚本，使 `tests/scripts/**` 在 `E2E_PROFILE=local` 下**冷启动 5 分钟内**可跑通：
  1. `scripts/dev/seed-e2e.sql` — 幂等创建 E2E 测试用户（A/B + 4 角色 admin）/ 测试房间 / JWT Token，并把所有 ID + Token **回填**到 `scripts/dev/.seed-output.env`，由下游 globalSetup 转出到 `process.env`。
  2. `scripts/dev/reset-e2e.sh` — 幂等清空 E2E 测试数据，**不**修改业务表结构、**不**触碰非 E2E 行。
  3. `scripts/dev/preflight.sh` — 5 端（Postgres / Redis / AppServer / AdminServer / Web）健康检查，任一挂掉 **2 秒内**彩色定位并附修复 hint，按检查项分配独立退出码。
- **修复体检缺口**：T-0000E §1 体检表 **C2**（`scripts/dev/seed-e2e.sql` 不存在）+ **C12**（无统一 5 端预检脚本）。
- **不在本 Task 范围内**：
  1. envLoader / globalSetup 实现 → T-0000H；
  2. `package.json` 的 `db:seed` `db:reset` `preflight` 一键命令 → T-0000I；
  3. AppServer / AdminServer 自身 `/health` 端点 — 假设已存在，本 Task 仅消费；
  4. Seed 生成的 JWT 需要的签名 CLI（`shared/bin/sign-jwt.rs`）— 本 Task 提供调用契约，实现可由 TDD Agent 在落地阶段一并完成（非主契约）。

## 二、方案设计

### 2.1 核心数据流

```
                       ┌──────────────────────────────────────┐
[ tests/scripts/env/ ] │ globalSetup (T-0000H)                │
   .env.<profile> ─────┤   1) loadE2EEnv()                    │
                       │   2) preflight.sh --profile $P  ◄────┼── 本 Task ③
                       │   3) [local] seed-e2e.sql       ◄────┼── 本 Task ① + sign-jwt CLI
                       │   4) source .seed-output.env    ◄────┼── 本 Task ① 回填
                       │   5) Playwright 起飞                  │
                       └──────────────────────────────────────┘
                       ┌──────────────────────────────────────┐
                       │ globalTeardown (T-0000H)             │
                       │   [local only] reset-e2e.sh     ◄────┼── 本 Task ②
                       └──────────────────────────────────────┘
```

> **不变量**：脚本三件套是 **profile-aware** 但 **profile-agnostic 实现**——逻辑层只读 env 变量（`PGHOST` / `REDIS_URL` / `APP_SERVER_BASE_URL` 等），不内置 profile 分支。Profile 路由完全由上游 envLoader 完成；本脚本对 staging/prod 的安全约束以「拒绝执行」形式存在（见 §2.4 / §2.6）。

|----------|----------|------|

### 2.4 Seed 数据契约（实现 T-0000E §2.5）

#### 2.4.1 用户表（`users`）

| 字段（确定性 ID） | phone | role | nickname | 备注 |
|---|---|---|---|---|
| `E2E_USER_A_ID = uuidv5(E2E_NS, 'user_a')` | `+966500000900` | `user` | `E2E User A` | 主测试用户，绑定 `E2E_VALID_TOKEN` |
| `E2E_USER_B_ID = uuidv5(E2E_NS, 'user_b')` | `+966500000901` | `user` | `E2E User B` | 辅助用户（连麦/送礼对象） |

`ON CONFLICT (id) DO UPDATE SET phone=EXCLUDED.phone, nickname=EXCLUDED.nickname, updated_at=now();`

#### 2.4.2 管理员表（`admins`）

| 字段（确定性 ID） | username | role | 备注 |
|---|---|---|---|
| `uuidv5(E2E_NS, 'admin_super')` | `e2e_admin` | `admin` | `E2E_ADMIN_TOKEN` |
| `uuidv5(E2E_NS, 'admin_op')` | `e2e_op` | `op` | `E2E_OP_TOKEN` |
| `uuidv5(E2E_NS, 'admin_cs')` | `e2e_cs` | `cs` | `E2E_CS_TOKEN` |
| `uuidv5(E2E_NS, 'admin_fin')` | `e2e_fin` | `fin` | `E2E_FIN_TOKEN` |

#### 2.4.3 房间表（`rooms`）

| 字段 | 值 | 备注 |
|---|---|---|
| `id` | `E2E_ROOM_ID = uuidv5(E2E_NS, 'room_main')` | 唯一测试房间 |
| `owner_id` | `E2E_USER_A_ID` | 房主=A |
| `name` | `E2E Test Room` | |
| `capacity` | `8` | 麦位 8（与生产默认一致） |
| `is_locked` | `false` | 默认开放 |
| `password` | `null` | 无密码 |
| `created_at` | `'2026-01-01 00:00:00+00'` | **固定**，避免每次跑变化 |

`ON CONFLICT (id) DO UPDATE SET owner_id=EXCLUDED.owner_id, capacity=EXCLUDED.capacity;`

#### 2.4.4 JWT Token 生成策略

| Token | 主体 | 载荷字段 | 过期 | 来源 |
|-------|------|---------|------|------|
| `E2E_VALID_TOKEN` | E2E_USER_A_ID | `{ sub, role:'user', exp }` | `now + E2E_TOKEN_TTL_SECONDS`（默认 24h） | wrapper 调 `shared/bin/sign-jwt` |
| `E2E_ADMIN_TOKEN` / `E2E_OP_TOKEN` / `E2E_CS_TOKEN` / `E2E_FIN_TOKEN` | 各对应 admin id | `{ sub, role:<对应>, exp }` | 同上 | 同上 |
| `E2E_EXPIRED_TOKEN` / `E2E_EXPIRED_ADMIN_TOKEN` | A / super-admin | `exp = now - 90d`（固定） | 已过期 | **不**经 seed，由 `.env.<profile>.example` 直接写常量值（T-0000F §2.3） |

- **密钥**：`JWT_SECRET` 从 env 注入，wrapper **不** echo 该值，**不**写日志，**不**入 `.seed-output.env`。
- **签发器**：本 Task 约束 wrapper 必须调用工作区内的 sign-jwt CLI（推荐路径：`shared/bin/sign-jwt.rs`，由 TDD Agent 落地）。CLI 入参 `--sub <uuid> --role <user|admin|op|cs|fin> --ttl <sec>`，stdout 输出 token 单行。
- **回填文件**（`scripts/dev/.seed-output.env`）格式：
  ```
  # Generated by scripts/dev/seed-e2e.sh at 2026-05-31T01:23:45Z
  E2E_USER_A_ID=...
  E2E_USER_B_ID=...
  E2E_ROOM_ID=...
  E2E_VALID_TOKEN=...
  E2E_ADMIN_TOKEN=...
  E2E_OP_TOKEN=...
  E2E_CS_TOKEN=...
  E2E_FIN_TOKEN=...
  ```
  **不含** `E2E_EXPIRED_*`（来自 .env.example，由 envLoader 直读）。

#### 2.4.5 幂等约束（量化）

- 同一台 DB 执行 `seed-e2e.sh` 第 1 次和第 N 次后：
  - `users`/`admins`/`rooms` 三张表中 `id ∈ E2E ID 集合` 的行数严格相等（=2 / =4 / =1）；
  - 所有 token 字段在第 N 次输出**均能解码并通过签名校验**（exp 可能因 TTL 重置而推迟，这是预期行为）。
- 测试方法：连跑 2 次 `seed-e2e.sh` 后，对比 `psql -c "SELECT id,phone FROM users WHERE phone LIKE '+96650000090%' ORDER BY phone;"` 输出 hash 一致。

### 2.5 Reset 清理范围表

| 目标 | 操作 | 过滤条件 | 保留项 |
|------|------|----------|--------|
| `users` 表 | `DELETE FROM users WHERE phone LIKE '+96650000090%' OR id IN (E2E_USER_A_ID, E2E_USER_B_ID);` | phone 前缀 + 显式 ID 集合 | 表结构、其他生产用户 |
| `admins` 表 | `DELETE FROM admins WHERE username IN ('e2e_admin','e2e_op','e2e_cs','e2e_fin');` | username 集合 | 真实运营管理员 |
| `rooms` 表 | `DELETE FROM rooms WHERE id = E2E_ROOM_ID;` | 显式 ID | 其他房间 |
| 关联子表（`room_members` / `gift_orders` / `wallet_tx` 等） | `DELETE WHERE room_id=E2E_ROOM_ID OR user_id IN (E2E_USER_A_ID, E2E_USER_B_ID);` | 外键集合 | 其他业务行 |
| Redis 业务键 | `redis-cli --scan --pattern 'room:{E2E_ROOM_ID}:*' \| xargs -r redis-cli DEL`；同理 `user:{E2E_USER_*_ID}:*`、`mic_muted:*`、`chat_muted:*`、`kicked:{E2E_ROOM_ID}:*` | 限定 key 前缀含 E2E ID | 其他 key |
| `scripts/dev/.seed-output.env` | `rm -f` | — | — |
| **业务表 schema** | **不动** | — | 全部 DDL |
| `schema_migrations` | **不动** | — | 全部行 |
| 非 E2E 用户/房间/订单 | **绝不删除** | — | — |

**Profile 红线**：
- `E2E_PROFILE != local` → 立即输出 `[reset-e2e] refuse: profile=$P, only 'local' is allowed` 并退出码 **21**。
- `E2E_PROFILE` 未设置 → 同样退出码 **21**（默认拒绝）。
- 受 CI 调用时支持 `--yes` 跳过交互确认；交互模式下默认要求 `y` 确认。

### 2.6 Preflight 健康检查矩阵（实现 T-0000E §2.6）

| # | 检查项 | 命令（最简形式） | 期望结果 | 超时 | 失败退出码 | 失败彩色提示文案 |
|---|--------|-----------------|---------|------|-----------|------------------|
| 1 | Postgres | `pg_isready -h "$PGHOST" -p "$PGPORT" -U "$PGUSER" -d "$PGDATABASE" -t 2` | exit 0，stdout 含 `accepting connections` | **2s** | **11** | `[1/5] Postgres   ❌ FAIL  ($PGHOST:$PGPORT)\n        ↳ Hint: docker compose up -d postgres` |
| 2 | Redis | `redis-cli -u "$REDIS_URL" -t 2 ping` | stdout `PONG` | **2s** | **12** | `[2/5] Redis      ❌ FAIL  ($REDIS_URL)\n        ↳ Hint: docker compose up -d redis` |
| 3 | AppServer | `curl -fsS --max-time 2 "$APP_SERVER_BASE_URL/health"` | HTTP 200 | **2s** | **13** | `[3/5] AppServer  ❌ FAIL  $APP_SERVER_BASE_URL/health\n        ↳ Hint: cargo run -p voice-room-server -- --profile local` |
| 4 | AdminServer | `curl -fsS --max-time 2 "$ADMIN_SERVER_BASE_URL/health"` | HTTP 200 | **2s** | **14** | `[4/5] AdminServer ❌ FAIL  $ADMIN_SERVER_BASE_URL/health\n        ↳ Hint: cargo run -p voice-room-admin-server -- --profile local` |
| 5 | Web | `curl -fsS --max-time 2 -o /dev/null "$ADMIN_WEB_URL"` | HTTP 200 / 301 / 302 | **2s** | **15** | `[5/5] Web        ❌ FAIL  $ADMIN_WEB_URL\n        ↳ Hint: cd app/web && npm run dev` |

**全局约束**：
- 总挂壁时间上限 ≤ 10s（5 项 × 2s）。**任一项失败立即退出**（fail-fast），后续项标记为 `⏭ skipped`。
- 颜色：成功 `\033[32m✅\033[0m`，失败 `\033[31m❌\033[0m`，跳过 `\033[33m⏭\033[0m`，hint 用 `\033[36m`。无 TTY（CI）时自动关闭颜色（检查 `[ -t 1 ]`）。
- Profile 裁剪：
  - `E2E_PROFILE=local` → 全部 5 项；
  - `E2E_PROFILE=staging` / `prod` → **跳过 1/2**（远端不直连 PG/Redis），只跑 3/4/5 项，跳过项打印 `⏭ skipped (remote profile)`。
- 全部成功退出码 **0**；至少 1 项失败按**首个失败项**的退出码退出（11~15）。
- `--profile <p>` CLI 入参覆盖 env；缺省读 `E2E_PROFILE`。

**输出范例**（成功）：
```
[1/5] Postgres    ✅ ready (127.0.0.1:5432)
[2/5] Redis       ✅ ready (127.0.0.1:6379)
[3/5] AppServer   ✅ 200  http://localhost:3000/health  (32ms)
[4/5] AdminServer ✅ 200  http://localhost:3001/health  (29ms)
[5/5] Web         ✅ 200  http://localhost:5173         (44ms)
preflight: all 5 checks passed in 0.21s
```

### 2.7 与 T-0000H globalSetup 的接口契约

| 调用 | 命令行 | stdout/stderr 协议 | 退出码语义 | globalSetup 行为 |
|------|--------|-------------------|-----------|------------------|
| Preflight | `bash scripts/dev/preflight.sh --profile $E2E_PROFILE` | 见 §2.6 行格式（每行 `[i/5] …` 可机读） | `0` 通过；`11~15` 对应 5 端 | 非 0 立即抛 `PreflightFailedError`，错误信息 = stderr 末 20 行 |
| Seed | `bash scripts/dev/seed-e2e.sh`（要求 `E2E_PROFILE=local && E2E_ALLOW_WRITES=1`） | 进度行 `[seed] step=N …`；最后 1 行打印 `seed: wrote scripts/dev/.seed-output.env` | `0` 成功；`21` profile 拒绝；`22` JWT 签发失败；`23` SQL 执行失败 | 成功后 `dotenv.parse(scripts/dev/.seed-output.env)` 注入 `process.env` |
| Reset | `bash scripts/dev/reset-e2e.sh --yes` | 每张表 `[reset] users -2 admins -4 rooms -1 …` | `0` 成功；`21` profile 拒绝；`24` 无连接 | globalTeardown 仅 local 调用，失败仅打 warning，不影响测试结果 |

> **契约不可变**：上述 3 条命令名、入参标志、退出码集合作为公共 API 冻结，T-0000H 与 T-0000I 直接消费。本 Task 内部实现可重构（如增加 `--quiet`），但**不得**变更上述四列。

## 三、TDD 验收用例

### 3.1 Seed 幂等性
- [ ] **正向**：`bash scripts/dev/seed-e2e.sh` 连跑 2 次，第 2 次退出码 0；对比 `psql -c "SELECT id,phone FROM users WHERE phone LIKE '+96650000090%' ORDER BY phone"` 输出 hash 一致。
- [ ] **正向**：连跑 2 次后 `users` E2E 行数严格 = 2，`admins` E2E 行数 = 4，`rooms` E2E 行数 = 1。
- [ ] **正向**：`.seed-output.env` 含全部 8 个键（`E2E_USER_A_ID/B_ID/ROOM_ID/VALID_TOKEN/ADMIN_TOKEN/OP_TOKEN/CS_TOKEN/FIN_TOKEN`），无空值。
- [ ] **正向**：随机抽 1 个 token base64 解码 payload，`exp - iat` 等于 `E2E_TOKEN_TTL_SECONDS`（默认 86400）。
- [ ] **异常**：`E2E_PROFILE=staging` 调 seed wrapper，立即退出码 21，**不** touch 任何 DB。
- [ ] **异常**：未设 `JWT_SECRET` 调 wrapper，退出码 22，stderr 含 `JWT_SECRET is required`。

### 3.2 Reset 安全
- [ ] **正向**：seed → 在 `users` 表手动 INSERT 一条 phone=`+966599999999` 的非 E2E 行 → reset → 该行仍在，且 E2E 4 行被清。
- [ ] **正向**：reset 后 `psql -c "\d users"` schema 与 reset 前 byte-for-byte 一致；`schema_migrations` 行数不变。
- [ ] **正向**：reset 后 `redis-cli --scan --pattern 'room:{E2E_ROOM_ID}:*' | wc -l` = 0；其他业务键 (`room:{other}:*`) 不动。
- [ ] **异常**：`E2E_PROFILE=prod bash scripts/dev/reset-e2e.sh --yes` 退出码 21，**不**触发任何 DELETE（用 `pg_stat_user_tables` 的 n_tup_del 前后差为 0 验证）。

### 3.3 Preflight 5 端 × 故意打挂
- [ ] **正向**：5 端全活，preflight 退出码 0，全程 < 2s。
- [ ] **异常**：`docker stop vr-postgres` → preflight 在 ≤ 2s 内退出，退出码 11，stdout 含 `[1/5] Postgres   ❌`，hint 含 `docker compose up -d postgres`。
- [ ] **异常**：`docker stop vr-redis` → 退出码 12，stdout 含 `[2/5] Redis`，**且**第 1 项仍打绿 ✅。
- [ ] **异常**：AppServer 未起 → 退出码 13 + hint 含 `cargo run -p voice-room-server`。
- [ ] **异常**：AdminServer 未起 → 退出码 14。
- [ ] **异常**：Web 未起 → 退出码 15。
- [ ] **异常**：5 端全挂 → 仍只在第 1 个失败项停下，退出码 11（fail-fast 行为验证）。
- [ ] **异常**：`E2E_PROFILE=staging` 时跳过 PG/Redis（项 1/2 显示 `⏭ skipped`），仅跑 3/4/5。
- [ ] **正向**：`CI=1` 或非 TTY 环境下颜色字节不出现（`preflight | cat | grep -c $'\033'` = 0）。
