# TDS: 短信验证码发送接口 (Task ID: T-00002)

## 一、背景与目标
- **关联需求**：product.md - 手机号一步登录
- **前置依赖**：T-00001 (数据库表设计)
- **本 Task 的目标**：实现 POST `/api/v1/auth/send-code` 接口，通过 SmsProvider 防腐层发送验证码，Redis 存储

## 二、方案设计
### 核心数据流
Client → POST /api/v1/auth/send-code → Controller → AuthService → 检查 Redis cooldown/daily → 生成6位验证码 → SmsProvider.send() → 写入 Redis (code + cooldown + daily incr) → 返回 {expires_in, cooldown}

### 涉及文件清单
| 文件路径 | 变更类型 | 说明 |
|----------|----------|------|
| `app/server/src/modules/auth/controller.rs` | 新增 | send_code handler |
| `app/server/src/modules/auth/service.rs` | 新增 | 验证码生成 + Redis 读写逻辑 |
| `app/server/src/infrastructure/third_party/sms/mod.rs` | 新增 | SmsProvider trait |
| `app/server/src/infrastructure/third_party/sms/twilio.rs` | 新增 | Twilio 实现 |
| `app/server/src/infrastructure/third_party/sms/mock.rs` | 新增 | Mock 实现（固定 000000） |

### 接口 / 信令定义
参见 `doc/protocol.md` §二 2.1 POST /api/v1/auth/send-code

### Redis Key 设计
参见 protocol.md §六 6.2：
- `sms:code:{phone}` Hash (TTL 300s)
- `sms:cooldown:{phone}` String (TTL 60s)
- `sms:daily:{phone}:{date}` INCR (TTL 86400s)

## 三、TDD 验收用例
- [ ] 正向：发送成功返回 {expires_in: 300, cooldown: 60}
- [ ] 正向：验证码为 6 位数字，存入 Redis
- [ ] 异常：同一手机号 60 秒内重复发送返回 429 (42901)
- [ ] 异常：当日超过 10 次返回 429 (42902)
- [ ] 异常：手机号格式无效返回 400 (40001)
- [ ] 异常：SmsProvider 失败时返回 500，不写入 Redis

### 第二轮 Review（Review Agent，commit `df327b9`）
> 状态枚举：🟢 通过 | 🔴 未通过

#### Review 结论：🔴 未通过（3 HIGH）

**H-01（跨 T-00002/03/05）错误响应 `request_id` 永远为空字符串**  
- 位置：`src/common/error.rs:116`  
- `AppError::into_response()` 硬编码 `request_id: String::new()`；注释说"middleware 会注入"，但 `request_context_middleware` 只写入 response header，不修改 body JSON。  
- `err_response()` 函数存在（L123）但从未被调用——死代码。  
- 修复：controller 中 error 路径改用 `err_response(e, rc.request_id())`，或引入 `AppErrorWithContext(AppError, String)` wrapper。

**H-02 SMS 发送失败后 Redis 状态未回滚**  
- 位置：`src/modules/auth/service.rs:53-54`  
- `save_code` 已写入 Redis（code + cooldown + daily），SMS 失败时 Redis 状态不回滚。  
- 用户进入冷却期但从未收到验证码，60s 内无法重发。  
- 修复：先发 SMS（成功后再 `save_code`）；或 SMS 失败后调用 `code_store.undo_save_code(phone, today)`。

**H-03 `save_code` 冷却/日限检查非原子 TOCTOU**  
- 位置：`src/infrastructure/redis_store/mod.rs:58-71`  
- 三步非原子：EXISTS cooldown → GET daily → pipeline(write)。并发请求可双重通过检查，同一手机号接收两条验证码。  
- 修复：用 Lua 脚本将检查+写入原子化（EVALSHA）。

**M-01 `err_response` 死代码**（配合 H-01 一起修复或删除）  
**M-02 `is_in_cooldown`/`daily_count` trait 方法 service 层不调用，暴露冗余接口**  
**M-03 `DatabaseError`/`SmsSendFailed` 原始错误字符串输出客户端，泄露内部信息**

### 第三轮 Review（Review Agent，commit `86a8fd9`）
> 状态枚举：🟢 通过 | 🔴 未通过

#### Review 结论：🔴 未通过（2 HIGH）

**H-01 `verify_and_consume` 非原子 TOCTOU（安全）**
- 位置：`src/infrastructure/redis_store/mod.rs:118-147`
- HGETALL → 判断 → HSET/DEL 三步非原子。并发两个正确验证码请求均能通过，同一 OTP 可被消费两次。
- 修复：用 Lua 脚本原子化 verify_and_consume（同 save_code H-03 方案）

**H-02 `daily_count` 静默返回 0 绕过限流（安全/可靠性）**
- 位置：`src/infrastructure/redis_store/mod.rs:159`
- `.unwrap_or(0)` 在 Redis 不可用时静默返回 0，导致限流失效，短信可被无限发送。
- 修复：改为传播 `AppError::RedisError`，`None` 时才返回 0。

**M-01 Lua 错误字符串匹配脆弱**
- 位置：`src/infrastructure/redis_store/mod.rs:107-110`
- `contains("COOLDOWN")` 可能误匹配 Redis 代理返回的包含该词的错误。
- 修复：Lua 使用项目前缀 `VR:COOLDOWN` / `VR:DAILY_LIMIT`，Rust 侧精确匹配。

**M-02 `now_secs()` 使用 `unwrap()` 潜在 panic 路径**
- 位置：`src/modules/auth/service.rs:171-175`
- `duration_since(UNIX_EPOCH).unwrap()` 在系统时钟异常时 panic。
- 修复：改为 `.unwrap_or_default()`。
