# TDS: AppServer / AdminServer 暴露统一 `/health` 端点 (Task ID: T-0000N)

## 一、背景与目标

- **关联需求**：模块 9 E2E 测试基建 follow-up；T-0000H 起的预存在缺陷；T-0000M Round 1 GlobalReview 缺陷 2（DoD #1 失实）拆出。
- **本 Task 的目标**：让 AppServer 与 AdminServer 各自暴露一个轻量、零鉴权的 `GET /health` 端点（200 OK + 极简 JSON），使 `scripts/dev/e2e-up.sh` 中 `wait-on http-get://localhost:3000/health` 与 `http-get://localhost:3001/health` 真正生效，同时让 `scripts/dev/preflight.sh` 的 5 端健康检查（PG/Redis/AppServer/AdminServer/Web）由「假绿」变为「真绿」。
- **现状证据**：
  - `app/server/src/bootstrap/mod.rs:335` 仅注册 `/ping`，`grep -rn '"/health"' app/server/src` 0 命中。
  - `app/adminServer/src/bootstrap/mod.rs:172-194` 全部为 `/api/v1/admin/*` 业务路由，无任何健康端点。
  - `scripts/dev/e2e-up.sh` 的 `wait-on http-get://...:3000/health` 在 T-0000M 闭环时实测 60s 超时，是模块 9 E2E `npm run e2e:up` 5 端 wait-on 全绿的最后一个阻塞。
- **不在范围**：
  - 不做 readiness vs liveness 分离（不引入 `/healthz/live`、`/healthz/ready` 二元体系）。
  - 不做依赖深度探测（不在 /health 内部 ping DB/Redis；理由：探活端点必须毫秒级返回，深度探测放到 preflight.sh 即可）。
  - 不做 K8s probe 路径标准化（当前部署形态为 docker compose + 裸进程，统一 `/health` 已满足）。

## 二、方案设计

### 核心数据流

```
[wait-on / preflight.sh / 监控探针]
       │  GET /health
       ▼
[axum Router]  ──> health_handler  ──> 200 + JSON { status: "ok", service, version }
       │
       └─ 与 /ping 同层挂载，免鉴权（不进 auth middleware），免 request_context 注入
```

### 设计要点

1. **零依赖、零副作用**：handler 不访问 DB / Redis / 外部服务，纯返回静态 JSON。理由：探活语义 = 「进程是否存活并能响应 HTTP」，非「下游是否健康」；后者由 preflight 单独检查。
2. **响应体统一**：`{ "status": "ok", "service": "app-server" | "admin-server", "version": env!("CARGO_PKG_VERSION") }`。`status` 字段固定 `"ok"`，便于运维脚本 `jq -e '.status == "ok"'` 校验；`service` 字段区分两服务，便于聚合监控查询。
3. **路由位置**：与 `/ping` 同级挂载（`/health` 不进 `/api/v1` 前缀，符合业界惯例与 wait-on 默认路径预期）。
4. **不进鉴权 / 上下文中间件**：`/health` 必须能在「无 cookie / 无 JWT / 无 request_id header」状态下被任何客户端探测，故不能让 `request_context_middleware` 介入。**实现做法**：在 `Router::new()` 上注册 `/health` 后再 `.layer(middleware::from_fn(request_context_middleware))` 的位置之前；当前 AppServer `/ping` 已与中间件同层挂载，调用方 `request_context` 来自 Extension，意味着中间件实际上覆盖到 `/ping`——为避免改变现有行为，新 `/health` handler 直接不读 Extension 即可（中间件注入 Extension 但不强制要求 handler 消费）。
5. **不读取业务 state**：`async fn health() -> Json<HealthResponse>`，签名零参数，与 AppState 无耦合，便于在测试中独立断言。

### 接口定义

```
GET /health
Response: 200 OK
Content-Type: application/json
Body:
{
  "status": "ok",
  "service": "app-server" | "admin-server",
  "version": "0.1.0"  // env!("CARGO_PKG_VERSION") 编译期注入
}
```

无错误响应分支（handler 不会失败；进程死亡则 TCP 连接被拒，由探活方判定 down）。

### 核心 handler 草图

```rust
// AppServer: app/server/src/bootstrap/mod.rs
#[derive(Serialize)]
struct HealthResponse {
    status: &'static str,
    service: &'static str,
    version: &'static str,
}

async fn health() -> Json<HealthResponse> {
    Json(HealthResponse {
        status: "ok",
        service: "app-server",
        version: env!("CARGO_PKG_VERSION"),
    })
}
```

AdminServer 同形，仅 `service: "admin-server"`。
