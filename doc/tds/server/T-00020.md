# TDS: SendGift 事务 + 广播 (Task ID: T-00020)

## 一、背景与目标
- **关联需求**：E-07 核心 — 送礼扣减 + 收礼魅力累加 + 广播 + 榜单 + 幂等 ([phase1_gift_economy.md §4](../../product/phase1_gift_economy.md))
- **本 Task 的目标**：在 WS 信令层实现 `SendGift`，一条强事务完成 6 张资源变更（users 余额 / charm / wallet_transactions / gift_records / Redis 日榜 / Redis 周榜），并基于 `(sender_id, msg_id)` 做幂等，广播 `GiftReceived` 给房间。

## 二、方案设计

### 核心数据流
```
Client WS SendGift {gift_id, receiver_id, count, msg_id}
   └─► ws/handler → gift::handle_send_gift()
        1. 校验用户在房间、接收者在同房间且在麦上
        2. 幂等：SELECT FROM gift_records WHERE sender_id=? AND msg_id=?
               命中 → 返回首次 code & 不重发广播
        3. 查 gift，total = price * count
        4. BEGIN TX
            a) wallet.apply_delta(sender, -total, GiftSend, ref=None)
                 ↳ 余额不足 → 回滚 → INSUFFICIENT_BALANCE
            b) UPDATE users SET charm_balance += total WHERE id=receiver
            c) INSERT gift_records(id, sender_id, receiver_id, room_id, gift_id, count, total, msg_id)
            d) wallet.apply_delta(sender) 的 wallet_txn 补 ref_id=gift_record.id
           COMMIT
        5. Redis ZINCRBY charm:day:{date} receiver total; ZINCRBY wealth:day:{date} sender total
                        charm:week:{isoweek}、wealth:week:{isoweek} 同步
        6. 广播 GiftReceived 给 registry.get_connections_in_room(room_id)
        7. 发送者单播 BalanceUpdated（由 wallet.apply_delta 已触发）
```

### 迁移脚本 (006_create_gift_records.sql)
```sql
ALTER TABLE users
    ADD COLUMN IF NOT EXISTS charm_balance BIGINT NOT NULL DEFAULT 0 CHECK (charm_balance >= 0);

CREATE TABLE IF NOT EXISTS gift_records (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    sender_id    UUID NOT NULL REFERENCES users(id),
    receiver_id  UUID NOT NULL REFERENCES users(id),
    room_id      UUID NOT NULL REFERENCES rooms(id),
    gift_id      UUID NOT NULL REFERENCES gifts(id),
    count        INT NOT NULL CHECK (count >= 1 AND count <= 9999),
    total_price  BIGINT NOT NULL CHECK (total_price >= 1),
    msg_id       VARCHAR(64) NOT NULL,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (sender_id, msg_id) -- 幂等
);
CREATE INDEX idx_gift_records_receiver_created ON gift_records(receiver_id, created_at DESC);
CREATE INDEX idx_gift_records_room_created ON gift_records(room_id, created_at DESC);
```

### WS 信令
**请求 C→S `SendGift`**
```json
{
  "type": "SendGift",
  "msg_id": "uuid",
  "payload": { "gift_id": "uuid", "receiver_id": "uuid", "count": 1 }
}
```

**响应 S→C（发送者）`SendGiftResult`**
```json
{
  "type": "SendGiftResult",
  "msg_id": "uuid",
  "code": 0,
  "payload": { "gift_record_id":"uuid", "total_price":520 }
}
```
错误码：`40002`（参数缺失）、`40400`（用户不在房间）、`40402`（gift 不存在/下架）、`40403`（接收者不在房间或不在麦 → `RECEIVER_UNAVAILABLE`）、`40290`（`INSUFFICIENT_BALANCE`）、`40001`（count 非法）。

**广播 S→C `GiftReceived`**
```json
{
  "type": "GiftReceived",
  "payload": {
    "gift_record_id": "uuid",
    "sender": { "user_id":"uuid", "nickname":"...", "avatar":"..." },
    "receiver": { "user_id":"uuid", "nickname":"...", "avatar":"..." },
    "gift": { "id":"uuid","code":"castle_01","name":"قصر","icon_url":"...","effect_level":4,"animation_url":"..." },
    "count": 1,
    "total_price": 520
  },
  "timestamp": 1720000000000
}
```

### 幂等实现
`UNIQUE (sender_id, msg_id)` 约束 + 在业务层先 SELECT 命中则返回首次结果。并发下若两个同 msg_id 同时到达，第二条 INSERT 失败被捕获后也会转为 SELECT 返回首条成功结果（并返回相同 code=0，但**不重发广播**）。

### 接收者可用性
- 读取 `room_state.mic_slots`（T-00014 维护），receiver 必须在麦
- 允许房主送给任意在麦用户（包括自己？MVP：允许给自己）

### Redis 榜单键
- 日榜：`ranking:charm:day:{YYYY-MM-DD}` / `ranking:wealth:day:{YYYY-MM-DD}`（Asia/Riyadh 时区）
- 周榜：`ranking:charm:week:{YYYY-WW}` / `ranking:wealth:week:{YYYY-WW}`
- TTL：日榜 48h，周榜 10d（防止跨日丢数据，留归档窗口）

## 三、TDD 验收用例
- [ ] **SG01** 送礼成功：发送者余额 -total、接收者 charm_balance += total、gift_records +1、wallet_transactions +1
- [ ] **SG02** 房间所有成员（含麦上/麦下观众）收到 `GiftReceived`
- [ ] **SG03** 发送者单独收到 `BalanceUpdated { delta: -total }`
- [ ] **SG04** Redis `ZSCORE ranking:charm:day:...` 收到 total
- [ ] **SG05** 余额不足（total > balance）整体回滚，返回 `INSUFFICIENT_BALANCE (40290)`
- [ ] **SG06** 幂等：相同 `(sender, msg_id)` 二次发送返回首次 code，不再扣款、不再广播
- [ ] **SG07** 接收者离开麦位后送礼返回 `RECEIVER_UNAVAILABLE (40403)`
- [ ] **SG08** gift 被下架 `is_active=false`：返回 40402
- [ ] **SG09** count=0 / count=10000：返回 40001
- [ ] **SG10** 并发 20 个送礼请求同一发送者：无超扣、事务隔离
- [ ] **SG11** 发送者自己不在房间：返回 40400
- [ ] **SG12** 事务中途任一步失败（手动注入）：余额不变、榜单不变、无广播

### 关键实现决策

1. **事务内联实现**：直接在 `GiftSendService::execute_transaction()` 中执行 `SELECT FOR UPDATE + UPDATE + INSERT`，未复用 `WalletService::apply_delta`（避免跨 Arc 具体类型依赖），SQL 等效。
2. **幂等设计**：业务层先 `SELECT` 命中返回 `Ok` （不重发广播）；DB 层 `UNIQUE(sender_id, msg_id)` 约束作为最终兜底防线；`Idempotent` 枚举变体已删除（[H-2]）。
3. **Redis 非阻断**：Redis 连接失败仅记录 `warn` 日志，不影响事务结果（依据 TDS "非核心操作不阻断主链路"）。
4. **BalanceEvent 投递**：改为 `send().await` 确保事件不丢失；若接收端已关闭则记录 `warn` 日志（[M-1]）。

#### 🔴 CRITICAL（必须修复）

**[C-1] `ranking.rs` charm_day 双重计数 + 历史覆盖 Bug**
- **文件**：`app/server/src/modules/gift/ranking.rs:61-65`
- **问题**：`charm_day` 更新逻辑先调用 `zadd`（ZADD：SET/覆盖分数），再调用 `zincr`（ZINCRBY：累加），导致：
  1. **双倍计数**：每次送礼分数变成 `2 × total`（zadd 设为 total，zincr 再加 total）
  2. **历史覆盖**：第二次礼物 zadd 先将历史累积值覆盖为 `total`，再 zincr 加 total，所有历史数据丢失
  3. 其他三个榜单（charm_week / wealth_day / wealth_week）均正确地只用 `zincr`，charm_day 是唯一异常
  4. SG04 断言为 `score >= total`，未能捕获此 Bug（双倍结果同样满足 `>= total`）
- **修复**：删除 charm_day 的 `zadd` 逻辑，改为与其他三个榜单一致的纯 `zincr`：
  ```rust
  // 修复后 charm_day（与 charm_week 保持一致）
  let charm_day = charm_day_key();
  let _ = increment_zscore(conn, &charm_day, &recv_str, score).await;
  let _: Result<(), _> = conn.expire(&charm_day, DAY_TTL_SECS as i64).await;
  ```
  同时更新 SG04，断言 `score == total`（精确值）而非 `>= total`。

**[H-2] `SendGiftError::Idempotent` 是死代码，handler 对应分支不可达**
- **文件**：`send_gift.rs:61-65`（枚举定义）、`send_gift.rs:292`（实际返回 Ok）、`send_gift.rs:444-446`（不可达的 handler 分支）
- **问题**：`send()` 在幂等命中时返回 `Ok(SendGiftResult { ... })`，从不返回 `Err(Idempotent)`；handler 中处理 `Err(Idempotent)` 的 match 臂永远不会被执行。功能上行为正确（幂等返回 code=0），但枚举变体、handler 分支以及文档注释（`send_gift.rs:102`）均构成死代码。
- **修复**：删除 `SendGiftError::Idempotent` 枚举变体，删除 handler 中对应 match 臂，更新文档注释说明幂等命中返回 `Ok(...)`。

**[H-3] `protocol.md §6.4.2` 错误码草稿值未更新，与实现不一致**
