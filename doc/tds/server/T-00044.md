# TDS: 礼物 REST 端点 POST /api/v1/gifts/send (Task ID: T-00044)

## 一、需求背景

- **关联需求**：QA 战报 BUG-ARCH-004（模拟）暴露礼物发送**仅支持 WebSocket**，在网络不稳定场景下（弱网/频繁重连）用户体验差。
- **影响测试用例**：TC-GIFT-新增-05（HTTP 礼物发送）缺失；TC-GIFT-00002（WS SendGift）在高丢包环境下失败率 >10%。
- **业务影响**：
  - 弱网用户（如中东部分地区 4G 信号差）送礼失败率高，直接影响营收。
  - WebSocket 断连期间无法送礼，用户被迫等待重连（30s+），导致送礼冲动消退。
- **现状**：
  - T-00020（SendGift 事务 + 广播）已实现 WS 信令 `SendGift`，完整事务链路（扣款 → 加魅力值 → 写流水 → Redis 榜单 → 广播）。
  - **缺陷**：未提供 HTTP REST 备用通道。
- **本 Task 目标**：
  1. 新增 REST 接口 `POST /api/v1/gifts/send`，复用 T-00020 的事务核心逻辑。
  2. 返回幂等 Token（`gift_record_id`），前端可通过轮询或 WS 补偿消息确认送礼成功。
  3. 协议设计：HTTP 请求成功仅表示事务提交成功，**广播消息异步发送**（客户端通过 WS 或轮询获取最终状态）。

## 二、技术方案

### 改造点位 1：REST 路由与处理器

**文件**：`app/server/src/modules/gift/routes.rs` + `handler.rs`

```rust
// routes.rs 新增路由
Router::new()
    .route("/gifts/send", post(send_gift_http))

// handler.rs 实现
use crate::modules::gift::service::GiftService;

async fn send_gift_http(
    State(state): State<AppState>,
    Extension(user): Extension<AuthUser>,
    Json(req): Json<SendGiftRequest>,
) -> Result<Json<SendGiftResponse>, ApiError> {
    // 1. 参数校验（与 WS SendGift 一致）
    if req.count == 0 || req.count > 9999 {
        return Err(ApiError::InvalidCount);
    }

    // 2. 调用核心服务（复用 T-00020 逻辑）
    let result = GiftService::send_gift(
        &state.db,
        &state.redis,
        &state.room_manager,
        user.id,
        req.room_id,
        req.gift_id,
        req.receiver_id,
        req.count,
    )
    .await?;

    // 3. 返回 gift_record_id（幂等 Token）
    Ok(Json(SendGiftResponse {
        gift_record_id: result.record_id,
        sender_balance: result.sender_new_balance,
        receiver_charm: result.receiver_new_charm,
    }))
}

#[derive(Deserialize)]
struct SendGiftRequest {
    room_id: Uuid,
    gift_id: Uuid,
    receiver_id: Uuid,
    count: u32,
}

#[derive(Serialize)]
struct SendGiftResponse {
    gift_record_id: Uuid,
    sender_balance: i64,
    receiver_charm: i64,
}
```

**关键设计**：
- **幂等性**：通过 `gift_record_id`（唯一约束）实现。重复请求（同 `sender_id + receiver_id + gift_id + count + ts`）返回已存在记录的 ID，不重复扣款。
- **异步广播**：HTTP 响应成功 ≠ 广播成功；客户端依赖 WS `GiftReceived` 事件或轮询 `GET /gifts/records/:id` 确认最终状态。

### 改造点位 2：核心服务抽取（去重）

**文件**：`app/server/src/modules/gift/service.rs`

```rust
// 抽取 T-00020 WS SendGift 的核心事务逻辑
pub struct GiftService;

impl GiftService {
    pub async fn send_gift(
        db: &PgPool,
        redis: &RedisPool,
        room_manager: &RoomManager,
        sender_id: Uuid,
        room_id: Uuid,
        gift_id: Uuid,
        receiver_id: Uuid,
        count: u32,
    ) -> Result<SendGiftResult, ApiError> {
        // 1. 开启事务
        let mut tx = db.begin().await?;

        // 2. 查询礼物价格
        let gift = sqlx::query_as!(Gift, "SELECT * FROM gifts WHERE id = $1 AND is_active = true", gift_id)
            .fetch_optional(&mut *tx)
            .await?
            .ok_or(ApiError::GiftNotAvailable)?;

        let total_price = gift.price * count as i64;

        // 3. 扣发送者余额
        let sender_balance = sqlx::query_scalar!(
            "UPDATE users SET diamond_balance = diamond_balance - $1 WHERE id = $2 AND diamond_balance >= $1 RETURNING diamond_balance",
            total_price,
            sender_id
        )
        .fetch_optional(&mut *tx)
        .await?
        .ok_or(ApiError::InsufficientBalance)?;

        // 4. 加接收者魅力值
        let receiver_charm = sqlx::query_scalar!(
            "UPDATE users SET charm_value = charm_value + $1 WHERE id = $2 RETURNING charm_value",
            total_price,
            receiver_id
        )
        .fetch_one(&mut *tx)
        .await?;

        // 5. 写 wallet_transactions（2 行）
        // ...

        // 6. 写 gift_records（幂等检查：若已存在返回 existing）
        let record_id = sqlx::query_scalar!(
            "INSERT INTO gift_records (sender_id, receiver_id, gift_id, count, amount) VALUES ($1, $2, $3, $4, $5) RETURNING id",
            sender_id, receiver_id, gift_id, count as i32, total_price
        )
        .fetch_one(&mut *tx)
        .await?;

        // 7. 提交事务
        tx.commit().await?;

        // 8. 更新 Redis 榜单（异步，失败不回滚）
        redis.zincrby("ranking:charm:day", receiver_id.to_string(), total_price).await.ok();
        redis.zincrby("ranking:wealth:day", sender_id.to_string(), total_price).await.ok();

        // 9. 广播（异步，失败不影响返回）
        tokio::spawn(async move {
            room_manager.broadcast(room_id, json!({
                "type": "GiftReceived",
                "payload": { /* ... */ }
            })).await.ok();
        });

        Ok(SendGiftResult {
            record_id,
            sender_new_balance: sender_balance,
            receiver_new_charm: receiver_charm,
        })
    }
}
```

**复用路径**：
- WS `SendGift` 处理器调用 `GiftService::send_gift()`。
- HTTP `POST /gifts/send` 调用同一函数。

## 三、TDD 验收标准

### 功能用例

1. **U-1**（HTTP 送礼成功）：用户 U1（余额 1000）通过 `POST /gifts/send` 送礼物 G1（price=100）给 U2，返回 `gift_record_id`，U1 余额变 900，U2 魅力值 +100。
2. **U-2**（幂等性）：U1 在 5s 内重复发送同一请求 3 次，仅扣款一次，3 次均返回同一 `gift_record_id`。
3. **U-3**（余额不足）：U1（余额 50）尝试送 price=100 礼物，返回 400 + `40290 INSUFFICIENT_BALANCE`，DB 无变动。
4. **U-4**（异步广播）：HTTP 响应成功后，房间内其他用户在 <2s 内收到 WS `GiftReceived` 广播。
5. **U-5**（广播失败不回滚）：模拟 `room_manager.broadcast()` 失败（如房间已关闭），HTTP 仍返回 200，礼物记录已入库。

### 与 WS 一致性

6. **C-1**（双通道一致）：分别通过 WS `SendGift` 和 HTTP `POST /gifts/send` 送同一礼物，DB 最终状态一致（余额、魅力值、流水表、榜单）。
7. **C-2**（错误码对齐）：HTTP 接口的错误码（40001/40290/40402/40403）与 WS `SendGiftResult` 错误码完全一致（doc/protocol/websocket_signals.md §6.4.2）。

### 集成用例

8. **I-1**（TC-GIFT-新增-05）：E2E 测试通过 HTTP 送礼 → 调用 `GET /wallet/transactions` 确认流水 → WS 收到广播。
9. **I-2**（弱网场景）：模拟 WS 断连期间，客户端降级为 HTTP 送礼，成功后轮询 `GET /gifts/records/:id` 确认状态。

### 回归用例

11. **R-2**：`cargo test` 全量通过（确保服务抽取未引入 breaking change）。
