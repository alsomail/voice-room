### 前置确认结论（2026-04-29 Plan 实地核实，覆盖上节假设）

> Plan Agent 已 grep + 阅读源码核验下列前置；上节"AppServer 侧前置确认"假设**与现状不符**，本节为权威方案，TDD 实现以本节为准。

**A. `/auth/test-login` 端点不存在**
- `app/server/src/modules/auth/routes.rs` 仅暴露 3 条路由：`/api/v1/auth/verification-codes` / `/api/v1/auth/login` / `/api/v1/users/me`。
- `app/server/src/modules/auth/controller.rs` 无 `test_login` / `dev_login`。
- T-00041 = WS 心跳；T-00042 = Admin 事件订阅 — 均与 test-login 无关，原 TDS 引用有误。
- `app/shared/src/jwt/token.rs::AppClaims` 字段集 = `{sub, iss, exp, iat}`，**不含 `role` / `muted`**（`role` 仅在 `AdminClaims` 中，iss=`voiceroom-admin`，与用户 token 域隔离）。
- 结论：业务侧零改动前提下，**放弃** `/auth/test-login` 路径。

**B. 改用现成的 `sign-jwt` CLI + `seed-e2e.sh` 注入（业务代码 0 行改动）**
- `app/shared/src/bin/sign_jwt.rs` 已提供 `--sub/--role/--ttl` 与 `--uuid5` 模式；`role=user` 走 AppClaims，`role=admin/op/cs/fin` 走 AdminClaims。
- `scripts/dev/seed-e2e.sh` 已完成：USER_A/B 用户行 + 5 个 admin 行，并签发 `E2E_VALID_TOKEN`(USER_A,user) / `E2E_ADMIN_TOKEN` / `E2E_OP_TOKEN`(operator) / `E2E_CS_TOKEN` / `E2E_FIN_TOKEN`，回写 `scripts/dev/.seed-output.env`。
- **缺口仅 2 个**：
  - `E2E_USER_B_TOKEN` — USER_B 行已 seed，但 wrapper 未对其 `sign --role user`。
  - `E2E_MUTED_TOKEN` — `e2e_muted` 用户行 + token 都缺。

**C. muted 状态通过 Redis SETEX 实现（不依赖 AdminServer REST）**
- 真值：`app/server/src/modules/governance/mute.rs` 中 SendMessage 前置拦截读取 key `chat_muted:{room_id}:{user_id}`，存在即 40305 / 40303（TC-CHAT-00004 期望 40303）。
- 因此 setup 阶段只需 `redis-cli SETEX chat_muted:<E2E_ROOM_ID>:<E2E_USER_MUTED_ID> 86400 e2e_seed` 即可，不需要 AdminServer mute REST 端点（该端点目前是基于 WS 信令的 `MuteUser`，不适合在 globalSetup 直接调用）。

**D. Docker 容器名修正**
- `docker-compose.yml`：`container_name: vr-redis` / `vr-postgres`（**不是** TDS §2.1 写的 `voice-room-redis-1` / `voice-room-pg-1`）。
- `redisCli.ts` 实现以 `docker exec vr-redis redis-cli ...` 为准；INFRA follow-up（Docker stop 用例）相应使用 `vr-postgres`。
