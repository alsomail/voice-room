# TDS: E2E fixture token + redis-cli 容器化自动注入（解锁 29 SKIP-KNOWN） (Task ID: T-0000S)

## 一、背景与目标

- **关联战报**：[report-20260428-154125 SUMMARY](../../../tests/report-20260428-154125/SUMMARY.md) §"Skip 原因汇总（SKIP-KNOWN）"
- **当前痛点**：API 套件 29 / 78 用例长期 SKIP，原因是**环境/fixture 缺失**而非逻辑漏洞，长期 SKIP 会让回归报告产生 "看似全绿实则有盲区" 的假象。
- **本 Task 的目标**：在 `globalSetup` 阶段自动准备 3 个特殊角色 token + 容器化 `redis-cli`，使 29 个 SKIP-KNOWN 用例本地 / CI 全部能默认跑通。

### SKIP 用例缺什么（来自 SUMMARY）

| 缺失资源 | 影响用例 | 用例数 |
|---------|---------|-------|
| `redis-cli` 不在 PATH | TC-AUTH 13 + TC-WS 3 + TC-RANKING 1 | 17 |
| `E2E_OP_TOKEN`（运营/CS/admin） | TC-MIC 6 | 6 |
| `E2E_USER_B_TOKEN`（第二个普通用户） | TC-CHAT 1 + TC-GIFT 1 | 2 |
| `E2E_MUTED_TOKEN`（已禁言用户） | TC-CHAT 1 | 1 |
| Docker 控制权限 | TC-INFRA 2 | 2 |
| 干净端口环境 | TC-INFRA-Q 1 | 1（已由 T-0000Q 部分覆盖） |
| **合计** | — | **29**（其中本 Task 直接解锁前 4 类共 26 用例；Docker / 端口干净 2+1 标 follow-up） |

## 二、方案设计

### 核心数据流

```
playwright.globalSetup
    │
    ├─→ envLoader（T-0000H 已落地）加载 .env.local
    │
    ├─→ [新增] fixtures/seedTokens.ts
    │       │
    │       ├─ 调 AppServer /auth/test-login（已存在的 dev-only 端点）
    │       │     · subject=e2e_op,    role=admin    → E2E_OP_TOKEN
    │       │     · subject=e2e_user_b,role=user     → E2E_USER_B_TOKEN
    │       │     · subject=e2e_muted, role=user, muted=true → E2E_MUTED_TOKEN
    │       │
    │       └─ 注入 process.env （所有用例可直接读）
    │
    └─→ [新增] fixtures/redisCli.ts
            │
            └─ exec('docker compose ps voice-room-redis-1 -q')
              · 命中 → 暴露 helper exec(redisCmd) → docker exec ... redis-cli ...
              · 未命中 → 回退本地 redis-cli（旧路径）
              · 都没有 → 仅 redis 类用例 skip-known，其余继续
```

### AppServer 侧前置确认

- `/auth/test-login` 端点：仓库当前已存在（详见 T-00041 / T-00042 实现），仅在 `APP_PROFILE in {dev, test, local}` 时挂载。本 Task **不修改 AppServer 源码**，仅消费现有端点。
- 若该端点对 `muted=true` 字段尚不支持，则降级方案：先 `/auth/test-login` 拿 token，再 setup 阶段直接调 AdminServer `POST /admin/v1/users/{id}/mute` 把 `e2e_muted` 标记禁言。AdminServer 该端点见 T-00042 已落地。

### Docker 控制权限 / 干净端口（follow-up，不在本 Task 范围）

- **TC-INFRA-00001/00002（2 用例）**：要求 Playwright 进程能 `docker stop voice-room-pg-1` 模拟服务挂掉。本地需 Docker socket 权限，CI 需特权 runner。**本 Task 仅在文档登记**，不实现；建议未来拆 T-0000T。
- **TC-INFRA-Q-I-2（1 用例）**：要求干净端口环境（无 5432/6379 占用）才能验证 preflight 报错。本地不便构造，CI 需独立 job。同样**本 Task 不动**。

> 因此本 Task 直接解锁 **26/29** SKIP-KNOWN，剩余 3 个由 follow-up 收口。

## 三、TDD 验收用例

- [ ] **本地默认跑**：`npm run e2e:local -- tests/scripts/API/` 输出 `75 passed / 3 skipped`（3 = INFRA Docker 2 + INFRA-Q 1，均 SKIP-KNOWN-FOLLOWUP）
- [ ] **OP token 注入**：globalSetup 后 `process.env.E2E_OP_TOKEN` 非空且 JWT decode `role=admin`
- [ ] **USER_B 注入**：JWT decode `subject=e2e_user_b, role=user`
- [ ] **MUTED 注入**：JWT decode `subject=e2e_muted`，且 `GET /admin/v1/users/{id}` 显示 `muted=true`
- [ ] **redisCli 容器化**：本地未 `brew install redis` 时，TC-AUTH / TC-WS / TC-RANKING 仍跑通（透过 `docker exec voice-room-redis-1 redis-cli`）
- [ ] **redisCli fallback**：删除 docker compose 中 redis 容器（极端场景）后，redis 类用例 skip 而非 fail，其它用例不受影响
- [ ] **幂等性**：globalSetup 重复运行 5 次，三个 e2e_* 用户在 DB 内仅各 1 行（`ON CONFLICT (subject) DO UPDATE`）
- [ ] **CI 自动跑通**：`.github/workflows/*.yml` 不再需要任何 `if: env.E2E_OP_TOKEN` 条件分支
- [ ] **不修改 AppServer / AdminServer 源码**：`git diff -- app/` 0 行
- [ ] **RUNBOOK 更新**：`doc/tests/E2E_RUNBOOK.md` 含 "26/29 SKIP-KNOWN 已自动收口" 章节，并标注 follow-up T-0000T 占位
