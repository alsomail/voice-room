# TDS: Redis Pub/Sub `admin:events` 双端契约对齐（adminServer 发布端 strict-typed + server 消费端 schema 校验）(Task ID: T-00105)

## 一、背景与目标

- **关联需求**：实证 — adminServer `AdminEvent { r#type: String, admin_id: String, payload: serde_json::Value }` 完全 schema-less，对端 server `AdminEvent` 是 strict enum，任何字符串拼写错误都会**静默丢事件**。`admin:events` 是 BAN/CLOSE_ROOM/BROADCAST 的关键管道，丢失即治理失效。
- **本 Task 的目标**：
  1. adminServer publisher 改用与 server consumer **共享同一份枚举**（通过 `app/shared` crate 提供 strict enum）；publisher 不再可能传错 `event_type` 字符串。
  2. 消费端 server 加运行时 JSON Schema 校验（基于 T-00100 schemas/pubsub/*.schema.json）；所有未通过 schema 的事件丢入 dead-letter 日志 + 上报埋点。
  3. 双端往返集成测试：adminServer 发布 → Redis → server 消费 → 4 类事件全部往返成功。

#### MEDIUM（不阻塞，建议后续修复）

**[MEDIUM] `subscriber.rs:80` 未知/非法 event_type 使用 `tracing::error!` 而非 `tracing::warn!`**

- **位置**：`app/server/src/events/subscriber.rs:80-84`
- **现状**：serde JSON 解析失败（含未知 `event_type`）时记录 `tracing::error!`
- **规范要求**：TDS §1 及审查要求明确"未知 event_type → WARN 日志，不崩溃"
- **影响**：多版本滚动发布时，旧 server 接收新 adminServer 事件类型会产生大量 ERROR 告警，造成告警疲劳；实际无数据丢失风险
- **建议**：将该 `error!` 改为 `warn!`，或区分"协议解析失败（error）"与"未知类型（warn）"两条日志路径

```rust
// 当前（subscriber.rs:80）
tracing::error!(error = ?e, raw = %payload, "Failed to parse admin event from Redis");

// 建议
tracing::warn!(error = ?e, raw = %payload, "Unknown or malformed admin event type, dropped");
```
