# TDS: WebSocket 连接封装 (Task ID: T-30008)

- **关联需求**：语聊房实时通信基础设施（`product.md` § 房间模块长连接）

## 二、方案设计

### 核心数据流

```
AppContainer
    └─ OkHttpWebSocketClient(okHttpClient, scope)
           │
           │  connect(url, token)
           ▼
       OkHttp WebSocket ──── onOpen ────────────────────► StateFlow: Connected
           │                onMessage ──────────────────► StateFlow: Message(text)
           │                onClosed / onFailure ───────► StateFlow: Disconnected / Error
           │
           │  心跳协程 (每 15s)
           │    send({"type":"ping"})
           │    收到 {"type":"pong"} → 重置计时
           │
           │  重连协程 (非主动关闭触发)
           └─ 指数退避: 1s→2s→4s→8s→16s→32s, 最多 8 次
                超限 → StateFlow: Error(MaxRetryExceeded)
```

### 接口 / 信令定义

引用 `doc/protocol.md § 6.2 心跳` 与 `§ 6.3 消息通用格式`：

**心跳（客户端主动）**：
```json
{ "type": "ping" }
```
**心跳回复（服务端）**：
```json
{ "type": "pong" }
```
- 客户端每 **15 秒**发送一次 ping（server 30s 无心跳断开，留出冗余）
- 收到 pong 后重置 15s 计时器

**消息通用格式**：
```json
{
  "type": "EventType",
  "msg_id": "uuid",
  "payload": {},
  "timestamp": 1713312000
}
```

### 状态定义

```kotlin
// core/ws/WebSocketState.kt
sealed class WebSocketState {
    /** 正在建立连接（含重连中） */
    object Connecting : WebSocketState()
    /** 连接已就绪，可发送消息 */
    object Connected : WebSocketState()
    /** 连接已断开（非错误，包含主动关闭） */
    data class Disconnected(val reason: String = "") : WebSocketState()
    /** 收到服务端文本消息 */
    data class Message(val text: String) : WebSocketState()
    /** 不可恢复错误（含超出最大重试次数） */
    data class Error(val throwable: Throwable) : WebSocketState()
}
```

### 接口定义

```kotlin
// core/ws/IWebSocketClient.kt
interface IWebSocketClient {
    /** 当前连接状态流，初始为 Disconnected */
    val state: StateFlow<WebSocketState>

    /**
     * 建立 WebSocket 连接。
     * @param spec  由 RoomSocketRequestFactory.create() 生成的请求规格
     */
    suspend fun connect(spec: RoomSocketRequestSpec)

    /**
     * 发送文本消息。连接未就绪时静默丢弃（调用方需判断状态）。
     */
    fun send(message: String)

    /**
     * 主动断开连接，不触发自动重连。
     * 会清理心跳与重连协程。
     */
    fun disconnect()
}
```

### 重连策略

| 重试次数 | 退避时长 |
|---------|---------|
| 1 | 1s |
| 2 | 2s |
| 3 | 4s |
| 4 | 8s |
| 5 | 16s |
| 6 | 32s |
| 7 | 32s |
| 8 | 32s（最后一次） |
| >8 | 发出 `Error(MaxRetryExceededException)` |

- 重连触发条件：`onFailure()` 或非主动关闭的 `onClosed()`
- 主动调用 `disconnect()` 设置 `isManualClose = true`，不进入重连
- 重连成功后重置计数器

## 五、Reviewer意见

> 状态枚举：🟢 通过 | 🔴 未通过
> 每轮 Review 追加一条记录，不要覆盖历史。

### Round 1

- **状态**：🔴 未通过

- **审查意见**：

  发现 **2 个 HIGH 问题** 和 **2 个 MEDIUM 问题**，按审查决定（任何 HIGH → 不通过）判定为未通过。

  #### HIGH-1：`pingJob` 缺少 `@Volatile` — 跨线程可见性缺陷

  **位置**：`OkHttpWebSocketClient.kt` 第 62 行
  ```kotlin
  private var pingJob: Job? = null   // ❌ 缺少 @Volatile
  ```
  `pingJob` 从两类线程并发访问：
  - **OkHttp 内部线程**（`onOpen` / `onMessage` 内调用 `startHeartbeat()`；`onClosed` / `onFailure` 内调用 `stopHeartbeat()`）
  - **调用者任意线程**（`disconnect()` 内调用 `stopHeartbeat()`）

  对比同文件其他共享字段（`webSocket`、`isManualDisconnect`、`currentSpec`、`isConnectionActive`），它们均已标注 `@Volatile`，唯独 `pingJob` 遗漏。
  JVM 内存模型不保证无 `@Volatile` 字段的跨线程可见性：`disconnect()` 可能读到 stale 的 `pingJob`（仍为 null 或旧 Job），导致 `cancel()` 无效——心跳在 disconnect 后仍额外发送一次 ping。
  虽然 `isConnectionActive` 的保护使心跳在下一次迭代自行退出，不会造成严重资源泄漏，但这是标准的 Kotlin/JVM 线程安全规范违规。

  #### HIGH-2：`scheduleReconnect()` 的 `catch (_: Exception)` 吞噬 `CancellationException`

  **位置**：`OkHttpWebSocketClient.kt` 第 193 行
  ```kotlin
  } catch (_: Exception) {   // ❌ 吞噬 CancellationException
      isReconnecting.set(false)
  }
  ```
  在 Kotlin 协程中，`CancellationException` 继承自 `Exception`。`scope` 被取消时，`delay()` 抛出 `CancellationException`；代码将其捕获后**未重新抛出**，违反了 Kotlin 结构化并发的协作取消协议。
  后果：父 scope 取消时，正在退避等待中的重连协程不能被及时取消——它完成 `catch` 块后才"正常退出"，这会延迟整个 `scope` 的 `Job` 完成，阻碍 App 关闭或生命周期切换时的资源清理。

  #### MEDIUM-1：`disconnect()` 未调用 `webSocket?.cancel()`

  **位置**：`OkHttpWebSocketClient.kt` 第 84 行
  ```kotlin
  webSocket?.close(1000, "manual")   // ⚠️ 只发关闭帧，不强制释放
  ```
  `close()` 仅发送 WebSocket 关闭握手并**等待**服务端回应（OkHttp 默认 60s 关闭超时）。若服务端无响应，底层 Socket 资源最多 60s 才会释放。
  TDS 审查要点明确要求验证 `cancel()` 是否调用。应在 `close()` 之后或替代 `close()` 调用 `webSocket?.cancel()` 以立即释放连接。

  #### MEDIUM-2：缺少错误日志

  整个 `OkHttpWebSocketClient.kt` 中没有任何 `Log.w` / `Log.e` 调用。
  以下关键事件均无日志记录：
  - `onFailure`（连接失败 / 网络中断）
  - `scheduleReconnect` 触发重连（含当前重试次数和退避时长）
  - `MaxRetryExceededException`（超出最大重试次数）

  生产环境定位 WebSocket 断连问题时将无任何日志可追溯。

- **改进建议**：

  1. **HIGH-1 修复**：为 `pingJob` 添加 `@Volatile`：
     ```kotlin
     @Volatile private var pingJob: Job? = null
     ```

  2. **HIGH-2 修复**：将 `catch (_: Exception)` 改为正确的取消异常处理——推荐使用 `finally` 块：
     ```kotlin
     scope.launch {
         try {
             // ... 现有逻辑 ...
         } finally {
             // isReconnecting 仅在非手动断开路径中残留 true 时才需重置
             // 正常路径已在 doConnect() 前 set(false)；异常/取消时统一兜底
             if (isReconnecting.get()) isReconnecting.set(false)
         }
     }
     ```
     或最小改动方式（保留现有结构）：
     ```kotlin
     } catch (e: CancellationException) {
         isReconnecting.set(false)
         throw e   // 必须重新抛出
     } catch (_: Exception) {
         isReconnecting.set(false)
     }
     ```

  3. **MEDIUM-1 修复**：在 `disconnect()` 中补充 `cancel()`：
     ```kotlin
     override fun disconnect() {
         isManualDisconnect = true
         isConnectionActive = false
         stopHeartbeat()
         webSocket?.close(1000, "manual")
         webSocket?.cancel()   // 新增：强制立即释放 Socket
         webSocket = null
         _state.value = WebSocketState.Disconnected("manual")
     }
     ```

  4. **MEDIUM-2 修复**：在关键路径补充日志（示例）：
     ```kotlin
     // onFailure
     Log.w(TAG, "WebSocket onFailure (manual=$isManualDisconnect)", t)

     // scheduleReconnect
     Log.w(TAG, "Scheduling reconnect #${retryCount.get() + 1}/$maxRetries, backoff=${backoffSec}s")

     // MaxRetryExceeded
     Log.e(TAG, "WebSocket max retry exceeded ($currentRetry attempts)")
     ```
