# TDS: App Server 离开房间逻辑 (Task ID: T-00013)

## 一、背景与目标

- **关联需求**：语音房间核心功能——用户离开房间（T-00012 的对称操作）
- **前置依赖**：T-00012（进入房间逻辑 ✅ Done）
- **本 Task 的目标**：处理 `LeaveRoom` WS 信令及连接断开两种触发场景，将用户从房间内存状态移除，若在麦上自动下麦，并向房间剩余成员广播 `UserLeft` 事件；提取 `do_leave_room` 复用函数，主动离开与被动断线共享同一逻辑路径

## 二、方案设计

### 信令格式（对齐 protocol.md）

**请求（客户端 → 服务端）**：
```json
{
  "type": "LeaveRoom",
  "msg_id": "uuid"
}
```
> 无需 payload，服务端从 `ConnectionRegistry` 中通过 `connection_id` 查询 `room_id`。

**成功响应（服务端 → 请求方，仅主动离开时发送）**：
```json
{
  "type": "LeaveRoomResult",
  "msg_id": "uuid",
  "code": 0,
  "timestamp": 1713312000
}
```

**广播（服务端 → 房间剩余成员，不含离开者本人）**：
```json
{
  "type": "UserLeft",
  "payload": { "user_id": "uuid" },
  "timestamp": 1713312000
}
```

### do_leave_room 处理流程（9 步）

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | `registry.get_room_id(connection_id)` | → None：用户不在任何房间，静默返回（非错误） |
| 2 | `room_manager.get_room(room_id)` | → None：内存中房间已不存在，调用 `clear_room_id` 后静默返回 |
| 3 | `room_state.members.remove(user_id)` | 从 DashMap 移除成员 |
| 4 | `room_state.remove_from_mic_slots(user_id)` | 若在麦上则置 None（自动下麦） |
| 5 | `registry.clear_room_id(connection_id)` | **先清除再广播**，使广播时自然排除离开者 |
| 6 | `stats.user_leave_room(room_id).await.ok()` | 失败不阻断主流程 |
| 7 | 广播 `UserLeft` | `get_connections_in_room(room_id)`（此时离开者已被排除） |
| 8 | `room_state.member_count() == 0` | 若房间空了则 `room_manager.remove_room(room_id)` |
| 9 | 仅主动离开时 | `handle_leave_room` 返回 `LeaveRoomResult { code: 0 }` |

### 关键设计决策

**决策 1：`do_leave_room` 提取为独立函数，主动/被动复用**

**决策 2：先 `clear_room_id` 后广播**
> `get_connections_in_room` 依赖 `handle.room_id` 过滤，清除后该连接自然不会出现在广播列表中

**决策 3：断线路径中 `do_leave_room` 在 `unregister` 之前调用**
> `unregister` 删除 `ConnectionHandle` 后 `get_room_id` 返回 None，顺序必须严格

**决策 4：room_id 不存在时静默成功**

**决策 5：`LeaveRoomDeps` 不含 `room_service`/`auth_service`**

**决策 6：空房间即时清理**
