# TDS: Splash 启动页 (Task ID: T-30019)

## 一、背景与目标
- **关联需求**：`doc/product/android_app_design.md` §2.1 Splash 启动页 + `doc/design/android/T-30019.md`
- **前置依赖**：T-30018（MenaTheme 中东黑金主题系统，已完成 ✅）
- **本 Task 的目标**：实现品牌 Splash 启动页，展示 App Logo 缩放+淡入动画，动画结束后检测本地 JWT Token 有效性，自动导航到 MainScreen（有效）或 LoginScreen（无效/缺失），导航后不可返回 Splash。同时引入 Compose Navigation (`NavHost`) 作为全局导航骨架，为后续 T-30020（MainScreen 三Tab）和 T-30021（登录页升级）铺路。

## 二、方案设计

### 核心数据流

```
App Launch
    │
    ▼
MainActivity (setContent)
    │
    ▼
NavHost(startDestination = "splash")
    │
    ▼
SplashScreen (Composable)
    │  ① Logo 缩放 0.5→1.0 + alpha 0→1，duration 800ms EaseOut
    │  ② 动画完成后延迟 500ms（总停留 ~1.3s）
    │
    ▼
SplashViewModel.checkAuth()
    │  读取 ITokenManager.getToken()
    │
    ├── token != null && token.isNotBlank()
    │   → 发射 SplashNavEvent.NavigateToMain
    │   → NavController.navigate("main") {
    │       popUpTo("splash") { inclusive = true }
    │   }
    │
    └── token == null || token.isBlank()
        → 发射 SplashNavEvent.NavigateToLogin
        → NavController.navigate("login") {
            popUpTo("splash") { inclusive = true }
        }
```

### 关键设计决策

1. **引入 Compose Navigation**：当前 `MainActivity` 使用传统 View 渲染 (`setContentView(R.layout.activity_main)`)。本 Task 将 `MainActivity` 改造为 `setContent { MenaTheme { NavHost(...) } }`，以 Compose Navigation 作为全局导航骨架。旧 View 页面（auth/room/profile buttons）暂保留为 `MainScreen` 的占位内容，后续 T-30020 替换为真正的三 Tab 框架。

2. **JWT 检测策略 — 仅判断本地存在性**：Phase 0.5 阶段不做远程 token 校验（无 refresh token 接口），仅判断 `getToken() != null && isNotBlank()`。后续可扩展为解析 JWT exp 字段做本地过期判断。

3. **动画与导航解耦**：`SplashScreen` 通过 `Animatable` 驱动动画，动画完成后调用 `SplashViewModel.checkAuth()`，ViewModel 发射 `SplashNavEvent`，UI 层 `LaunchedEffect` 监听事件驱动 `NavController` 导航。保持 ViewModel 不持有 NavController 引用。

4. **popUpTo inclusive**：导航到 main/login 时使用 `popUpTo("splash") { inclusive = true }` 清除 Splash 出栈，防止返回键回退。

5. **AppContainer.tokenManager 暴露**：当前 `tokenManager` 仅在 `AppContainer.fromBuildConfig()` 内部创建为匿名实现，未暴露给外部。本 Task 需将 `tokenManager` 作为 `AppContainer` 的公开属性，供 `SplashViewModel.Factory` 注入。

### SplashViewModel 核心伪代码

```kotlin
class SplashViewModel(
    private val tokenManager: ITokenManager
) : ViewModel() {

    private val _navEvent = MutableSharedFlow<SplashNavEvent>()
    val navEvent: SharedFlow<SplashNavEvent> = _navEvent.asSharedFlow()

    fun checkAuth() {
        viewModelScope.launch {
            val token = try {
                tokenManager.getToken()
            } catch (_: Exception) {
                null  // 读取失败视为未登录
            }
            if (token != null && token.isNotBlank()) {
                _navEvent.emit(SplashNavEvent.NavigateToMain)
            } else {
                _navEvent.emit(SplashNavEvent.NavigateToLogin)
            }
        }
    }

    class Factory(
        private val tokenManager: ITokenManager
    ) : ViewModelProvider.Factory { ... }
}
```

### AppNavGraph 核心伪代码

```kotlin
@Composable
fun AppNavGraph(appContainer: AppContainer) {
    val navController = rememberNavController()
    NavHost(navController = navController, startDestination = "splash") {
        composable("splash") {
            SplashScreen(
                appContainer = appContainer,
                onNavigateToMain = {
                    navController.navigate("main") {
                        popUpTo("splash") { inclusive = true }
                    }
                },
                onNavigateToLogin = {
                    navController.navigate("login") {
                        popUpTo("splash") { inclusive = true }
                    }
                }
            )
        }
        composable("login") {
            // 现有 LoginScreen 入口（后续 T-30021 升级视觉）
        }
        composable("main") {
            // 现有壳层占位（后续 T-30020 替换为三 Tab）
        }
    }
}
```

### Round 1
- **状态**：🔴 未通过

- **改进建议**：在 `catch (_: Exception)` 前添加 `catch (e: CancellationException) { throw e }`
- **修复记录**：

  - 修改 `SplashViewModel.kt`：`checkAuth()` 方法 catch 块添加 `catch (e: CancellationException) { throw e }`

### Round 2
- **状态**：🟢 通过
- **审查意见**：R1 HIGH 问题已正确修复。`catch (e: CancellationException) { throw e }` 位于 `catch (_: Exception)` 之前（第 36 行 vs 第 38 行），与项目既有模式（`RoomViewModel.kt` 6 处同类 catch）完全一致。SP-05b 测试逻辑正确——验证 CancellationException 重抛后协程取消退出、不走 emit 分支、events 为空。全量 6/6 测试通过，无新引入问题。
