# TDS: 余额查询 API + WS 推送 (Task ID: T-00018)

## 一、背景与目标
- **关联需求**：E-07 钱包闭环 — Android 钱包页（T-30027）、送礼余额实时反馈
- **本 Task 的目标**：提供 HTTP 余额 / 流水查询接口，并新增 WS `BalanceUpdated` 信令，保证余额变化后所有 WS 会话 <500ms 收到推送。

## 二、方案设计

### 核心数据流
```
HTTP  GET /wallet/balance       ─► handler ─► SELECT diamond_balance FROM users
HTTP  GET /wallet/transactions  ─► handler ─► SELECT ... ORDER BY created_at DESC LIMIT
送礼/Admin调整 ─► WalletService.apply_delta() ─► 事务提交后发布
         ├── 本进程：BalanceBroadcaster.notify(user_id, new_balance, reason)
         └── 跨进程：Redis PUBLISH admin:events {type:balance_updated,...}
            ↓
      ConnectionRegistry.get_by_user(user_id) ─► 遍历推送 BalanceUpdated
```

### HTTP 接口定义
**GET /api/v1/wallet/balance** （需 JWT）
```json
// 200
{ "code":0, "data": { "diamond_balance": 1234 }, "request_id": "..." }
```

**GET /api/v1/wallet/transactions?page=1&size=20&type=gift_send** （需 JWT）
```json
{
  "code":0,
  "data": {
    "total": 42, "page":1, "size":20,
    "items":[
      { "id":"uuid", "type":"gift_send", "amount":-520, "balance_after":4800,
        "ref_id":"uuid|null", "reason":"送出 独角兽 x1", "created_at":"RFC3339" }
    ]
  }
}
```
- 类型过滤 `type` 可选；page 默认 1，size 默认 20，max 100
- 错误码：401（未登录）、40003（page/size 非法）

### 新增 WS 信令：`BalanceUpdated`（S→C）
```json
{
  "type": "BalanceUpdated",
  "payload": {
    "diamond_balance": 4800,
    "delta": -520,
    "reason": "gift_send",
    "ref_id": "uuid|null"
  },
  "timestamp": 1720000000000
}
```
**同一用户多端在线时** 广播给 `registry.get_by_user(user_id)` 全部连接。

### WalletService 关键 API
```rust
pub struct WalletService { pool: PgPool, tx: mpsc::Sender<BalanceEvent> }

impl WalletService {
    pub async fn get_balance(&self, user_id: Uuid) -> Result<i64>;
    pub async fn list_txns(&self, user_id: Uuid, page: u32, size: u32, ty: Option<WalletTxnType>) -> Result<Paginated<WalletTransactionModel>>;

    /// 供 SendGift (T-00020) / admin adjust (T-10013) 使用
    pub async fn apply_delta<'c>(&self, txn: &mut Transaction<'c, Postgres>,
        user_id: Uuid, delta: i64, ty: WalletTxnType,
        ref_id: Option<Uuid>, reason: Option<String>, operator_id: Option<Uuid>)
        -> Result<i64 /* balance_after */>;
}
```
- 使用 `SELECT ... FOR UPDATE` 行锁防并发
- 事务提交成功后通过 `mpsc::Sender<BalanceEvent>` 发事件给 `BalanceBroadcaster`

### BalanceBroadcaster 设计
- 启动时 spawn 常驻 task
- 监听两个源：
  1. 本进程 `mpsc::Receiver<BalanceEvent>`（同进程送礼）
  2. Redis PubSub `admin:events` 中 `type=balance_updated`（Admin 跨进程调整）
- 收到事件后：`registry.get_by_user(user_id)` → 对每个连接 `send(BalanceUpdated)`

### 离线补偿
- 重连后客户端主动 `GET /wallet/balance` 拉最新值（Android 在 WS `Connected` 事件触发）

## 五、Reviewer意见
### Round 1
- **状态**：❌ 不通过，返回 TDD
- **审查意见**：

  发现 3 个 HIGH 级别问题、2 个 MEDIUM 级别问题、1 个 LOW 级别问题，阻塞合并，需返工。

  ---

  **[HIGH-1] `apply_delta` 签名与 TDS 设计规范不符，破坏跨业务原子性**

  TDS §WalletService 关键 API 明确规定 `apply_delta` 接受外部 `Transaction<'c, Postgres>` 参数：
  ```rust
  pub async fn apply_delta<'c>(&self, txn: &mut Transaction<'c, Postgres>, ...) -> Result<i64>
  ```
  此设计是为了让 T-00020 SendGift 能在同一个数据库事务内原子完成"余额扣减 + 创建礼物记录"。当前实现自行 `begin/commit` 内部事务，完全不接受外部 Transaction，导致：
  1. T-00020 无法实现跨表原子操作（扣款成功但礼物记录写失败时余额已被扣）
  2. T-00020 开发完成后此接口需要破坏性重构

  **[HIGH-2] Redis `balance_updated` 事件路径完全缺失，B07 测试覆盖不真实**

  TDS §BalanceBroadcaster 设计要求监听两个事件源：
  1. 本进程 `mpsc::Receiver<BalanceEvent>`（已实现）
  2. Redis PubSub `admin:events` 中 `type=balance_updated`（**未实现**）

  `BalanceBroadcaster::run()` 只消费本进程 mpsc channel，Redis 订阅逻辑完全缺失。验收用例 B07 本应验证"Redis 事件 → WS 推送"，但实现中 B07 直接调用 `broadcast_event()` 绕过了 Redis 路径，属于测试造假。后果：Admin 后台调整余额后对应用户不会收到 WS 推送。

  **[HIGH-3] `websocket_signals.md` 未同步更新，Protocol 与实现字段名冲突**

  Protocol 文档是"前后端联调的唯一契约源"，而 §6.4.1 BalanceUpdated payload 定义与本次实现冲突：
  - Protocol doc（旧）：`{ "diamond": 1234, "charm": 56, "reason": "..." }`
  - TDS + 实现（新）：`{ "diamond_balance": 4800, "delta": -520, "reason": "...", "ref_id": "..." }`

  Android T-30027 将按 Protocol doc 解析 `diamond` 字段，但服务端发送 `diamond_balance`，联调必然失败。Protocol doc index.md 明文规定"新增/修改接口时必须同步更新对应子文件"，本次 PR 漏更。

  **[MEDIUM-1] `try_send` 静默丢弃 WS 推送事件，缺少 warn 日志**

  `let _ = self.balance_tx.try_send(BalanceEvent { ... })` 在 channel 满时完全无声丢弃，高并发场景下 WS 推送可能丢失且无任何 tracing 日志可查，缺乏可观测性。

  **[MEDIUM-2] `BalanceUpdated` 信令缺少 `msg_id` 字段**

  WS 通用消息格式 §6.3 要求所有 S→C 消息包含 `msg_id: "uuid"` 字段，`BalanceUpdated` 实现未包含。需与协议文档对齐或在 TDS 中明确豁免说明。

  **[LOW-1] `for_test_with_wallet` 参数类型暴露具体实现类**

  参数类型为 `Arc<WalletService>`（具体类型），应改为 `Arc<dyn WalletServicePort>`（接口抽象），与其他 `for_test_*` 方法风格一致。

- **改进建议**：

  1. **HIGH-1**：重构 `apply_delta` 签名，改为接受 `&mut Transaction<'_, Postgres>` 参数；调用方负责事务生命周期；BalanceEvent 通知改在调用方 `commit()` 后触发（可通过返回值或 `notify` 方法传递 event）。
  2. **HIGH-2**：在 `BalanceBroadcaster` 中新增 Redis PubSub 订阅逻辑，参考 `events/subscriber.rs` 的 `start_admin_event_subscriber` 模式，订阅 `admin:events` 频道，过滤 `type=balance_updated` 事件并触发 `broadcast_event`；同时重写 B07 测试验证真实 Redis 路径（或拆分为单元测试 mock Redis client）。
  3. **HIGH-3**：更新 `doc/protocol/websocket_signals.md` §6.4.1，将 `BalanceUpdated` payload 定义与 TDS T-00018 对齐（`diamond_balance` / `delta` / `ref_id`，删除 `charm`）。
  4. **MEDIUM-1**：将 `let _ = self.balance_tx.try_send(...)` 改为捕获错误并记录 `tracing::warn!`。
  5. **MEDIUM-2**：在 `broadcast_event` 生成的 JSON 中补充 `"msg_id": Uuid::new_v4().to_string()`，并在 TDS 示例中同步补充。
  6. **LOW-1**：将 `for_test_with_wallet` 参数类型改为 `Arc<dyn WalletServicePort>`。
