# TDS: Web 端数据看板首页 (Task ID: T-20003)

## 一、背景与目标

- **关联需求**：product.md — Admin Web 数据大盘
- **本 Task 的目标**：在 Dashboard 首页实现数据大盘，展示实时在线人数、活跃房间数、今日 DAU、新增用户，并通过 ECharts 折线图展示历史趋势，支持 30 秒自动刷新。
- **前置依赖**：T-20002（路由守卫）、T-10010（Admin Stats 接口，后端实现中；前端按契约 Mock）

## 二、方案设计

### 核心数据流

```
DashboardPage
  └─ useDashboardStats (Hook)
       ├─ adminGetRooms()                → totalRooms
       ├─ adminGetRooms({ status:'active' }) → activeRooms
       └─ adminGetStatsOverview()        → onlineUsers / dau / newUsersToday / trend
  └─ <StatCards stats loading />        → 4 张 Ant Design 统计卡片
  └─ <TrendChart trend loading />       → ECharts 折线图 / Empty / Skeleton
```

- `Promise.allSettled` 并发三个请求；任一失败不影响其他
- 仅当三个请求**全部失败**时才将 `error` 置为非 null
- 30s `setInterval` 自动刷新；卸载时 `clearInterval`

#### 🔴 HIGH — TrendChart `option` 对象未使用 `useMemo` 缓存

**文件**：`app/web/src/pages/dashboard/TrendChart.tsx:38`

**问题**：`const option = { ... }` 在函数体内每次 render 都构造新对象，包含两次 `trend.map()` 调用。`ReactECharts` 对 `option` prop 进行浅比较（引用相等），每次新对象引用都会触发 ECharts 内部的 `setOption` 重绘，导致：
1. 不必要的 ECharts 重绘（可见图表闪烁）
2. 大数据量趋势点时 `trend.map()` 重复执行的 CPU 开销

**修复方案**：

```tsx
// TrendChart.tsx
import { useMemo } from 'react';

export function TrendChart({ trend, loading }: TrendChartProps) {
  const { t } = useTranslation();

  const option = useMemo(() => ({
    tooltip: { trigger: 'axis' },
    legend: { data: [t('dashboard.dau'), t('dashboard.newUsersToday')] },
    xAxis: { type: 'category', data: trend.map((p) => p.date) },
    yAxis: { type: 'value' },
    series: [
      { name: t('dashboard.dau'), type: 'line', smooth: true, data: trend.map((p) => p.dau) },
      { name: t('dashboard.newUsersToday'), type: 'line', smooth: true, data: trend.map((p) => p.new_users) },
    ],
  }), [trend, t]);

  // ... render
}
```

#### �� MEDIUM — 组件卸载时无法取消飞行中的 HTTP 请求

**文件**：`app/web/src/pages/dashboard/useDashboardStats.ts:64` / `app/web/src/core/network/apiClient.ts:55`

**问题**：`adminFetch` 仅使用内部 15 秒超时的 `AbortController`，不接受外部 `signal` 参数。当 `useDashboardStats` 所在组件卸载时（如快速页面切换），`fetchData` 的三个并发请求无法被中止，请求完成后仍会调用 `setStats`/`setLoading` 等状态更新函数。虽然 React 18 已移除"向未挂载组件更新状态"的警告，但依然会造成无意义的网络流量和潜在的竞态问题。

**修复方案**：

```ts
// apiClient.ts — adminFetch 增加 signal 参数
async function adminFetch<T>(
  path: string,
  init?: RequestInit & { signal?: AbortSignal },
): Promise<ApiResponse<T>> { ... }

// useDashboardStats.ts — 挂载 AbortController，卸载时 abort
useEffect(() => {
  const abortController = new AbortController();
  void fetchData(abortController.signal);
  const timer = setInterval(() => {
    void fetchData(abortController.signal);
  }, AUTO_REFRESH_MS);
  return () => {
    abortController.abort();
    clearInterval(timer);
  };
}, [fetchData]);
```

#### 🟡 MEDIUM — `fetchDataRef` 模式冗余，增加无谓复杂度

**文件**：`app/web/src/pages/dashboard/useDashboardStats.ts:62`

**问题**：`fetchDataRef` 的设计目的是让 `setInterval` 回调始终调用"最新版本"的函数，以避免闭包捕获旧值。但当前 `fetchData` 的 `useCallback` 依赖数组为 `[]`，函数引用永远不会变化，因此 ref 是冗余的。多余的间接调用（`fetchDataRef.current?.()`）和赋值语句（`fetchDataRef.current = fetchData`）降低了代码可读性。

**修复方案**：删除 `fetchDataRef`，在 `useEffect` 中直接使用 `fetchData`：

```ts
useEffect(() => {
  void fetchData();
  const timer = setInterval(() => { void fetchData(); }, AUTO_REFRESH_MS);
  return () => { clearInterval(timer); };
}, [fetchData]);
```

#### 🟢 LOW — `as Error` 类型断言缺乏安全注释

**文件**：`app/web/src/pages/dashboard/useDashboardStats.ts:82`

**问题**：`totalResult.reason as Error` 使用了类型断言。Promise rejection 的 `reason` 类型为 `unknown`，若后端返回非 `Error` 对象（如字符串 `"timeout"`）则会在 `.message` 访问时得到 `undefined`。

**修复建议**：

```ts
const err = totalResult.reason instanceof Error
  ? totalResult.reason
  : new Error(String(totalResult.reason));
setError(err);
```
