## 二、方案设计

### 2.1 核心数据流（Vite mode → env 文件 → import.meta.env → 启动期校验 → apiClient）

```
              vite --mode {dev|test|staging|production}
                          │
                          ▼
       Vite 构建期按 mode 加载（官方约定，详见 §2.4）：
       1. .env                           （所有 mode，永远加载，可选；当前不存在）
       2. .env.local                     （所有 mode，gitignored；当前不存在）
       3. .env.{mode}                    （✅ 入库的 profile 模板）
       4. .env.{mode}.local              （✅ 开发者私有覆盖，gitignored）
                          │
                          ▼
       仅以 VITE_ 前缀的字段被 inline 到 import.meta.env.*
       （非 VITE_ 字段构建期可被 vite.config.ts 读到，但不进 bundle）
                          │
                          ▼
       app/web/src/core/config/env.ts
       ──────────────────────────────────
       export const webEnv = readWebEnv();   // 模块顶层立即执行
       readWebEnv() = {
         apiBaseUrl:        requireEnv('VITE_API_BASE_URL'),
         wsUrl:             requireEnv('VITE_WS_URL'),
         adminApiBaseUrl:   requireEnv('VITE_ADMIN_API_BASE_URL'),  // 🆕
         analyticsEndpoint: requireEnv('VITE_ANALYTICS_ENDPOINT'),
       };
       requireEnv(name):
         const v = (import.meta.env as Record<string,string|undefined>)[name];
         if (!v || v.trim() === '') {
           throw new Error(`[CONFIG ERROR] ${name} must be set`);
         }
         return v;
                          │
            模块导入时 throw → main.tsx 顶层 import 链失败
                          │
                          ▼
       浏览器控制台首屏可见红色 [CONFIG ERROR] xxx must be set
       （阻塞 React 渲染，不会出现 undefined/api 拼接的脏 URL）
                          │
                          ▼
       apiClient.ts:getAdminApiBaseUrl()
       ──────────────────────────────────
       return webEnv.adminApiBaseUrl;        // 删除 ?? 'http://localhost:3001/api/v1/admin'
       client.ts:apiClient(path)
       ──────────────────────────────────
       fetch(joinConfiguredUrl(webEnv.apiBaseUrl, path), init);
                          │
                          ▼
              真实请求落到正确域名（dev / test / staging / prod）
```

**关键不变量**：
- VITE_ 前缀字段在**构建期 inline**（产物里直接是字符串字面量），运行时无法通过 `process.env` 或 `import.meta.env` 改值；多 profile 切换**必须**通过构建/启动时的 `--mode` 参数。
- 启动期校验**必须发生在 `webEnv` 模块顶层**，由 `main.tsx` → `App` 链路的 import 自动触发；不能放在 `apiClient` 函数体内（否则只在调用时才报，违反 fail-fast）。
- 错误信息**统一前缀** `[CONFIG ERROR]`：与 T-00040 / T-10020 后端 `CONFIG ERROR:` 前缀风格一致（多冒号位置不同，但前缀关键词 `CONFIG ERROR` 一致），便于跨端 grep `CONFIG ERROR` 收敛诊断。

### 2.4 Vite mode 注入机制（必须在 TDD 阶段实测验证，不假设）

Vite 5 官方 mode 加载链（[Vite 文档 Env Variables and Modes](https://vitejs.dev/guide/env-and-mode.html)）：

| 命令 | mode 值 | 加载文件顺序（后者覆盖前者） |
|------|---------|------------------------------|
| `vite` (= `vite serve` = `npm run dev`) | `development` | `.env` → `.env.local` → `.env.development` → `.env.development.local` |
| `vite --mode test` | `test` | `.env` → `.env.local` → `.env.test` → `.env.test.local` |
| `vite --mode staging` | `staging` | `.env` → `.env.local` → `.env.staging` → `.env.staging.local` |
| `vite build` | `production` | `.env` → `.env.local` → `.env.production` → `.env.production.local` |
| `vite build --mode staging` | `staging` | 同 staging serve |

**TDD 阶段必须实测**：
- U1.1 `npx vite --mode staging --port 4173` 起后，访问 `http://127.0.0.1:4173/__env_probe`（不开发新页面，改用 `view-source` 看 bundle 中 `webEnv` 对象的 inline 字面量，或临时 `console.log(webEnv)` 然后 grep DevTools 输出）应包含 `.env.staging` 的占位 URL；
- U1.2 同上 `--mode test`，应包含 `.env.test` 的占位；
- U1.3 `npx vite build` (默认) 后 `dist/assets/*.js` 中 `grep "127.0.0.1:3001"` 应**不命中**（production 不能 leak dev URL）。

**vite.config.ts 是否需要改造**：
- 当前 `defineConfig({...})` 静态对象形态，不读 mode → vitest test 字段不依赖 mode → **默认不改**；
- 若 U1.* 任一项异常（如 vite 5.4 在 staging mode 下未加载 `.env.staging`），降级方案：改为 `defineConfig(({ mode }) => ({ ... }))` 并在内部 `console.log('[vite] mode:', mode)` 自证，作为 D-1 偏离记录。

### 2.5 启动期校验契约

#### 2.5.1 错误格式

```
[CONFIG ERROR] VITE_ADMIN_API_BASE_URL must be set
[CONFIG ERROR] VITE_API_BASE_URL must be set
[CONFIG ERROR] VITE_WS_URL must be set
[CONFIG ERROR] VITE_ANALYTICS_ENDPOINT must be set
```

- 前缀 `[CONFIG ERROR] ` 固定（含尾空格 + 字段名 + ` must be set`）；
- 与后端 T-00040 / T-10020 的 `CONFIG ERROR:` 前缀风格统一（关键词 `CONFIG ERROR` 一致即可，cross-tier `grep -r "CONFIG ERROR"` 全收敛）；
- 不暴露字段值（无值才报，故无脱敏问题）。

#### 2.5.2 触发时机

| 时机 | 行为 |
|------|------|
| 浏览器加载首个 JS chunk | `main.tsx` import `./app/App` → import `webEnv`（或 import `apiClient` 间接 import `webEnv`）→ `readWebEnv()` 模块顶层立即执行 → 缺值 throw |
| 浏览器控制台 | 首屏红色未捕获错误 `Uncaught Error: [CONFIG ERROR] VITE_ADMIN_API_BASE_URL must be set`，React 不渲染（白屏 + 控制台错误） |
| 单元测试（vitest） | `setup.ts` 通过 `vi.stubEnv` 注入四字段默认值，避免 `webEnv` 模块顶层 throw 导致**所有用例集体红**；新增 U7.* 用例显式覆盖 throw 路径 |
| `vite build`（构建期） | VITE_ 字段空值不会让构建失败（vite 不校验值），但 bundle 中会内嵌 `undefined`，运行时校验兜底；T-0000L Runbook 应在 staging/prod 构建前增加 `grep VITE_ADMIN_API_BASE_URL= app/web/.env.{mode} \| grep -v '=$'` 自检 |

#### 2.5.3 与 R8 的关系

T-0000E §2.12 R8「Web `apiClient.ts` 删默认值后线下运行白屏」——本设计**接受白屏作为 fail-fast 的代价**，但用三道防线缓解：
1. 错误信息明确指向字段名（开发者一眼定位）；
2. `.env.development` 入库且包含完整四字段（开箱即用，无需 dev 自己 copy）；
3. T-0000L Runbook 中"首次 clone 启动"步骤明示 `cp .env.example .env.local` + 校验四字段。

### 2.6 测试基础设施改造（确保现有 web unit test 0 回归）

#### 现有约束

`app/web/src/core/network/apiClient.test.ts` 等现有用例**未对 `import.meta.env.*` 做 stub**——意味着 vitest 跑时，jsdom 环境下 `import.meta.env.VITE_ADMIN_API_BASE_URL` 当前返回 `undefined`，靠 `?? 'http://localhost:3001/api/v1/admin'` 兜底。

**删除默认值后**：现有所有用例的第一次 `import './apiClient'` 链将立即 throw `[CONFIG ERROR]`，所有用例集体红。

#### 解决方案（写入 `src/test/setup.ts` 顶部）

```typescript
// 测试环境默认 stub 四字段，避免 webEnv 模块顶层 throw
// 单测如需测 throw 路径请用 vi.unstubAllEnvs() + vi.stubEnv() 重新设置
import { vi } from 'vitest';
vi.stubEnv('VITE_API_BASE_URL', 'http://127.0.0.1:3000/api');
vi.stubEnv('VITE_WS_URL', 'ws://127.0.0.1:3000/ws');
vi.stubEnv('VITE_ADMIN_API_BASE_URL', 'http://127.0.0.1:3001/api/v1/admin');
vi.stubEnv('VITE_ANALYTICS_ENDPOINT', 'https://analytics-test.example.com/collect');
```

> ⚠️ vitest 的 `vi.stubEnv` 在 v0.34+ 支持 `import.meta.env`（参考 [vitest #2613](https://github.com/vitest-dev/vitest/pull/2613)）；当前项目 `"vitest": "^3.1.2"` 已在支持范围内。

#### 新增 U7.* 用例时的反向操作

```typescript
import { vi, beforeEach, afterEach } from 'vitest';
beforeEach(() => { vi.unstubAllEnvs(); /* 重新精确 stub */ });
afterEach(() => { vi.unstubAllEnvs(); });
```
